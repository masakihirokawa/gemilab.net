## thinking_level パラメータとは

Gemini 3 シリーズ以降のモデルには、応答生成前にモデルが行う「内部推論（thinking）」の深さを制御できる `thinking_level` パラメータが導入されています。このパラメータを適切に設定することで、応答の品質とAPIコスト・レイテンシのバランスを細かく調整できます。

デフォルトでは Gemini 3 モデルは `HIGH`（動的思考モード）で動作し、複雑なプロンプトに対して十分な推論を行います。しかし、すべてのリクエストに最大の推論深度が必要なわけではありません。単純な翻訳や要約タスクに `HIGH` を使うのはコストの無駄になりますし、逆に数学的証明や複雑なコード生成に `LOW` を使えば品質が低下します。

`thinking_level` を使いこなすことは、Gemini API を本番環境で効率的に運用するための重要なスキルです。

## 利用可能なモデルとレベル一覧

`thinking_level` パラメータは、Gemini 3 以降のモデルで利用可能です。モデルによって設定できるレベルが異なるため、まずは対応表を確認しましょう。

| モデル | 利用可能なレベル | デフォルト | 備考 |
|--------|------------------|-----------|------|
| Gemini 3.1 Pro | LOW, MEDIUM, HIGH | HIGH | HIGH で Deep Think Mini が有効化 |
| Gemini 3 Pro | LOW, MEDIUM, HIGH | HIGH | Gemini 3.1 Pro への移行推奨 |
| Gemini 3.1 Flash | MINIMAL, LOW, MEDIUM, HIGH | HIGH | MINIMAL は Flash 専用 |
| Gemini 3 Flash | MINIMAL, LOW, MEDIUM, HIGH | HIGH | 同上 |

Gemini 3.1 Pro で `HIGH` を指定すると、**Deep Think Mini** と呼ばれる専門的な推論モードが有効になります。これは ARC-AGI-2 ベンチマークで 77.1% を達成した高度な推論エンジンで、数学・論理・コード生成などの複雑なタスクで特に威力を発揮します。

## Python SDK での基本実装

Google AI Python SDK を使った `thinking_level` の設定方法を見ていきましょう。

```python
from google import genai
from google.genai import types

# クライアントの初期化
client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

# thinking_level を MEDIUM に設定してリクエスト
response = client.models.generate_content(
    model="gemini-3.1-pro",
    contents="Pythonでバイナリサーチを実装するコードを書いてください",
    config=types.GenerateContentConfig(
        thinking_config=types.ThinkingConfig(
            thinking_level="MEDIUM"
        )
    )
)

# 応答テキストの取得
print(response.text)

# thinking トークン数の確認（コスト計算に有用）
if response.usage_metadata:
    print(f"入力トークン: {response.usage_metadata.prompt_token_count}")
    print(f"出力トークン: {response.usage_metadata.candidates_token_count}")
    print(f"思考トークン: {response.usage_metadata.thoughts_token_count}")
```

## JavaScript / TypeScript での実装

Node.js 環境では `@google/genai` パッケージを使います。

```javascript
import { GoogleGenAI } from "@google/genai";

const ai = new GoogleGenAI({ apiKey: "YOUR_GEMINI_API_KEY" });

async function generateWithThinking(prompt, level = "MEDIUM") {
  const response = await ai.models.generateContent({
    model: "gemini-3.1-flash",
    contents: prompt,
    config: {
      thinkingConfig: {
        thinkingLevel: level,  // "MINIMAL" | "LOW" | "MEDIUM" | "HIGH"
      },
    },
  });

  console.log(response.text);

  // トークン使用量の確認
  if (response.usageMetadata) {
    console.log(`思考トークン: ${response.usageMetadata.thoughtsTokenCount}`);
  }

  return response.text;
}

// 使用例
await generateWithThinking("JSONデータをCSVに変換する関数を書いて", "LOW");
await generateWithThinking("再帰的な型推論アルゴリズムを設計して", "HIGH");
```

## レベル別の動作特性とユースケース

各レベルには明確な特性があり、タスクの性質に応じて使い分けることが重要です。

### MINIMAL（Flash 系モデル専用）

内部推論をほぼ行わず、最も高速かつ低コストで応答を生成します。推論が不要な単純なタスクに最適です。

**適したタスク**: テキストのフォーマット変換、定型文の生成、簡単な翻訳、キーワード抽出

### LOW

最小限の推論を行います。事実の確認や簡単な論理判断が必要な場合に使います。

**適したタスク**: 要約、感情分析、FAQ への回答、テンプレートベースのコード生成

### MEDIUM

標準的な推論深度です。多くのビジネスユースケースで品質とコストのバランスが最も良い設定です。

**適したタスク**: 技術文書の作成、中程度の複雑さのコード生成、データ分析のレポート作成、比較検討

### HIGH（デフォルト）

最大の推論深度で、Deep Think Mini（3.1 Pro の場合）が有効になります。複雑な問題に対して段階的に推論を積み重ねます。

**適したタスク**: 数学的証明、アルゴリズム設計、複雑なバグの特定、アーキテクチャ設計、論理パズル

## タスク種別に応じた自動ルーティング実装

本番環境では、リクエストの内容に応じて `thinking_level` を動的に切り替えるルーティングロジックが効果的です。以下はその実装例です。

```python
from google import genai
from google.genai import types

client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

# タスク種別ごとの thinking_level マッピング
TASK_ROUTING = {
    "translate": "LOW",
    "summarize": "LOW",
    "extract": "MINIMAL",
    "code_simple": "MEDIUM",
    "code_complex": "HIGH",
    "analysis": "MEDIUM",
    "reasoning": "HIGH",
    "math": "HIGH",
}

def classify_task(prompt: str) -> str:
    """プロンプトの内容からタスク種別を推定する"""
    prompt_lower = prompt.lower()
    if any(w in prompt_lower for w in ["翻訳", "translate", "英訳", "和訳"]):
        return "translate"
    if any(w in prompt_lower for w in ["要約", "summarize", "まとめ"]):
        return "summarize"
    if any(w in prompt_lower for w in ["抽出", "extract", "取り出"]):
        return "extract"
    if any(w in prompt_lower for w in ["証明", "proof", "数学", "math", "計算"]):
        return "math"
    if any(w in prompt_lower for w in ["設計", "design", "アーキテクチャ", "アルゴリズム"]):
        return "reasoning"
    if any(w in prompt_lower for w in ["コード", "code", "実装", "関数"]):
        return "code_simple"
    return "analysis"

def smart_generate(prompt: str, model: str = "gemini-3.1-flash") -> str:
    """タスク種別に応じた最適な thinking_level で応答を生成"""
    task_type = classify_task(prompt)
    level = TASK_ROUTING.get(task_type, "MEDIUM")

    response = client.models.generate_content(
        model=model,
        contents=prompt,
        config=types.GenerateContentConfig(
            thinking_config=types.ThinkingConfig(
                thinking_level=level
            )
        )
    )

    # ログ出力（本番では監視システムに送信）
    thoughts = response.usage_metadata.thoughts_token_count or 0
    print(f"[{task_type}] level={level}, thoughts={thoughts} tokens")

    return response.text

# 使用例
result = smart_generate("この英文を日本語に翻訳してください: Hello World")
# → [translate] level=LOW, thoughts=12 tokens

result = smart_generate("マージソートの時間計算量をO記法で証明してください")
# → [math] level=HIGH, thoughts=450 tokens
```

この方法により、単純なタスクでは `LOW` でコストを抑え、複雑なタスクでは `HIGH` で品質を確保する運用が可能になります。

## コスト比較と最適化のポイント

`thinking_level` の選択はAPIコストに直結します。思考トークンは出力トークンとは別にカウントされるため、`HIGH` に設定すると思考トークンの分だけコストが増加します。

以下は、同じプロンプト（中程度の複雑さのコード生成タスク）に対する各レベルの比較例です。

```python
import time
from google import genai
from google.genai import types

client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

prompt = "Pythonで非同期HTTPクライアントを実装し、リトライロジックとレート制限を組み込んでください"

levels = ["LOW", "MEDIUM", "HIGH"]

for level in levels:
    start = time.time()
    response = client.models.generate_content(
        model="gemini-3.1-flash",
        contents=prompt,
        config=types.GenerateContentConfig(
            thinking_config=types.ThinkingConfig(
                thinking_level=level
            )
        )
    )
    elapsed = time.time() - start
    meta = response.usage_metadata
    print(f"Level: {level}")
    print(f"  レイテンシ: {elapsed:.2f}秒")
    print(f"  思考トークン: {meta.thoughts_token_count}")
    print(f"  出力トークン: {meta.candidates_token_count}")
    print()

# 期待される出力例:
# Level: LOW
#   レイテンシ: 1.20秒
#   思考トークン: 30
#   出力トークン: 280
#
# Level: MEDIUM
#   レイテンシ: 2.15秒
#   思考トークン: 150
#   出力トークン: 350
#
# Level: HIGH
#   レイテンシ: 4.80秒
#   思考トークン: 520
#   出力トークン: 480
```

この例では、`LOW` から `HIGH` に変更することでレイテンシが約4倍、思考トークンが約17倍になっています。ただし、`HIGH` ではエラーハンドリングやエッジケースの考慮がより充実した出力が得られる傾向があります。

コスト最適化のポイントとしては、まず全リクエストの `thinking_level` 分布を可視化し、`HIGH` が使われている比率を確認しましょう。多くの場合、リクエストの 60〜70% は `LOW` または `MEDIUM` で十分な品質が得られます。

## Thought Signatures による推論の検証

Gemini 3.1 Pro の `HIGH` レベルで有効になる Deep Think Mini には、**Thought Signatures**（思考署名）という機能が搭載されています。これは推論過程の各ステップを暗号化・改ざん防止した形で記録する仕組みで、AIの推論が途中で改変されていないことを検証できます。

```python
response = client.models.generate_content(
    model="gemini-3.1-pro",
    contents="量子コンピューティングにおけるグローバーのアルゴリズムの計算量を証明してください",
    config=types.GenerateContentConfig(
        thinking_config=types.ThinkingConfig(
            thinking_level="HIGH"
        )
    )
)

# 思考プロセスの確認
for part in response.candidates[0].content.parts:
    if part.thought:
        print("[思考]", part.text[:200], "...")
    else:
        print("[応答]", part.text[:200], "...")
```

Thought Signatures は、医療・法律・金融など、AIの推論過程の透明性が求められる分野で特に有用です。推論の各ステップが記録されるため、「なぜこの結論に至ったか」を後から追跡できます。

さらに詳しい推論モデルの使い分け戦略については、[Gemini Deep Think × Adaptive Thinking 推論モデルの使い分け戦略](/articles/gemini-advanced/gemini-deep-think-adaptive-thinking-strategy) で APIコストを50%削減するテクニックを含めて解説しています。

## よくある質問（FAQ）

### Q1: thinking_level を指定しない場合のデフォルト値は？

Gemini 3 シリーズのすべてのモデルで、デフォルトは `HIGH` です。明示的に指定しない場合、モデルは最大の推論深度で動作します。コストを最適化したい場合は、タスクの複雑さに応じて明示的にレベルを指定することをおすすめします。

### Q2: thinking_level は Gemini 2.5 モデルでも使えますか？

`thinking_level` パラメータは Gemini 3 以降のモデルで推奨されています。Gemini 2.5 Pro/Flash では `thinking_budget`（トークン数ベース）による制御が用いられていましたが、3 系以降ではより直感的な `thinking_level` に移行しました。既存のコードを移行する場合は、`thinking_budget` を対応するレベルに読み替えてください。

### Q3: HIGH と MEDIUM で応答品質にどの程度の差がありますか？

タスクの種類によります。単純な質問応答や要約では差はほぼありません。一方、数学的推論、マルチステップの論理問題、複雑なコード生成では `HIGH` が明確に優れた結果を出します。まずは `MEDIUM` で試し、品質が不足する場合に `HIGH` に切り替えるアプローチが実用的です。

### Q4: ストリーミングと組み合わせて使えますか？

はい、`thinking_level` はストリーミングモードでも利用可能です。思考部分（`thought: true`）は最初に配信され、その後に応答テキストが続きます。ストリーミングと thinking_level の組み合わせについては、[Gemini API ストリーミング × Function Calling 実装ガイド](/articles/gemini-api/gemini-api-streaming-function-calling-guide) も参考にしてください。

### Q5: Flash モデルの MINIMAL と LOW の違いは？

`MINIMAL` はほぼ推論を行わず、トークン予測ベースで応答を生成します。`LOW` は最低限の推論ステップ（事実確認、簡単な論理判断）を行います。フォーマット変換やテンプレート出力など、推論が本質的に不要なタスクでは `MINIMAL` でコストを最小化できます。
