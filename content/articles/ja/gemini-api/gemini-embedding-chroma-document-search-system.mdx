## はじめに

社内に蓄積されたドキュメントやナレッジベースから、目的の情報を素早く見つけ出すのは意外と難しい作業です。従来のキーワード検索では、表現の揺れや類義語に対応できず、求めている情報にたどり着けないことがあります。

この課題を解決するのが **セマンティック検索**（意味検索）です。テキストを「ベクトル」と呼ばれる数値の配列に変換し、意味の近さで検索することで、キーワードが完全に一致しなくても関連性の高い結果を返せます。

この記事では、**Gemini Embedding API** でテキストをベクトル化し、軽量なベクトルデータベース **ChromaDB** に格納して、Pythonだけで社内ドキュメント検索システムを構築する方法をステップバイステップで解説します。

## セマンティック検索の仕組みと Gemini Embedding API の特徴

### セマンティック検索とは

セマンティック検索は、テキストの「意味」を数値ベクトルとしてエンコードし、ベクトル空間上の距離（コサイン類似度など）で関連性を計算する手法です。たとえば「社員の休暇申請の手順」と「有給休暇の取り方」は、キーワードが異なっていても意味的に近いため、セマンティック検索ではマッチします。

### Gemini Embedding API の強み

Googleの **text-embedding-004** モデル（およびマルチモーダル対応の **gemini-embedding-exp-03-07**）は、次のような特徴を備えています。

- **768次元の高精度ベクトル**: 文章の意味を豊かに表現できる次元数
- **日本語への高い対応力**: 多言語対応で、日本語テキストも高品質にベクトル化
- **タスクタイプの指定**: `RETRIEVAL_DOCUMENT`（格納用）と `RETRIEVAL_QUERY`（検索用）を使い分けることで検索精度が向上
- **大きな入力トークン数**: 最大 2,048 トークンまでの入力に対応

### ChromaDB を選ぶ理由

ChromaDBは、Pythonネイティブで動作する軽量なベクトルデータベースです。

- **インストールが簡単**: `pip install chromadb` だけですぐに使える
- **サーバー不要**: ローカルファイルに永続化できるため、PostgreSQLやDockerが不要
- **メタデータフィルタリング**: ベクトル検索とメタデータ条件を組み合わせた絞り込みが可能
- **本番移行も容易**: クライアント/サーバーモードに切り替えれば、そのままスケールアップできる

## ドキュメントの取り込みとベクトル化

まず、社内ドキュメントを読み込み、チャンク分割してベクトル化し、ChromaDBに格納するスクリプトを作成します。

```python
# ingest.py
import os
import chromadb
from google import genai

# Gemini クライアント初期化
client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

# ChromaDB クライアント（ローカル永続化）
chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection = chroma_client.get_or_create_collection(
    name="company_docs",
    metadata={"hnsw:space": "cosine"}  # コサイン類似度を使用
)


def chunk_text(text: str, chunk_size: int = 500, overlap: int = 100) -> list[str]:
    """テキストをオーバーラップ付きでチャンク分割する"""
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start += chunk_size - overlap
    return chunks


def embed_texts(texts: list[str]) -> list[list[float]]:
    """Gemini Embedding API でテキストをベクトル化する"""
    result = client.models.embed_content(
        model="text-embedding-004",
        contents=texts,
        config={
            "task_type": "RETRIEVAL_DOCUMENT",
            "output_dimensionality": 768,
        },
    )
    return [e.values for e in result.embeddings]


def ingest_document(filepath: str, doc_name: str):
    """ドキュメントを読み込み、チャンク分割してChromaDBに格納する"""
    with open(filepath, "r", encoding="utf-8") as f:
        text = f.read()

    chunks = chunk_text(text)
    embeddings = embed_texts(chunks)

    ids = [f"{doc_name}_chunk_{i}" for i in range(len(chunks))]
    metadatas = [
        {"source": doc_name, "chunk_index": i}
        for i in range(len(chunks))
    ]

    collection.add(
        ids=ids,
        embeddings=embeddings,
        documents=chunks,
        metadatas=metadatas,
    )
    print(f"✅ {doc_name}: {len(chunks)} チャンクを格納しました")


# documents/ ディレクトリ内の全ファイルを取り込む
doc_dir = "./documents"
for filename in os.listdir(doc_dir):
    if filename.endswith(".txt"):
        filepath = os.path.join(doc_dir, filename)
        ingest_document(filepath, filename.replace(".txt", ""))

print(f"📊 合計 {collection.count()} チャンクがデータベースに格納されています")

# 期待される出力:
# ✅ onboarding: 8 チャンクを格納しました
# ✅ leave-policy: 5 チャンクを格納しました
# ✅ security-guide: 12 チャンクを格納しました
# 📊 合計 25 チャンクがデータベースに格納されています
```

このスクリプトのポイントは次の3つです。

1. **チャンク分割**: ドキュメントを500文字ごとに分割し、100文字のオーバーラップを設けることで文脈の断絶を防ぎます
2. **タスクタイプの指定**: `RETRIEVAL_DOCUMENT` を指定することで、格納用に最適化されたベクトルが生成されます
3. **メタデータの付与**: 検索結果にドキュメント名やチャンク位置を含めることで、結果の追跡が容易になります

## 検索機能の実装

次に、ユーザーのクエリに基づいてセマンティック検索を行うスクリプトを作成します。

```python
# search.py
import os
import chromadb
from google import genai

client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection = chroma_client.get_collection("company_docs")


def search(query: str, n_results: int = 5, source_filter: str = None) -> dict:
    """セマンティック検索を実行する"""
    # クエリをベクトル化（RETRIEVAL_QUERY タイプを使用）
    result = client.models.embed_content(
        model="text-embedding-004",
        contents=[query],
        config={
            "task_type": "RETRIEVAL_QUERY",
            "output_dimensionality": 768,
        },
    )
    query_embedding = result.embeddings[0].values

    # ChromaDB で類似検索
    where_filter = {"source": source_filter} if source_filter else None
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=n_results,
        where=where_filter,
        include=["documents", "metadatas", "distances"],
    )

    return results


def display_results(results: dict):
    """検索結果を見やすく表示する"""
    for i, (doc, meta, dist) in enumerate(zip(
        results["documents"][0],
        results["metadatas"][0],
        results["distances"][0],
    )):
        similarity = 1 - dist  # コサイン距離を類似度に変換
        print(f"\n--- 結果 {i + 1} (類似度: {similarity:.4f}) ---")
        print(f"📄 ソース: {meta['source']}")
        print(f"📝 内容: {doc[:200]}...")


# 使用例
query = "新入社員のPCセットアップ手順を教えてください"
print(f"🔍 検索クエリ: {query}\n")
results = search(query)
display_results(results)

# 期待される出力:
# 🔍 検索クエリ: 新入社員のPCセットアップ手順を教えてください
#
# --- 結果 1 (類似度: 0.8742) ---
# 📄 ソース: onboarding
# 📝 内容: 入社初日のPC設定について。まずIT部門から支給されたノートPCの電源を入れ...
#
# --- 結果 2 (類似度: 0.8105) ---
# 📄 ソース: onboarding
# 📝 内容: VPN接続の設定手順。社内ネットワークにリモートからアクセスするために...
```

検索時のポイントとして、クエリのベクトル化には `RETRIEVAL_QUERY` タスクタイプを指定します。これは `RETRIEVAL_DOCUMENT` と対になるタスクタイプで、検索クエリに最適化されたベクトルを生成します。この使い分けによって検索精度が向上します。

## RAG パイプラインへの拡張

検索結果をそのまま返すだけでなく、**Gemini の生成AI機能**と組み合わせて、検索結果を基に自然な回答を生成する RAG（Retrieval-Augmented Generation）パイプラインに拡張しましょう。

```python
# rag_search.py
import os
import chromadb
from google import genai

client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection = chroma_client.get_collection("company_docs")


def rag_answer(query: str) -> str:
    """検索結果を基にGeminiで回答を生成する"""
    # Step 1: セマンティック検索で関連チャンクを取得
    embed_result = client.models.embed_content(
        model="text-embedding-004",
        contents=[query],
        config={"task_type": "RETRIEVAL_QUERY", "output_dimensionality": 768},
    )
    query_embedding = embed_result.embeddings[0].values

    search_results = collection.query(
        query_embeddings=[query_embedding],
        n_results=3,
        include=["documents", "metadatas"],
    )

    # Step 2: 検索結果をコンテキストとして整形
    context_parts = []
    for doc, meta in zip(
        search_results["documents"][0],
        search_results["metadatas"][0],
    ):
        context_parts.append(f"[{meta['source']}]\n{doc}")
    context = "\n\n".join(context_parts)

    # Step 3: Gemini で回答生成
    prompt = f"""以下の社内ドキュメントの情報を基に、ユーザーの質問に正確に回答してください。
ドキュメントに記載がない情報は「該当する情報が見つかりませんでした」と回答してください。

## 参照ドキュメント
{context}

## ユーザーの質問
{query}

## 回答"""

    response = client.models.generate_content(
        model="gemini-2.5-flash",
        contents=prompt,
    )

    return response.text


# 使用例
question = "リモートワーク時のVPN接続手順は？"
print(f"❓ 質問: {question}\n")
answer = rag_answer(question)
print(f"💡 回答:\n{answer}")

# 期待される出力:
# ❓ 質問: リモートワーク時のVPN接続手順は？
#
# 💡 回答:
# リモートワーク時のVPN接続は以下の手順で行います。
# 1. タスクバーのネットワークアイコンをクリックします
# 2. 「VPN接続」を選択し、社内IT部門から配布された...
```

このRAGパイプラインでは、まずセマンティック検索で関連性の高いドキュメントチャンクを取得し、それをコンテキストとしてGeminiに渡すことで、社内情報に基づいた正確な回答を生成します。

## メタデータフィルタリングと高度な検索

ChromaDBのメタデータフィルタリングを活用すると、ベクトル検索に加えて条件を絞り込むことができます。

```python
# 特定のドキュメントソースに限定して検索
results = collection.query(
    query_embeddings=[query_embedding],
    n_results=5,
    where={"source": "security-guide"},  # セキュリティガイドのみ検索
)

# 複数条件の組み合わせ（$and / $or）
results = collection.query(
    query_embeddings=[query_embedding],
    n_results=5,
    where={
        "$or": [
            {"source": "onboarding"},
            {"source": "leave-policy"},
        ]
    },
)
```

たとえば「セキュリティポリシーの中から、パスワードに関する規定を探したい」といった場面では、`source` フィルタでドキュメントを絞り込んだうえでセマンティック検索を行うことで、より精度の高い結果が得られます。

## 本番運用に向けた設計のポイント

ローカル開発からチームで使える本番システムに移行する際に考慮すべきポイントをまとめます。

**インクリメンタルな更新**: ドキュメントが更新された場合は、該当チャンクのIDを指定して `collection.update()` を呼び出すことで、全体の再取り込みを避けられます。削除されたドキュメントに対しては `collection.delete()` でベクトルを削除します。

**バッチ処理の最適化**: Gemini Embedding API には1回のリクエストで複数テキストを渡せるため、チャンクをまとめてバッチ処理することでAPIコールの回数を削減できます。レート制限（RPM/TPM）にも注意し、必要に応じてリトライロジックを実装しましょう。

**ChromaDBのスケールアップ**: 小規模なチームであればローカルファイル永続化で十分ですが、ドキュメント数が数万件を超える場合はChromaDBのクライアント/サーバーモード（`chromadb.HttpClient`）に移行し、専用サーバーとして運用することを検討してください。

**セキュリティ**: 社内ドキュメントを扱う場合、APIキーの管理やアクセス制御は欠かせません。環境変数やシークレットマネージャでキーを管理し、検索結果の閲覧権限を適切に設定しましょう。

## よくある質問（FAQ）

### Q1: ChromaDB と pgvector はどちらを選ぶべきですか？

プロトタイプや小〜中規模のプロジェクトでは ChromaDB が手軽でおすすめです。すでに PostgreSQL を運用していて大規模なデータを扱う場合は pgvector が適しています。ChromaDB はPythonだけで完結するため、インフラ構築のコストが低いのが大きなメリットです。Gemini API と pgvector を組み合わせた構成については、[Gemini API × pgvector セマンティック検索エンジン構築ガイド](/articles/gemini-api/gemini-api-pgvector-semantic-search-engine)で詳しく解説しています。

### Q2: text-embedding-004 と gemini-embedding-exp-03-07 の違いは何ですか？

`text-embedding-004` はテキスト専用のエンベディングモデルで、安定性とコスト効率に優れています。`gemini-embedding-exp-03-07` はマルチモーダル対応の実験的モデルで、テキストに加えて画像のベクトル化にも対応しています。テキスト検索のみであれば `text-embedding-004` が推奨です。エンベディングの基本については [Gemini Embedding API 入門ガイド](/articles/gemini-api/embeddings-api-guide) も参考にしてください。

### Q3: ドキュメントのチャンクサイズはどのくらいが最適ですか？

一般的には 300〜800 文字程度が推奨されます。短すぎると文脈が失われ、長すぎると検索精度が下がります。ドキュメントの性質によって最適値は異なるため、実際のデータでテストしながら調整するのがベストプラクティスです。オーバーラップ（重なり）を設けることで、チャンク境界をまたぐ情報も検索で拾えるようになります。

### Q4: 検索結果の精度を上げるにはどうすればよいですか？

いくつかのアプローチがあります。チャンクサイズとオーバーラップの調整、メタデータフィルタリングの活用、検索クエリのリライト（ユーザーの入力をGeminiで最適化してから検索する）、そして Re-ranking（検索結果をGeminiで並べ替える）などが効果的です。

## まとめ

Gemini Embedding API と ChromaDB を組み合わせることで、Python だけでセマンティック検索システムを構築できます。キーワードの一致に頼らない「意味ベース」の検索は、社内ナレッジの活用やカスタマーサポートの効率化に大きく貢献します。

さらに Gemini の生成AI機能と組み合わせた RAG パイプラインに拡張すれば、検索結果を基にした自然な回答生成も可能です。まずはローカル環境で試してみて、使い心地を確かめてみてください。
