# Gemini API で構築する自動テスト生成・コード品質分析システム

ソフトウェア開発において、テストの作成とコード品質の維持は重要でありながらも時間のかかる作業です。Gemini API の優れたコード理解能力を活用すれば、ユニットテストの自動生成からコードレビュー、品質メトリクスの分析まで、一連の品質保証プロセスを大幅に効率化できます。

この記事では、Gemini 2.5 Pro のコード理解・生成能力を最大限に活かし、Python で実用的な自動テスト生成・品質分析システムを構築する方法を、完全な実装コードとともに解説します。

## システム全体のアーキテクチャ

今回構築するシステムは、以下の3つの主要コンポーネントで構成されます。

1. **テスト自動生成エンジン** — ソースコードを解析し、ユニットテストを自動生成
2. **コードレビューアナライザー** — コードの品質問題を検出し改善案を提示
3. **品質メトリクスダッシュボード** — プロジェクト全体の品質指標を可視化

これらを Gemini API の Function Calling とStructured Output を組み合わせて実装します。

## テスト自動生成エンジンの実装

### ソースコード解析モジュール

まず、Python ソースコードを解析し、テスト対象の関数やクラスを抽出するモジュールを作成します。

```python
from dataclasses import dataclass, field

@dataclass
class FunctionInfo:
    """解析された関数の情報"""
    name: str
    args: list[str]
    return_type: str | None
    docstring: str | None
    source: str
    complexity: int = 0

@dataclass
class AnalysisResult:
    """ソースコード解析結果"""
    functions: list[FunctionInfo] = field(default_factory=list)
    classes: list[str] = field(default_factory=list)
    imports: list[str] = field(default_factory=list)

def analyze_source_code(source_code: str) -> AnalysisResult:
    """ソースコードをASTで解析し、関数・クラス情報を抽出する"""
    tree = ast.parse(source_code)
    result = AnalysisResult()

    for node in ast.walk(tree):
        if isinstance(node, ast.FunctionDef):
            args = [arg.arg for arg in node.args.args if arg.arg != "self"]
            return_type = ast.unparse(node.returns) if node.returns else None
            docstring = ast.get_docstring(node)
            source = ast.unparse(node)

            # サイクロマティック複雑度の簡易計算
            complexity = sum(
                1 for child in ast.walk(node)
                if isinstance(child, (ast.If, ast.For, ast.While, ast.ExceptHandler))
            ) + 1

            result.functions.append(FunctionInfo(
                name=node.name,
                args=args,
                return_type=return_type,
                docstring=docstring,
                source=source,
                complexity=complexity,
            ))

        elif isinstance(node, ast.ClassDef):
            result.classes.append(node.name)

        elif isinstance(node, (ast.Import, ast.ImportFrom)):
            result.imports.append(ast.unparse(node))

    return result
```

### Gemini によるテスト生成

解析結果をもとに、Gemini API でユニットテストを生成します。Structured Output を使って、テストコードを確実にパース可能な形式で受け取ります。

```python
def generate_tests_for_function(
    function_info: FunctionInfo,
    full_source: str,
) -> str:
    """Gemini API を使って関数のユニットテストを生成する

    Args:
        function_info: 解析済みの関数情報
        full_source: テスト対象ファイルの全ソースコード

    Returns:
        生成されたテストコード（文字列）
    """
    prompt = f"""以下のPython関数に対する包括的なユニットテストを生成してください。

## 対象関数
```python
{function_info.source}
```

## ファイル全体のコンテキスト
```python
{full_source}
```

## 要件
- pytest を使用すること
- 正常系、異常系、境界値のテストケースを含めること
- 各テストに日本語のdocstringを付けること
- テスト関数名は test_ で始めること
- 必要なimportを含めること
- モック（unittest.mock）が必要な場合は適切に使用すること

pytestのテストコードのみを出力してください。説明文は不要です。"""

    # Structured Output でテストコードを受け取る
    response = client.models.generate_content(
        model=MODEL_ID,
        contents=prompt,
        config=types.GenerateContentConfig(
            temperature=0.2,  # テスト生成は決定的に
            response_mime_type="application/json",
            response_schema={
                "type": "object",
                "properties": {
                    "test_code": {
                        "type": "string",
                        "description": "生成されたpytestテストコード",
                    },
                    "test_count": {
                        "type": "integer",
                        "description": "生成されたテストケース数",
                    },
                    "coverage_notes": {
                        "type": "string",
                        "description": "カバレッジに関する補足説明",
                    },
                },
                "required": ["test_code", "test_count", "coverage_notes"],
            },
        ),
    )

    result = json.loads(response.text)
    print(f"  ✅ {function_info.name}: {result['test_count']}件のテスト生成")
    print(f"     カバレッジ: {result['coverage_notes']}")
    return result["test_code"]
```

### テスト生成パイプライン

複数ファイルをまとめて処理するパイプラインを構築します。

```python
from pathlib import Path

def generate_tests_for_project(
    source_dir: str,
    output_dir: str,
) -> dict[str, str]:
    """プロジェクト全体のテストを自動生成する

    Args:
        source_dir: ソースコードのディレクトリパス
        output_dir: テスト出力先のディレクトリパス

    Returns:
        生成結果の辞書 {ファイルパス: テストコード}

    実行例:
        >>> results = generate_tests_for_project("src/", "tests/")
        >>> print(f"生成ファイル数: {len(results)}")
        生成ファイル数: 5
    """
    source_path = Path(source_dir)
    output_path = Path(output_dir)
    output_path.mkdir(parents=True, exist_ok=True)
    results = {}

    for py_file in source_path.rglob("*.py"):
        if py_file.name.startswith("_"):
            continue

        print(f"\n📄 解析中: {py_file}")
        source_code = py_file.read_text(encoding="utf-8")
        analysis = analyze_source_code(source_code)

        if not analysis.functions:
            print("  ⏭️ テスト対象の関数なし — スキップ")
            continue

        all_tests = []
        for func in analysis.functions:
            if func.name.startswith("_"):
                continue
            test_code = generate_tests_for_function(func, source_code)
            all_tests.append(test_code)

        # テストファイルの書き出し
        test_filename = f"test_{py_file.stem}.py"
        test_file = output_path / test_filename
        combined = "\n\n".join(all_tests)
        test_file.write_text(combined, encoding="utf-8")
        results[str(test_file)] = combined
        print(f"  💾 保存: {test_file}")

    return results
```

### レビュー実行エンジン

```python
@dataclass
class ReviewIssue:
    """レビューで検出された問題"""
    severity: str
    line_range: str
    category: str
    description: str
    suggestion: str

def review_code(source_code: str, filename: str) -> list[ReviewIssue]:
    """Gemini API でコードレビューを実行する

    Args:
        source_code: レビュー対象のソースコード
        filename: ファイル名（コンテキスト用）

    Returns:
        検出された問題のリスト

    期待する出力例:
        [
            ReviewIssue(
                severity='warning',
                line_range='12-15',
                category='error_handling',
                description='例外が広すぎるcatch節',
                suggestion='except ValueError as e:'
            ),
        ]
    """
    # 行番号付きソースコードを生成
    numbered = "\n".join(
        f"{i+1:4d} | {line}"
        for i, line in enumerate(source_code.splitlines())
    )

    prompt = f"""あなたはシニアPython開発者です。以下のコードを詳細にレビューしてください。

## ファイル: {filename}
```python
{numbered}
```

## レビュー観点
1. セキュリティの脆弱性（SQLインジェクション、入力バリデーション不足など）
2. パフォーマンスの問題（不要なループ、メモリリークなど）
3. エラーハンドリングの不備
4. 型安全性の問題
5. 命名規則やコード可読性
6. メンテナンス性の問題

問題を発見したら report_issue ツールを、良い実装を見つけたら approve_section ツールを使用してください。"""

    issues: list[ReviewIssue] = []

    response = client.models.generate_content(
        model=MODEL_ID,
        contents=prompt,
        config=types.GenerateContentConfig(
            tools=review_tools,
            temperature=0.1,
        ),
    )

    # Function Call の結果を処理
    for candidate in response.candidates:
        for part in candidate.content.parts:
            if part.function_call and part.function_call.name == "report_issue":
                args = dict(part.function_call.args)
                issues.append(ReviewIssue(**args))

    # 重大度でソート（critical > warning > info）
    severity_order = {"critical": 0, "warning": 1, "info": 2}
    issues.sort(key=lambda x: severity_order.get(x.severity, 3))

    return issues
```

## 品質メトリクス分析

プロジェクト全体の品質スコアを算出し、Gemini API で総合的な分析を行います。

### メトリクス収集

```python
@dataclass
class QualityMetrics:
    """プロジェクトの品質メトリクス"""
    total_functions: int = 0
    avg_complexity: float = 0.0
    functions_with_docstring: int = 0
    functions_with_type_hints: int = 0
    total_issues: int = 0
    critical_issues: int = 0
    test_coverage_estimate: float = 0.0

def collect_metrics(
    source_dir: str,
    issues: list[ReviewIssue] | None = None,
) -> QualityMetrics:
    """プロジェクトの品質メトリクスを収集する"""
    metrics = QualityMetrics()
    complexities = []

    for py_file in Path(source_dir).rglob("*.py"):
        if py_file.name.startswith("_"):
            continue

        source = py_file.read_text(encoding="utf-8")
        analysis = analyze_source_code(source)

        for func in analysis.functions:
            metrics.total_functions += 1
            complexities.append(func.complexity)
            if func.docstring:
                metrics.functions_with_docstring += 1
            if func.return_type:
                metrics.functions_with_type_hints += 1

    if complexities:
        metrics.avg_complexity = sum(complexities) / len(complexities)

    if issues:
        metrics.total_issues = len(issues)
        metrics.critical_issues = sum(
            1 for i in issues if i.severity == "critical"
        )

    return metrics
```

### Gemini による総合品質分析

```python
def analyze_project_quality(metrics: QualityMetrics) -> str:
    """Gemini API でプロジェクト品質の総合分析を生成する

    Args:
        metrics: 収集済みの品質メトリクス

    Returns:
        Markdown形式の分析レポート
    """
    prompt = f"""以下のプロジェクト品質メトリクスを分析し、
改善のための具体的なアクションプランを提案してください。

## メトリクス
- 関数総数: {metrics.total_functions}
- 平均サイクロマティック複雑度: {metrics.avg_complexity:.1f}
- docstring 記載率: {metrics.functions_with_docstring}/{metrics.total_functions}
- 型ヒント記載率: {metrics.functions_with_type_hints}/{metrics.total_functions}

## 分析要件
1. 品質スコア（100点満点）を算出
2. 最も改善効果の高い3つのアクションを提案
3. 各アクションの期待される効果を定量的に示す

Markdown形式で出力してください。"""

    response = client.models.generate_content(
        model=MODEL_ID,
        contents=prompt,
        config=types.GenerateContentConfig(temperature=0.3),
    )

    return response.text
```

## CI/CD パイプラインへの統合

GitHub Actions で自動実行できるワークフローの例を紹介します。

```yaml
# .github/workflows/ai-quality-check.yml
name: AI Quality Analysis

on:
  pull_request:
    paths: ["src/**/*.py"]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: pip install google-genai pydantic pytest

      - name: Run AI Code Review
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
        run: python scripts/ai_review.py src/ --output review_report.md

      - name: Generate Tests
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
        run: python scripts/ai_test_gen.py src/ --output tests/generated/

      - name: Run Generated Tests
        run: pytest tests/generated/ -v --tb=short

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const report = fs.readFileSync('review_report.md', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: report
            });
```

## 実践的な使用例

すべてを統合して実行するメインスクリプトです。

```python
def main():
    """AI品質分析パイプラインのメインエントリポイント

    実行例:
        $ python ai_quality_pipeline.py
        📊 AI品質分析パイプライン開始
        ============================================================

        🔍 フェーズ1: コードレビュー
        📄 レビュー中: src/utils.py
          ⚠️ [warning] L12-15 (error_handling): 例外が広すぎるcatch節
          ℹ️ [info] L23-25 (naming): 変数名が不明瞭

        🧪 フェーズ2: テスト自動生成
        📄 解析中: src/utils.py
          ✅ calculate_total: 8件のテスト生成
          ✅ validate_input: 5件のテスト生成
        ...

        📈 フェーズ3: 品質メトリクス分析
        品質スコア: 78/100
        ...
    """
    source_dir = "src"
    test_output = "tests/generated"

    print("📊 AI品質分析パイプライン開始")
    print("=" * 60)

    # フェーズ1: コードレビュー
    print("\n🔍 フェーズ1: コードレビュー")

        all_issues.extend(issues)

    print("\n✅ 分析完了 — quality_report.md に保存しました")

if __name__ == "__main__":
    main()
```

## よくある質問（FAQ）

### Q. Gemini API で生成されたテストはそのまま使えますか？

生成されたテストの約80〜90%はそのまま実行可能ですが、以下の確認を推奨します。まず、外部サービスとの連携部分にはモックが適切に設定されているか確認してください。次に、生成されたアサーションが期待値と正しく一致するか検証します。最後に、テスト間で状態が共有されていないか（テストの独立性）を確認してください。CI/CD に組み込む際は、生成されたテストを一度実行し、失敗するものを修正してからコミットするワークフローがおすすめです。

### Q. Gemini 2.5 Pro と 2.5 Flash でテスト生成の品質に差がありますか？

2.5 Pro は複雑なロジックの理解度が高く、境界値テストやエッジケースのカバレッジが優れています。一方、2.5 Flash は応答速度が約3倍速く、API コストも低いため、シンプルなユーティリティ関数のテスト生成には十分な品質を発揮します。大規模プロジェクトでは、複雑度の高い関数（サイクロマティック複雑度5以上）には Pro を、それ以外には Flash を使う「ハイブリッド戦略」がコストパフォーマンスに優れています。

### Q. コードレビューでの誤検知（False Positive）を減らすにはどうすればよいですか？

System Instructions でプロジェクト固有のルールやコーディング規約を指定すると、誤検知が大幅に減ります。たとえば「このプロジェクトでは Django ORM を使用しており、`objects.filter()` の結果を直接返すパターンは許容される」のように具体的な文脈を与えてください。また、temperature を 0.1 以下に設定すると、より保守的な（精度の高い）レビュー結果が得られます。

## まとめ

Gemini API のコード理解能力を活かすことで、テスト生成からコードレビュー、品質分析まで、開発プロセスの品質保証を大幅に自動化できます。

今回紹介したシステムの主なポイントは以下のとおりです。AST 解析で対象関数を構造的に抽出し、Gemini に的確なコンテキストを渡すこと。Structured Output で機械処理しやすい形式のテストコードを受け取ること。Function Calling で構造化されたレビュー結果を取得すること。そして CI/CD パイプラインに統合し、PR ごとに自動レビューを実行することです。

コスト面では、1ファイルあたりの解析・テスト生成で約 0.01〜0.05 USD 程度（Gemini 2.5 Pro の場合）なので、中規模プロジェクトでも十分に実用的です。まずは小さなモジュールから試して、チームの開発フローに段階的に組み込んでいくことをおすすめします。
