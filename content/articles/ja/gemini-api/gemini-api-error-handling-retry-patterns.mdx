## はじめに

Gemini API を使ったアプリケーションを本番環境で運用していると、ネットワーク障害や一時的なサーバーエラー、レート制限など、さまざまなエラーに遭遇します。開発中は手動でリトライすれば済みますが、本番環境ではユーザー体験に直結するため、堅牢なエラーハンドリングが不可欠です。

この記事では、Gemini API で発生する主要なエラーの種類を理解したうえで、**指数バックオフによるリトライ**、**サーキットブレーカーパターン**、**フォールバック戦略**といった本番運用に必要なパターンを、Python のコード例とともに解説します。

**この記事で学べること:**

- Gemini API のエラーコードと原因の整理
- 指数バックオフ付きリトライの正しい実装方法
- サーキットブレーカーで障害の連鎖を防ぐ方法
- モデルフォールバックで可用性を高める設計
- 本番環境でのログ・監視のベストプラクティス

**対象読者:** Gemini API を使った開発経験があり、本番環境への展開を検討している方

## Gemini API で発生する主なエラーの種類

Gemini API のエラーは、HTTP ステータスコードに基づいて大きく4つに分類できます。それぞれリトライすべきかどうかが異なるため、正しく判別することが重要です。

### リトライ可能なエラー（一時的な障害）

| ステータスコード | 意味 | 主な原因 |
|---|---|---|
| 429 | Too Many Requests | レート制限超過 |
| 500 | Internal Server Error | サーバー側の一時的な障害 |
| 503 | Service Unavailable | サービスの一時的な過負荷 |

これらは時間を置いて再試行すれば成功する可能性が高いエラーです。

### リトライ不可なエラー（設定・入力の問題）

| ステータスコード | 意味 | 主な原因 |
|---|---|---|
| 400 | Bad Request | 不正なリクエスト形式 |
| 401 / 403 | Unauthorized / Forbidden | API キーの無効・権限不足 |
| 404 | Not Found | 存在しないモデルやエンドポイント |

これらはリトライしても結果は変わりません。設定やリクエスト内容の修正が必要です。認証関連の問題については、[Gemini API 認証エラー FAQ](/articles/gemini-api/gemini-api-authentication-errors-faq) で詳しく解説しています。

## 指数バックオフ付きリトライの実装

リトライ可能なエラーに対しては、**指数バックオフ（Exponential Backoff）**が最も効果的なパターンです。リトライのたびに待機時間を指数的に増やし、ランダムなジッタ（揺らぎ）を加えることで、大量のクライアントが同時にリトライする「サンダリングハード問題」を防ぎます。

```python
import time
import random
import google.generativeai as genai
from google.api_core import exceptions

genai.configure(api_key="YOUR_API_KEY")

def call_gemini_with_retry(
    prompt: str,
    model_name: str = "gemini-3-flash",
    max_retries: int = 5,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
) -> str:
    """指数バックオフ付きでGemini APIを呼び出す関数"""
    model = genai.GenerativeModel(model_name)

    for attempt in range(max_retries + 1):
        try:
            response = model.generate_content(prompt)
            return response.text

        except exceptions.ResourceExhausted as e:
            # 429: レート制限 — リトライ対象
            if attempt == max_retries:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay * 0.5)
            wait_time = delay + jitter
            print(f"[Retry {attempt + 1}/{max_retries}] Rate limited. "
                  f"Waiting {wait_time:.1f}s...")
            time.sleep(wait_time)

        except exceptions.InternalServerError as e:
            # 500: サーバーエラー — リトライ対象
            if attempt == max_retries:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay * 0.3)
            wait_time = delay + jitter
            print(f"[Retry {attempt + 1}/{max_retries}] Server error. "
                  f"Waiting {wait_time:.1f}s...")
            time.sleep(wait_time)

        except exceptions.ServiceUnavailable as e:
            # 503: サービス一時停止 — リトライ対象
            if attempt == max_retries:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay * 0.3)
            wait_time = delay + jitter
            print(f"[Retry {attempt + 1}/{max_retries}] Service unavailable. "
                  f"Waiting {wait_time:.1f}s...")
            time.sleep(wait_time)

        except (exceptions.InvalidArgument, exceptions.PermissionDenied) as e:
            # 400, 403: リトライ不可 — 即座に例外を送出
            raise

# 使用例
try:
    result = call_gemini_with_retry("Pythonでリスト内包表記の利点を説明してください")
    print(result)
except Exception as e:
    print(f"全リトライ失敗: {e}")

# 期待する出力:
# リスト内包表記には以下の利点があります...
# (APIが一時エラーの場合はリトライ後に成功)
```

### リトライ実装のポイント

**ジッタ（Jitter）が重要な理由:** 多数のクライアントが同時にリトライすると、サーバーに再び負荷が集中します。ランダムな待機時間のばらつきを加えることで、リトライのタイミングを分散させます。

**最大待機時間の上限設定:** 指数バックオフでは待機時間が際限なく増えるため、`max_delay`（この例では60秒）で上限を設けます。

**リトライ回数の制限:** 無限リトライは危険です。5回程度に制限し、それでも失敗すればエラーとして上位に通知します。

## サーキットブレーカーパターン

リトライだけでは解決できない問題もあります。API が長時間ダウンしている場合、リトライを繰り返すとリソースを浪費し、アプリケーション全体の応答性が低下します。

**サーキットブレーカー**は、連続した失敗を検知すると一時的にリクエストを遮断し、一定時間後に回復を試みるパターンです。電気のブレーカーと同じ考え方です。

```python
import time
from enum import Enum
from dataclasses import dataclass, field

class CircuitState(Enum):
    CLOSED = "closed"       # 正常: リクエストを通す
    OPEN = "open"           # 遮断: リクエストをブロック
    HALF_OPEN = "half_open" # 試行: 一部リクエストを通す

@dataclass
class CircuitBreaker:
    """Gemini API用サーキットブレーカー"""
    failure_threshold: int = 5        # 連続失敗でOPENにする閾値
    recovery_timeout: float = 30.0    # OPEN→HALF_OPENまでの待機秒数
    success_threshold: int = 2        # HALF_OPEN→CLOSEDに必要な成功回数

    state: CircuitState = field(default=CircuitState.CLOSED)
    failure_count: int = field(default=0)
    success_count: int = field(default=0)
    last_failure_time: float = field(default=0.0)

    def can_execute(self) -> bool:
        """リクエストを実行してよいか判定する"""
        if self.state == CircuitState.CLOSED:
            return True
        elif self.state == CircuitState.OPEN:
            # 回復タイムアウトが経過したらHALF_OPENに移行
            if time.time() - self.last_failure_time >= self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                self.success_count = 0
                print("[Circuit] OPEN → HALF_OPEN: 回復を試行します")
                return True
            return False
        else:  # HALF_OPEN
            return True

    def record_success(self):
        """成功を記録する"""
        if self.state == CircuitState.HALF_OPEN:
            self.success_count += 1
            if self.success_count >= self.success_threshold:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                print("[Circuit] HALF_OPEN → CLOSED: 正常に回復しました")
        else:
            self.failure_count = 0

    def record_failure(self):
        """失敗を記録する"""
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.state == CircuitState.HALF_OPEN:
            self.state = CircuitState.OPEN
            print("[Circuit] HALF_OPEN → OPEN: 回復に失敗しました")
        elif self.failure_count >= self.failure_threshold:
            self.state = CircuitState.OPEN
            print(f"[Circuit] CLOSED → OPEN: {self.failure_count}回連続失敗")

# 使用例
circuit = CircuitBreaker(failure_threshold=3, recovery_timeout=30)

def safe_gemini_call(prompt: str) -> str:
    """サーキットブレーカー付きのAPI呼び出し"""
    if not circuit.can_execute():
        raise RuntimeError("Circuit OPEN: APIへのリクエストを一時停止中です")

    try:
        result = call_gemini_with_retry(prompt, max_retries=2)
        circuit.record_success()
        return result
    except Exception as e:
        circuit.record_failure()
        raise

# 期待する動作:
# 正常時 → そのままAPIを呼び出す
# 3回連続失敗 → Circuit OPEN (30秒間リクエスト遮断)
# 30秒後 → HALF_OPEN (試行)
# 2回成功 → CLOSED (正常復帰)
```

## モデルフォールバック戦略

Gemini API では複数のモデルが利用できるため、メインモデルが応答しない場合に別のモデルへフォールバックする戦略が有効です。たとえば Gemini 3 Pro で障害が発生した場合、Gemini 3 Flash に切り替えることで、品質は多少変わるものの、サービスの可用性は維持できます。

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class ModelConfig:
    name: str
    priority: int  # 数字が小さいほど優先

# モデルの優先順位リスト
FALLBACK_MODELS = [
    ModelConfig(name="gemini-3-pro", priority=1),
    ModelConfig(name="gemini-3-flash", priority=2),
    ModelConfig(name="gemini-2.5-flash", priority=3),
]

def call_with_fallback(
    prompt: str,
    models: list[ModelConfig] = FALLBACK_MODELS,
) -> tuple[str, str]:
    """フォールバック付きでGemini APIを呼び出す

    Returns:
        tuple: (レスポンステキスト, 使用したモデル名)
    """
    errors: list[str] = []

    for model_config in sorted(models, key=lambda m: m.priority):
        try:
            print(f"[Fallback] Trying model: {model_config.name}")
            result = call_gemini_with_retry(
                prompt,
                model_name=model_config.name,
                max_retries=2,  # フォールバック時はリトライ回数を抑える
            )
            return result, model_config.name

        except Exception as e:
            errors.append(f"{model_config.name}: {e}")
            print(f"[Fallback] {model_config.name} failed: {e}")
            continue

    raise RuntimeError(
        f"全モデルで失敗しました:\n" + "\n".join(errors)
    )

# 使用例
try:
    text, used_model = call_with_fallback("量子コンピュータの基礎を説明してください")
    print(f"応答モデル: {used_model}")
    print(text)
except RuntimeError as e:
    print(f"サービス利用不可: {e}")

# 期待する出力:
# [Fallback] Trying model: gemini-3-pro
# [Fallback] gemini-3-pro failed: 503 Service Unavailable
# [Fallback] Trying model: gemini-3-flash
# 応答モデル: gemini-3-flash
# 量子コンピュータとは...
```

## 本番環境でのログと監視

エラーハンドリングのコードを書いただけでは不十分です。本番環境では**何がどのくらいの頻度で起きているか**を可視化する仕組みが重要です。

### 構造化ログの実装

```python
import json
import logging
from datetime import datetime, timezone, timedelta

# JST タイムゾーン
JST = timezone(timedelta(hours=9))

# 構造化ログの設定
logger = logging.getLogger("gemini_api")
logger.setLevel(logging.INFO)
handler = logging.StreamHandler()
handler.setFormatter(logging.Formatter("%(message)s"))
logger.addHandler(handler)

def log_api_call(
    model: str,
    prompt_tokens: int,
    status: str,
    latency_ms: float,
    error: Optional[str] = None,
    retry_count: int = 0,
):
    """API呼び出しの結果を構造化ログとして記録する"""
    log_entry = {
        "timestamp": datetime.now(JST).isoformat(),
        "service": "gemini_api",
        "model": model,
        "prompt_tokens": prompt_tokens,
        "status": status,         # "success", "retry", "error"
        "latency_ms": round(latency_ms, 2),
        "retry_count": retry_count,
    }
    if error:
        log_entry["error"] = error

    logger.info(json.dumps(log_entry, ensure_ascii=False))

# 出力例:
# {"timestamp": "2026-03-24T20:15:00+09:00", "service": "gemini_api",
#  "model": "gemini-3-flash", "prompt_tokens": 128, "status": "success",
#  "latency_ms": 342.5, "retry_count": 0}
```

### 監視すべき主要メトリクス

本番環境では、以下のメトリクスをダッシュボードで常に監視することを推奨します。

- **エラー率（Error Rate）**: 全リクエストに対するエラーの割合。5%を超えたらアラート
- **リトライ率**: リトライが発生したリクエストの割合。頻発している場合はレート制限に近づいている兆候
- **レイテンシ P95/P99**: 95パーセンタイル・99パーセンタイルの応答時間
- **サーキットブレーカーの状態遷移**: OPEN状態の頻度と持続時間
- **フォールバック発動率**: メインモデル以外が使用された割合

レート制限に関するより詳しい対策は、[Gemini API レート制限＆クォータ管理ガイド](/articles/gemini-api/gemini-api-rate-limiting-quota-management)をご覧ください。

## FAQ

### Q1: リトライの最大回数は何回が適切ですか？

一般的には **3〜5回** が推奨されます。回数が少なすぎると一時的な障害を乗り越えられず、多すぎるとユーザーの待ち時間が長くなります。レート制限（429）の場合は5回、サーバーエラー（500/503）の場合は3回というように、エラーの種類に応じて分けるのも有効です。

### Q2: サーキットブレーカーの recovery_timeout はどのくらいが良いですか？

**30〜60秒** が一般的な出発点です。Gemini API のサーバー側障害は多くの場合数分以内に復旧するため、30秒程度で回復を試みるのがバランスの良い設定です。ただし、本番のトラフィックパターンに応じてチューニングしてください。

### Q3: フォールバックモデルを使った場合、出力品質の違いにはどう対処すべきですか？

フォールバックが発動したことをレスポンスのメタデータに含め、UI 側で「現在、代替モデルで応答しています」といった通知を表示する方法が一般的です。また、品質に影響が大きい処理（コード生成など）では、フォールバックを無効にして代わりにエラーメッセージを返すという選択肢もあります。

### Q4: 非同期処理でリトライパターンを実装するにはどうすればよいですか？

Python では `asyncio` と `tenacity` ライブラリの組み合わせが便利です。`tenacity` は `@retry` デコレータで指数バックオフやリトライ条件を宣言的に定義でき、非同期関数にも対応しています。

### Q5: ストリーミング応答でエラーが途中で発生した場合はどうすべきですか？

ストリーミング中のエラーでは、受信済みのチャンクを保持しつつ、中断ポイントからリトライするのが理想的です。ただし Gemini API のストリーミングはリクエスト単位の再実行になるため、**エラー検出→ストリーミングを最初からリトライ→受信済みチャンクを破棄して再受信** というフローが実用的です。

## まとめ

Gemini API を本番環境で安定運用するには、単純な try/catch だけでは不十分です。この記事で解説した3つのパターンを組み合わせることで、堅牢な API クライアントを構築できます。

1. **指数バックオフ付きリトライ** — 一時的なエラーを自動的に乗り越える
2. **サーキットブレーカー** — 長時間の障害でリソースを浪費しない
3. **モデルフォールバック** — 単一モデルの障害でもサービスを継続する

加えて、構造化ログと監視ダッシュボードを整備して、問題の早期検知と迅速な対応を可能にしましょう。
