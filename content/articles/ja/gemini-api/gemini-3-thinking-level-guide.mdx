# Gemini 3の「thinking_level」パラメータ完全ガイド：推論深度を自在に制御する

Google が 2026年初頭にリリースした **Gemini 3** では、モデルが応答を生成する前に行う「内部思考」の深さを開発者が直接コントロールできる新しいAPIパラメータ **`thinking_level`** が導入されました。本記事では、このパラメータの仕組み・設定値の違い・活用シナリオ・実装コードを徹底解説します。

---

## thinking_level とは何か？

Gemini 3 は応答を生成する前に、問題を内部で段階的に分析する「思考プロセス」を持っています。`thinking_level` はこの思考プロセスの**最大深度**を制御するパラメータです。

Gemini 3 はこれらの設定値を「トークン数の絶対的な上限」としてではなく、**推論の深さに関する相対的なガイドライン**として扱います。つまり、同じ `"high"` を指定しても問題の難易度によって実際の思考量は変化します。

### 設定できる値

| 値 | 推論深度 | 主な用途 |
|---|---|---|
| `"low"` | 浅い（高速） | シンプルな質問・レイテンシ重視のアプリ |
| `"medium"` | 中程度（バランス型） | 一般的なタスク・デフォルト推奨 |
| `"high"` | 深い（高精度） | 複雑な推論・数学・コーディング |

---

## なぜ thinking_level が重要なのか？

### コストとレイテンシのトレードオフ

内部思考は非常に強力ですが、その分トークン消費量とレイテンシが増大します。`thinking_level` を適切に設定することで：

- **コスト削減**：単純なタスクに `"low"` を使うことで不要なトークンを消費しない
- **速度向上**：`"low"` では複雑な思考プロセスをスキップし、素早く応答
- **精度向上**：複雑な問題には `"high"` を使って最大限の推論能力を引き出す

### PHDレベルの推論能力

Gemini 3 Flash であっても、`thinking_level: "high"` を設定することで GPQA Diamond ベンチマーク（90.4%）や Humanity's Last Exam（33.7%）といったPHDレベルの推論問題に対応できます。

## 実装コード例：thinking_level の基本的な使い方

以下のコードは Python で Gemini 3 API を呼び出し、`thinking_level` を動的に切り替える例です。

```python
import google.generativeai as genai
import os
import time

# APIキーの設定
genai.configure(api_key=os.environ["GEMINI_API_KEY"])

def ask_gemini(prompt: str, thinking_level: str = "medium") -> dict:
    """
    Gemini 3 に質問を送り、thinking_level に応じた回答を返す関数。

    Args:
        prompt: ユーザーからの質問文
        thinking_level: 推論深度。"low" / "medium" / "high" のいずれか

    Returns:
        回答テキストと所要時間を含む辞書
    """
    model = genai.GenerativeModel(
        model_name="gemini-3-flash",
        generation_config={
            "thinking_level": thinking_level,  # 推論深度を指定
        }
    )

    start = time.time()
    response = model.generate_content(prompt)
    elapsed = time.time() - start

    return {
        "text": response.text,
        "thinking_level": thinking_level,
        "elapsed_sec": round(elapsed, 2),
    }


# ── テスト実行 ──────────────────────────────────────
if __name__ == "__main__":
    # シンプルな質問 → "low" で十分
    simple_q = "今日の東京の天気は？"
    result_low = ask_gemini(simple_q, thinking_level="low")
    print(f"[LOW]    {result_low['elapsed_sec']}秒 → {result_low['text'][:80]}")

    # 複雑な数学問題 → "high" で精度を最大化
    hard_q = "1から1000までの整数のうち、3の倍数でも5の倍数でもない数の合計を求めよ。"
    result_high = ask_gemini(hard_q, thinking_level="high")
    print(f"[HIGH]   {result_high['elapsed_sec']}秒 → {result_high['text'][:80]}")

# 期待出力例:
# [LOW]    0.43秒 → 東京の最新天気情報は...
# [HIGH]   2.87秒 → 1から1000のうち、3の倍数の個数は333個、5の倍数は200個...
```

## Thought Signatures との連携

Gemini 3 では **Thought Signatures（思考シグネチャ）** も同時に導入されました。これはモデルの内部思考プロセスの暗号化された表現で、複数ターンの会話でも推論の連続性を保つ仕組みです。

`thinking_level` と Thought Signatures を組み合わせると、**複雑なエージェント型ワークフロー**で特に強力になります。

```python
import google.generativeai as genai
import os

genai.configure(api_key=os.environ["GEMINI_API_KEY"])

model = genai.GenerativeModel(
    model_name="gemini-3-flash",
    generation_config={
        "thinking_level": "high",  # 深い推論で一貫性を保つ
    }
)

chat = model.start_chat()

# ターン1: 初回の質問
response1 = chat.send_message("量子コンピュータのShorアルゴリズムを説明して。")

# Thought Signatures はSDKが自動的に次のターンへ引き継ぐ
# → ターン2ではターン1の思考プロセスを踏まえた応答が返る
response2 = chat.send_message(
    "では、そのアルゴリズムをPythonの疑似コードで実装して。"
)

print(response2.text)

# 期待出力例:
# ターン1の説明内容を踏まえた、一貫性のあるPython疑似コードが返る
```

### Thought Signatures の重要な注意点

- Thought Signatures は**同じモデルセッション内でのみ有効**
- 異なるモデルバージョンへ引き継ぐことはできない
- Gemini 3 以降でのみサポート（Gemini 2.x には非対応）

## ユースケース別の推奨 thinking_level

### `"low"` が適しているケース

- チャットボットの軽い雑談
- 単純な翻訳・要約タスク
- リアルタイム性が求められるストリーミング応答
- レイテンシが1秒未満であることが要件の場合

### `"medium"` が適しているケース（デフォルト推奨）

- 一般的なビジネス文書の作成
- コードレビューやバグの軽微な修正
- FAQ の回答生成
- マーケティングコンテンツの作成

### `"high"` が適しているケース

- 数学的な証明・計算問題
- 複雑なアルゴリズムの設計・実装
- 法律・医療・科学分野の専門的な分析
- 多段階推論が必要なエージェント型タスク
- Gemini Deep Research との連携

## コストの比較

`thinking_level` はトークン消費量に直接影響します。以下はおおよその目安です（実際の消費量はプロンプトと問題の複雑さによって変化）。

| 設定値 | 相対トークン消費量 | 相対レイテンシ |
|---|---|---|
| `"low"` | ×1（基準） | ×1（基準） |
| `"medium"` | ×2〜3 | ×2〜3 |
| `"high"` | ×5〜10 | ×5〜10 |

コスト最適化のポイントは、**タスクの複雑さを事前に分類し、適切なレベルを動的に選択する**ことです。

```python
def classify_and_ask(prompt: str) -> str:
    """
    プロンプトの複雑さを判定し、最適な thinking_level を選んで回答を返す。
    """
    # キーワードベースで複雑さを簡易判定
    complex_keywords = ["証明", "アルゴリズム", "最適化", "微分", "積分", "実装", "設計"]
    simple_keywords = ["翻訳", "要約", "天気", "日付", "挨拶"]

    if any(kw in prompt for kw in complex_keywords):
        level = "high"
    elif any(kw in prompt for kw in simple_keywords):
        level = "low"
    else:
        level = "medium"

    result = ask_gemini(prompt, thinking_level=level)
    print(f"→ thinking_level='{level}' を選択（{result['elapsed_sec']}秒）")
    return result["text"]

# 期待出力例:
# → thinking_level='high' を選択（3.12秒）
```

## FAQ

### Q1. thinking_level を設定しなかった場合のデフォルト動作は？

`thinking_level` を省略した場合、Gemini 3 は内部的にタスクの複雑さを判断し、自動的に適切な推論深度を選択します。ただし、コストやレイテンシを最適化したい場合は明示的に指定することを推奨します。

### Q2. thinking_level は Gemini 2.x でも使えますか？

いいえ、`thinking_level` パラメータは **Gemini 3 以降**でのみサポートされています。Gemini 2.x では無視されるか、APIエラーが返ります。Gemini 3 Flash または Gemini 3 Pro 以降のモデルを使用してください。

### Q3. thinking_level="high" にすると必ず正確な答えが返りますか？

`"high"` は推論の深さを最大化しますが、完全な正確性を**保証するものではありません**。特に最新情報が必要なタスクでは、`grounding_with_google_search` と組み合わせることで精度をさらに高められます。複雑な計算問題では、コードインタープリタツールとの併用も効果的です。
