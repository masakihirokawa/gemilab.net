## はじめに：Thought Signaturesとは何か

Gemini 3 の登場とともに、API に重要な新機能 **Thought Signatures（思考シグネチャ）** が導入されました。これはモデルの内部推論プロセスを暗号化した表現であり、複数ターンにわたるエージェントワークフローで推論の連続性を保つための仕組みです。

従来の Gemini API はステートレスなリクエスト／レスポンス型でした。各呼び出しは独立しており、前のターンで「なぜその判断をしたか」という文脈は次のターンに引き継がれませんでした。Thought Signatures はその課題を解決します。

### この記事で学べること

- Thought Signatures の概念と必要性
- Gemini 3 API での Thought Signatures の扱い方
- Python SDK を使った実装例（Function Calling、マルチターン）
- よくあるエラーと対処法
- Thought Signatures を活用した高度なエージェント設計

**対象読者**: Gemini API を使って開発している Python エンジニア。Function Calling やエージェントワークフローの構築経験があると理解が深まります。

## Thought Signatures の仕組み

### 推論の連続性とは

Gemini 3 はユーザーからのリクエストに応じる前に、内部で深く推論（Thinking）を行います。この推論プロセスは通常、レスポンスの `thought` フィールドに含まれますが、暗号化された Thought Signature として返される場合もあります。

```
ユーザーの質問
  → Gemini 3 内部推論（Thinking）
    → 推論結果を Thought Signature として暗号化
      → テキスト応答を生成
        → レスポンスに Thought Signature + テキストが含まれる
```

次のターンで **Thought Signature をそのままモデルに返す**ことで、Gemini 3 は前の推論の文脈を継続できます。これが「ステートフルな推論」の核心です。

### なぜ Gemini 3 では必須なのか

Gemini 3 では、Function Calling を使うワークフローで Thought Signatures の返却が**強制**されます。返却しないと API から 4xx エラーが返ります。これは設計上の仕様であり、エージェントの推論一貫性を保証するためのものです。

公式ドキュメントによると：

> *「Thought Signature を受け取った場合は、次のターンで会話履歴の一部としてそのままモデルへ返してください。」*

## ステップバイステップ実装ガイド

### ケース1：シンプルなマルチターン会話（SDK が自動処理）

Python SDK の `ChatSession` を使う場合、Thought Signatures は SDK が自動的に管理します。開発者は特別な操作は不要です。

```python
import os
from google import genai

client = genai.Client(api_key=os.environ["GOOGLE_API_KEY"])

# ChatSession を使うと Thought Signatures は自動管理される
chat = client.chats.create(model="gemini-3.1-pro-preview")

# 1ターン目
response1 = chat.send_message("Pythonのデコレータについて説明してください。")
print("ターン1:", response1.text[:200])
# 期待する出力:
# ターン1: Pythonのデコレータは、関数やクラスの動作を変更・拡張するための構文糖衣です...

# 2ターン目（前の推論文脈が自動で引き継がれる）
response2 = chat.send_message("具体的なコード例を見せてください。")
print("ターン2:", response2.text[:200])
# 期待する出力:
# ターン2: 先ほど説明したデコレータを使った実例です。以下のコードをご覧ください...
```

`ChatSession` を使う限り、Thought Signatures の処理は SDK に任せられます。

### ケース2：Function Calling での手動実装

Function Calling を手動で実装する場合は Thought Signatures を適切に処理する必要があります。以下は天気情報を取得するツールを持つエージェントの例です。

```python
import os
from google import genai
from google.genai import types

client = genai.Client(api_key=os.environ["GOOGLE_API_KEY"])

# ツール定義：天気情報を取得する関数
weather_tool = types.Tool(
    function_declarations=[
        types.FunctionDeclaration(
            name="get_weather",
            description="指定した都市の現在の天気を取得します",
            parameters=types.Schema(
                type="object",
                properties={
                    "city": types.Schema(
                        type="string",
                        description="天気を調べる都市名（例: Tokyo, Osaka）"
                    )
                },
                required=["city"]
            )
        )
    ]
)

# ダミーの天気取得関数
def get_weather(city: str) -> dict:
    weather_data = {
        "Tokyo": {"temp": 18, "condition": "晴れ", "humidity": 45},
        "Osaka": {"temp": 20, "condition": "曇り", "humidity": 60},
    }
    return weather_data.get(city, {"temp": 15, "condition": "不明", "humidity": 50})

# 初回リクエスト
contents = [
    types.Content(
        role="user",
        parts=[types.Part(text="東京と大阪の今日の天気を教えてください。")]
    )
]

response = client.models.generate_content(
    model="gemini-3.1-pro-preview",
    contents=contents,
    config=types.GenerateContentConfig(tools=[weather_tool])
)

# レスポンスを会話履歴に追加（Thought Signatures を含む model パートをそのまま保存）
contents.append(
    types.Content(role="model", parts=response.candidates[0].content.parts)
)

# Function Calls を処理
function_response_parts = []
for part in response.candidates[0].content.parts:
    if part.function_call:
        fc = part.function_call
        print(f"ツール呼び出し: {fc.name}({dict(fc.args)})")
        # 期待する出力例:
        # ツール呼び出し: get_weather({'city': 'Tokyo'})
        # ツール呼び出し: get_weather({'city': 'Osaka'})

        result = get_weather(**fc.args)
        function_response_parts.append(
            types.Part(
                function_response=types.FunctionResponse(
                    name=fc.name,
                    response=result
                )
            )
        )

# ツールの実行結果を会話履歴に追加
if function_response_parts:
    contents.append(
        types.Content(role="user", parts=function_response_parts)
    )

    # 最終レスポンスを取得
    final_response = client.models.generate_content(
        model="gemini-3.1-pro-preview",
        contents=contents,
        config=types.GenerateContentConfig(tools=[weather_tool])
    )

    print("最終応答:", final_response.text)
    # 期待する出力:
    # 最終応答: 本日の天気をお伝えします。
    # 東京：気温18℃、晴れ、湿度45%です。
    # 大阪：気温20℃、曇り、湿度60%となっています。
```

**ポイント**: `response.candidates[0].content.parts` をそのまま `model` ロールのパートとして会話履歴に追加することで、Thought Signatures が自動的に次のターンに引き渡されます。`parts` を個別に加工したり、テキスト部分だけを保存したりすると Thought Signatures が失われるので注意してください。

### ケース3：並列 Function Calls での Thought Signatures

複数のツールが並列で呼ばれる場合、**最初の Function Call パートに Thought Signature が含まれます**。後続のパートには含まれないため、すべての `parts` をまとめて保存することが重要です。

```python
# 並列 Function Calls のレスポンス処理
model_parts = response.candidates[0].content.parts

# ✅ 正しい方法：すべての parts をそのまま保存
contents.append(types.Content(role="model", parts=model_parts))

# ❌ 間違い：Function Call パートだけを抜き出す
# fc_only = [p for p in model_parts if p.function_call]
# contents.append(types.Content(role="model", parts=fc_only))  # Thought Signature が消える！
```

## よくあるエラーと対処法

### エラー1：400 Bad Request（INVALID_ARGUMENT）

**原因**: Thought Signature が会話履歴に含まれていない、または形式が変わっている。

**対処法**: `response.candidates[0].content.parts` を加工せず、そのまま `model` ロールとして `contents` に追加する。

```python
# ❌ NG：text のみ抽出
contents.append(types.Content(role="model", parts=[types.Part(text=response.text)]))

# ✅ OK：parts をそのまま使用
contents.append(types.Content(role="model", parts=response.candidates[0].content.parts))
```

### エラー2：Thought Signature がレスポンスに含まれない

**原因**: `thinking_level` を `none` に設定している場合、Thought Signatures は生成されません。エージェントワークフローでは `thinking_level` を省略（デフォルトの `high`）か `low` 以上に設定してください。

詳細は [Gemini 3 Thinking Level ガイド](/articles/gemini-api/gemini-3-thinking-level-guide) を参照してください。

### エラー3：長いワークフローでレスポンスが劣化する

**原因**: Function Call の連鎖が長くなると、Thought Signatures を含む会話履歴がトークン上限に近づく。

**対処法**: [Context Caching](/articles/gemini-api/context-caching-guide) を活用してシステムプロンプトや固定コンテンツをキャッシュし、実際の会話履歴トークン数を削減する。

## 応用：Thought Signatures を活用した本番エージェント設計

### 推奨アーキテクチャ

本番環境でエージェントを構築する際のベストプラクティスをまとめます。

1. **SDK の ChatSession を優先使用** — Thought Signatures の管理を SDK に委ねることで実装ミスを防ぐ
2. **手動実装が必要な場合は `content.parts` を丸ごと保存** — 個別の part を選別しない
3. **並列ツール呼び出しを積極活用** — Gemini 3 は並列 Function Calls をサポートし、複数の情報を同時取得できる
4. **Interactions API との組み合わせ** — より高度なセッション管理には [Gemini Interactions API](/articles/gemini-api/gemini-interactions-api-guide) の利用を検討する

### ログとデバッグ

Thought Signatures はエンドユーザーへのレスポンスには表示されませんが、デバッグのために内部でログを取ることをお勧めします。

```python
# Thought Signatures の存在確認（デバッグ用）
for i, part in enumerate(response.candidates[0].content.parts):
    has_thought = hasattr(part, 'thought') and part.thought
    has_fc = part.function_call is not None
    has_text = bool(part.text)
    print(f"Part {i}: thought={has_thought}, function_call={has_fc}, text={has_text}")
    # 期待する出力例（思考あり＋Function Call の場合）:
    # Part 0: thought=True, function_call=True, text=False
    # Part 1: thought=False, function_call=True, text=False
```

## FAQ

**Q1. Thought Signatures を手動で読み取ることはできますか？**

A. いいえ。Thought Signatures は暗号化された不透明なデータであり、内容を直接読み取ることはできません。モデルへの引き渡し専用の値として扱ってください。

**Q2. Gemini 2.5 系のモデルでも Thought Signatures は必要ですか？**

A. Thought Signatures は **Gemini 3 以降で導入**された仕様です。Gemini 2.5 Pro や 2.5 Flash を使用する場合は不要です。ただし今後 Gemini 3 系に移行する場合はこの対応が必要になります。

**Q3. REST API を直接使う場合はどう処理しますか？**

A. REST API でも同様に、モデルレスポンスの `content.parts` 配列をそのまま次のリクエストの `contents` に含めてください。`parts` 内の各オブジェクトに Thought Signature フィールドが含まれている場合は、それを変更・削除しないようにします。

**Q4. Thought Signatures はコスト（トークン数）に影響しますか？**

A. Thought Signatures は内部的にトークンを消費しますが、Gemini API では思考トークンは別途カウントされる仕組みです。[Gemini API コスト最適化](/articles/gemini-api/gemini-api-cost-optimization) の記事で詳しく解説しています。

**Q5. Function Calling を使わないシンプルな Q&A では Thought Signatures を気にする必要はありますか？**

A. SDK の `ChatSession` を使う限りは不要です。ただし `generate_content` を直接呼び出してマルチターン会話を手動管理する場合は、会話履歴に `parts` をそのまま含めることで自然に Thought Signatures が保持されます。

## まとめ

Gemini 3 の Thought Signatures は、エージェントAI開発における重要な新機能です。

- **何をするか**: モデルの推論プロセスを暗号化し、複数ターンで推論の連続性を保つ
- **いつ重要か**: Gemini 3 + Function Calling のマルチターンワークフロー
- **実装の要点**: `response.candidates[0].content.parts` をそのまま `model` ロールとして保存する
- **SDK を使えば**: `ChatSession` が自動管理するため、基本的に意識不要

Gemini 3 への移行を検討している開発者は、特に Function Calling を使っているコードを早めに確認し、Thought Signatures への対応を進めてください。より高度なエージェント設計については [Gemini Interactions API ガイド](/articles/gemini-api/gemini-interactions-api-guide) もあわせてご覧ください。
