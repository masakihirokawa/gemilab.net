## はじめに — キーワード検索の限界とセマンティック検索

従来のキーワード検索では「意味的に近い」ドキュメントを見つけることが難しいという課題がありました。たとえば「犬の散歩で気をつけること」と検索しても、「ペットの外出時の安全対策」という記事は引っかかりません。

セマンティック検索は、テキストの「意味」をベクトル（数値の配列）に変換し、ベクトル間の距離で類似度を計算することで、この問題を解決します。Gemini の Embedding API は、768次元または3072次元の高品質なベクトルを生成でき、多言語対応も優れています。

この記事では、**Gemini Embedding API** と **PostgreSQL pgvector** を組み合わせて、プロダクション品質のセマンティック検索エンジンを構築する方法を、ステップバイステップで解説します。

### この記事で学べること

- Gemini Embedding API（`text-embedding-004` / `gemini-embedding-exp-03-07`）の使い方
- PostgreSQL pgvector 拡張のセットアップとインデックス戦略
- バッチエンベディング生成と効率的なDB格納
- FastAPI によるセマンティック検索 API の構築
- HNSW / IVFFlat インデックスによるパフォーマンス最適化

**対象読者**: Gemini API の基本を理解しており、Python と SQL の経験がある中〜上級開発者

## Gemini Embedding API でベクトルを生成する

Gemini の Embedding API は、テキストを高品質なベクトルに変換します。`task_type` パラメータを指定することで、検索やクラスタリングなど用途に応じた最適なベクトルを生成できます。

```python
# embed_documents.py — Gemini Embedding API でテキストをベクトル化
from google import genai
import numpy as np

# クライアント初期化
client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

def generate_embedding(text: str, task_type: str = "RETRIEVAL_DOCUMENT") -> list[float]:
    """
    テキストからエンベディングベクトルを生成する。

    Args:
        text: ベクトル化するテキスト
        task_type: "RETRIEVAL_DOCUMENT"（格納用）or "RETRIEVAL_QUERY"（検索用）

    Returns:
        768次元の浮動小数点数リスト
    """
    result = client.models.embed_content(
        model="text-embedding-004",
        contents=text,
        config={
            "task_type": task_type,
            "output_dimensionality": 768  # 256〜3072 で指定可能
        }
    )
    return result.embeddings[0].values

# 使用例
doc_text = "PostgreSQLは高機能なオープンソースのリレーショナルデータベースです"
vector = generate_embedding(doc_text)
print(f"ベクトル次元数: {len(vector)}")  # 出力: ベクトル次元数: 768
print(f"先頭5要素: {vector[:5]}")
# 出力例: 先頭5要素: [0.0234, -0.0156, 0.0412, -0.0089, 0.0301]
```

### task_type の使い分け

| task_type | 用途 | 使用場面 |
|-----------|------|----------|
| `RETRIEVAL_DOCUMENT` | ドキュメント格納時 | DB にベクトルを保存する際 |
| `RETRIEVAL_QUERY` | 検索クエリ時 | ユーザーの検索入力をベクトル化する際 |
| `SEMANTIC_SIMILARITY` | 類似度計算 | 2つのテキストの意味的距離を測る際 |
| `CLASSIFICATION` | テキスト分類 | カテゴリ分けに使う際 |
| `CLUSTERING` | クラスタリング | グループ分けに使う際 |

> **ポイント**: ドキュメント格納時は `RETRIEVAL_DOCUMENT`、検索クエリ時は `RETRIEVAL_QUERY` を使い分けることで、検索精度が大幅に向上します。

## バッチ処理で大量ドキュメントを効率的にベクトル化

実際のプロダクション環境では、数千〜数万件のドキュメントを処理する必要があります。Gemini API のバッチエンベディング機能を活用して効率的に処理しましょう。

```python
# batch_embed.py — バッチ処理で大量ドキュメントをベクトル化＆DB格納
import psycopg2
from google import genai
import time

client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

def batch_embed_and_store(documents: list[dict], batch_size: int = 100):
    """
    ドキュメントをバッチ処理でベクトル化し、PostgreSQL に格納する。

    Args:
        documents: [{"title": "...", "content": "..."}] のリスト
        batch_size: 1回のAPI呼び出しで処理する件数
    """
    conn = psycopg2.connect(
        host="localhost",
        database="semantic_search",
        user="postgres",
        password="your_password"
    )
    cur = conn.cursor()

    for i in range(0, len(documents), batch_size):
        batch = documents[i:i + batch_size]
        texts = [doc["content"] for doc in batch]

        # バッチエンベディング生成
        result = client.models.embed_content(
            model="text-embedding-004",
            contents=texts,
            config={
                "task_type": "RETRIEVAL_DOCUMENT",
                "output_dimensionality": 768
            }
        )

        # DB に格納
        for doc, emb in zip(batch, result.embeddings):
            cur.execute(
                """
                INSERT INTO documents (title, content, embedding)
                VALUES (%s, %s, %s::vector)
                """,
                (doc["title"], doc["content"], str(emb.values))
            )

        conn.commit()
        print(f"  バッチ {i // batch_size + 1}: {len(batch)} 件を格納完了")

        # レートリミット対策（1,500 RPM を想定）
        time.sleep(0.5)

    cur.close()
    conn.close()
    print(f"全 {len(documents)} 件のベクトル化と格納が完了しました")

# 使用例
sample_docs = [
    {"title": "PostgreSQL入門", "content": "PostgreSQLは世界で最も先進的なオープンソースデータベースです..."},
    {"title": "ベクトル検索とは", "content": "ベクトル検索はテキストの意味的類似性に基づいて検索する手法です..."},
    {"title": "Python API開発", "content": "FastAPIを使うとPythonで高速なAPIサーバーを構築できます..."},
]
batch_embed_and_store(sample_docs)
# 出力:
#   バッチ 1: 3 件を格納完了
# 全 3 件のベクトル化と格納が完了しました
```

## pgvector インデックス戦略 — HNSW vs IVFFlat

ベクトル検索のパフォーマンスは、インデックスの選択によって大きく変わります。pgvector は2つの主要なインデックスタイプを提供しています。

### HNSW（Hierarchical Navigable Small World）

高い検索精度と安定したパフォーマンスを提供します。**プロダクション環境での第一選択**です。

```sql
-- HNSW インデックスを作成（推奨設定）
CREATE INDEX idx_documents_embedding_hnsw
ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 200);

-- 検索時のパラメータ調整
SET hnsw.ef_search = 100;  -- 高いほど精度↑、速度↓
```

### IVFFlat（Inverted File with Flat Compression）

インデックス構築が高速で、大規模データセットに向いています。ただし、検索精度は HNSW に劣る場合があります。

```sql
-- IVFFlat インデックスを作成
-- lists の推奨値: sqrt(行数) 〜 行数/1000
CREATE INDEX idx_documents_embedding_ivfflat
ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- 検索時のプローブ数を調整
SET ivfflat.probes = 10;  -- 高いほど精度↑、速度↓
```

### パフォーマンス比較

| 指標 | HNSW | IVFFlat |
|------|------|---------|
| 検索速度（10万件） | 〜5ms | 〜3ms |
| 再現率（Recall@10） | 99.2% | 95.8% |
| インデックス構築時間 | 長い | 短い |
| メモリ使用量 | 多い | 少ない |
| 推奨データ規模 | 〜100万件 | 100万件〜 |

> **推奨**: 100万件以下のデータセットでは HNSW を選択してください。検索精度の差が最終的なユーザー体験に直結します。

## FastAPI でセマンティック検索 API を構築する

ここまでの要素を組み合わせて、FastAPI でプロダクション対応のセマンティック検索 API を構築します。

```python
# api.py — FastAPI セマンティック検索 API
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field
import psycopg2
from psycopg2.extras import RealDictCursor
from google import genai
from contextlib import asynccontextmanager

# --- 設定 ---
GEMINI_API_KEY = "YOUR_GEMINI_API_KEY"
DB_CONFIG = {
    "host": "localhost",
    "database": "semantic_search",
    "user": "postgres",
    "password": "your_password"
}

client = genai.Client(api_key=GEMINI_API_KEY)

@asynccontextmanager
async def lifespan(app: FastAPI):
    """アプリケーション起動時にDB接続を確認"""
    conn = psycopg2.connect(**DB_CONFIG)
    conn.close()
    print("DB接続確認完了")
    yield

app = FastAPI(
    title="Semantic Search API",
    description="Gemini Embedding + pgvector によるセマンティック検索",
    version="1.0.0",
    lifespan=lifespan
)

class SearchRequest(BaseModel):
    query: str = Field(..., min_length=1, max_length=1000, description="検索クエリ")
    top_k: int = Field(default=5, ge=1, le=50, description="返却する結果数")
    similarity_threshold: float = Field(
        default=0.7, ge=0.0, le=1.0,
        description="類似度の閾値（コサイン類似度）"
    )

class SearchResult(BaseModel):
    id: int
    title: str
    content: str
    similarity: float
    metadata: dict

@app.post("/search", response_model=list[SearchResult])
async def semantic_search(request: SearchRequest):
    """セマンティック検索を実行する"""
    try:
        # クエリをベクトル化（task_type を RETRIEVAL_QUERY に設定）
        result = client.models.embed_content(
            model="text-embedding-004",
            contents=request.query,
            config={
                "task_type": "RETRIEVAL_QUERY",
                "output_dimensionality": 768
            }
        )
        query_vector = result.embeddings[0].values

        # pgvector でコサイン類似度検索
        conn = psycopg2.connect(**DB_CONFIG, cursor_factory=RealDictCursor)
        cur = conn.cursor()

        cur.execute(
            """
            SELECT
                id, title, content, metadata,
                1 - (embedding <=> %s::vector) AS similarity
            FROM documents
            WHERE 1 - (embedding <=> %s::vector) >= %s
            ORDER BY embedding <=> %s::vector
            LIMIT %s
            """,
            (
                str(query_vector),
                str(query_vector),
                request.similarity_threshold,
                str(query_vector),
                request.top_k
            )
        )

        results = cur.fetchall()
        cur.close()
        conn.close()

        return [
            SearchResult(
                id=row["id"],
                title=row["title"],
                content=row["content"][:500],  # プレビュー用に切り詰め
                similarity=round(float(row["similarity"]), 4),
                metadata=row["metadata"] or {}
            )
            for row in results
        ]

    except Exception as e:
        raise HTTPException(status_code=500, detail=f"検索エラー: {str(e)}")

@app.post("/index")
async def index_document(title: str, content: str):
    """新しいドキュメントをインデックスに追加する"""
    try:
        result = client.models.embed_content(
            model="text-embedding-004",
            contents=content,
            config={
                "task_type": "RETRIEVAL_DOCUMENT",
                "output_dimensionality": 768
            }
        )
        embedding = result.embeddings[0].values

        conn = psycopg2.connect(**DB_CONFIG)
        cur = conn.cursor()
        cur.execute(
            """
            INSERT INTO documents (title, content, embedding)
            VALUES (%s, %s, %s::vector)
            RETURNING id
            """,
            (title, content, str(embedding))
        )
        doc_id = cur.fetchone()[0]
        conn.commit()
        cur.close()
        conn.close()

        return {"id": doc_id, "message": "ドキュメントをインデックスに追加しました"}

    except Exception as e:
        raise HTTPException(status_code=500, detail=f"インデックスエラー: {str(e)}")

# 起動: uvicorn api:app --host 0.0.0.0 --port 8000
# 出力: INFO: Uvicorn running on http://0.0.0.0:8000
```

### 2. エンベディングキャッシュ

同じクエリの繰り返しに対応するため、Redis でエンベディングをキャッシュします。

```python
# embedding_cache.py — Redis によるエンベディングキャッシュ
import redis
import json
import hashlib

redis_client = redis.Redis(host="localhost", port=6379, db=0)
CACHE_TTL = 3600  # 1時間

def get_cached_embedding(text: str, task_type: str) -> list[float] | None:
    """キャッシュからエンベディングを取得"""
    key = f"emb:{hashlib.sha256(f'{text}:{task_type}'.encode()).hexdigest()}"
    cached = redis_client.get(key)
    if cached:
        return json.loads(cached)
    return None

def cache_embedding(text: str, task_type: str, embedding: list[float]):
    """エンベディングをキャッシュに保存"""
    key = f"emb:{hashlib.sha256(f'{text}:{task_type}'.encode()).hexdigest()}"
    redis_client.setex(key, CACHE_TTL, json.dumps(embedding))
```

### 3. ハイブリッド検索（セマンティック + 全文検索）

セマンティック検索と PostgreSQL の全文検索を組み合わせることで、より高い検索精度を実現できます。

```sql
-- 全文検索用のインデックスを追加
ALTER TABLE documents ADD COLUMN tsv tsvector
    GENERATED ALWAYS AS (to_tsvector('japanese', content)) STORED;
CREATE INDEX idx_documents_tsv ON documents USING gin(tsv);

-- ハイブリッド検索クエリ（RRF: Reciprocal Rank Fusion）
WITH semantic AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $1::vector) AS rank_s
    FROM documents
    ORDER BY embedding <=> $1::vector
    LIMIT 20
),
fulltext AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(tsv, query) DESC) AS rank_f
    FROM documents, plainto_tsquery('japanese', $2) query
    WHERE tsv @@ query
    LIMIT 20
)
SELECT d.id, d.title, d.content,
    COALESCE(1.0 / (60 + s.rank_s), 0) +
    COALESCE(1.0 / (60 + f.rank_f), 0) AS rrf_score
FROM documents d
LEFT JOIN semantic s ON d.id = s.id
LEFT JOIN fulltext f ON d.id = f.id
WHERE s.id IS NOT NULL OR f.id IS NOT NULL
ORDER BY rrf_score DESC
LIMIT 10;
```

## よくある質問（FAQ）

### Q1: text-embedding-004 と gemini-embedding-exp-03-07 のどちらを使うべきですか？

`text-embedding-004` は安定版で、768次元のベクトルを生成します。プロダクション環境での利用に推奨されます。`gemini-embedding-exp-03-07` は実験版で、最大3072次元のベクトルを生成でき、より高い精度を提供しますが、API の仕様が変更される可能性があります。安定性を重視するなら `text-embedding-004`、精度を最大化したいなら実験版を検討してください。

### Q2: ベクトルの次元数は768と3072のどちらが良いですか？

768次元はストレージとパフォーマンスのバランスが良く、多くのユースケースで十分な精度を提供します。3072次元は精度が向上しますが、ストレージ使用量が約4倍、検索速度も低下します。100万件以上のデータセットでは768次元を推奨します。

### Q3: pgvector の代わりに専用のベクトルデータベース（Pinecone、Weaviate など）を使うべきですか？

既に PostgreSQL を運用している場合は、pgvector で十分なケースが多いです。インフラの複雑さを増やさずにベクトル検索を導入できる点が大きなメリットです。ただし、10億件規模のデータや、マルチテナント対応が必要な場合は、専用のベクトルデータベースが適しています。

### Q4: エンベディングの更新頻度はどのくらいが適切ですか？

ドキュメントの内容が変更された場合のみ再ベクトル化すれば十分です。Gemini のエンベディングモデルがアップデートされた場合は、全ドキュメントの再ベクトル化を検討してください。ただし、異なるモデルバージョンのベクトルが混在すると検索精度が低下するため、再ベクトル化は一括で行うことを推奨します。

### Q5: セマンティック検索の精度を評価するにはどうすればよいですか？

Recall@K（上位K件に正解が含まれる割合）と MRR（Mean Reciprocal Rank）が代表的な指標です。評価用のクエリ・正解ペアのデータセットを作成し、定期的にベンチマークを実行することをお勧めします。

## まとめ — セマンティック検索で実現する次世代の情報探索

この記事では、Gemini Embedding API と PostgreSQL pgvector を使ったセマンティック検索エンジンの構築方法を解説しました。

重要なポイントを振り返ると、`task_type` の適切な使い分け（格納時は `RETRIEVAL_DOCUMENT`、検索時は `RETRIEVAL_QUERY`）が検索精度に直結すること、HNSW インデックスがほとんどのユースケースで最適な選択であること、そしてハイブリッド検索（セマンティック + 全文検索）により更なる精度向上が可能であることがわかりました。

セマンティック検索は RAG（Retrieval-Augmented Generation）パイプラインの基盤技術でもあります。Gemini API の [エンベディング機能](/articles/gemini-api/embeddings-api-guide) や [RAG パイプライン構築](/articles/gemini-advanced/gemini-flash-rag-pipeline) と組み合わせることで、より高度な AI アプリケーションを実現できます。また、本番環境での [API コスト最適化](/articles/gemini-api/gemini-api-cost-optimization) も忘れずに検討してください。

LLMの仕組みや活用法についてより実践的な知識を身につけたい方には、「[大規模言語モデル入門](https://www.amazon.co.jp/dp/B0D1KLJPYJ/?tag=pinocchio-22)」（山田育矢 著）が参考になります。
