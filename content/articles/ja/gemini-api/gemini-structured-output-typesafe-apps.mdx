# Gemini API の Structured Output を活用したタイプセーフなアプリケーション開発

LLM をアプリケーションに組み込む際、最大の課題の一つが「出力の予測不可能性」です。自由形式のテキスト応答をパースしてアプリケーションロジックに渡す処理は、エラーの温床になります。Gemini API の Structured Output 機能を使えば、JSON Schema に準拠した構造化データを確実に取得でき、この問題を根本から解決できます。

## Structured Output の基本

Structured Output は、Gemini API のレスポンスを指定した JSON Schema に強制的に従わせる機能です。`responseMimeType` と `responseSchema` を指定するだけで利用できます。

```typescript
import { GoogleGenerativeAI, SchemaType } from "@google/generative-ai";

const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY!);

// レスポンスの型を定義
interface ProductReview {
  productName: string;
  rating: number;
  pros: string[];
  cons: string[];
  summary: string;
}

const model = genAI.getGenerativeModel({
  model: "gemini-2.5-flash",
  generationConfig: {
    responseMimeType: "application/json",
    responseSchema: {
      type: SchemaType.OBJECT,
      properties: {
        productName: { type: SchemaType.STRING, description: "製品名" },
        rating: { type: SchemaType.NUMBER, description: "1-5のスコア" },
        pros: {
          type: SchemaType.ARRAY,
          items: { type: SchemaType.STRING },
          description: "良い点のリスト"
        },
        cons: {
          type: SchemaType.ARRAY,
          items: { type: SchemaType.STRING },
          description: "改善点のリスト"
        },
        summary: { type: SchemaType.STRING, description: "総合評価の要約" }
      },
      required: ["productName", "rating", "pros", "cons", "summary"]
    }
  }
});
```

この設定により、API レスポンスは必ず指定した JSON 構造に従います。パースエラーの心配がなく、TypeScript の型システムと完全に統合できます。

## Enum を使った分類タスク

Structured Output は分類タスクと特に相性が良く、`enum` フィールドで出力を制限できます。

```typescript
const classifierModel = genAI.getGenerativeModel({
  model: "gemini-2.5-flash",
  generationConfig: {
    responseMimeType: "application/json",
    responseSchema: {
      type: SchemaType.OBJECT,
      properties: {
        category: {
          type: SchemaType.STRING,
          enum: ["bug", "feature", "question", "documentation"],
          description: "Issue のカテゴリ"
        },
        priority: {
          type: SchemaType.STRING,
          enum: ["low", "medium", "high", "critical"],
          description: "優先度"
        },
        assignTeam: {
          type: SchemaType.STRING,
          enum: ["frontend", "backend", "infra", "design"],
          description: "担当チーム"
        }
      },
      required: ["category", "priority", "assignTeam"]
    }
  }
});

// Issue テキストを自動分類
const result = await classifierModel.generateContent(
  `以下の Issue を分類してください:\n\n${issueText}`
);
const classification = JSON.parse(result.response.text());
// classification.category は必ず "bug" | "feature" | "question" | "documentation" のいずれか
```

`enum` を使うことで、API が返す値が必ず定義済みの選択肢に限定されるため、アプリケーション側のバリデーションコードが大幅に簡素化されます。

## ネストされたオブジェクトの活用

複雑なデータ構造もスキーマで定義できます。たとえば、記事コンテンツを構造化して取得する例です。

```typescript
const contentSchema = {
  type: SchemaType.OBJECT,
  properties: {
    title: { type: SchemaType.STRING },
    sections: {
      type: SchemaType.ARRAY,
      items: {
        type: SchemaType.OBJECT,
        properties: {
          heading: { type: SchemaType.STRING },
          body: { type: SchemaType.STRING },
          codeExample: { type: SchemaType.STRING, nullable: true }
        },
        required: ["heading", "body"]
      }
    },
    metadata: {
      type: SchemaType.OBJECT,
      properties: {
        estimatedReadTime: { type: SchemaType.NUMBER },
        difficulty: {
          type: SchemaType.STRING,
          enum: ["beginner", "intermediate", "advanced"]
        },
        tags: {
          type: SchemaType.ARRAY,
          items: { type: SchemaType.STRING }
        }
      },
      required: ["estimatedReadTime", "difficulty", "tags"]
    }
  },
  required: ["title", "sections", "metadata"]
};
```

`nullable: true` を指定すれば、オプショナルなフィールドも表現できます。

## バリデーションとエラーハンドリング

Structured Output は JSON 構造を保証しますが、「意味的な正しさ」はアプリケーション側で検証が必要です。Zod などのバリデーションライブラリとの組み合わせが効果的です。

```typescript
import { z } from "zod";

const ReviewSchema = z.object({
  productName: z.string().min(1),
  rating: z.number().min(1).max(5),
  pros: z.array(z.string()).min(1),
  cons: z.array(z.string()),
  summary: z.string().max(500)
});

const raw = JSON.parse(result.response.text());
const validated = ReviewSchema.parse(raw);
// validated は完全に型安全
```

## パフォーマンスのベストプラクティス

Structured Output を本番環境で使う際のポイントをまとめます。

**スキーマはシンプルに保つ**: プロパティ数が多すぎると生成速度が低下します。必要最小限のフィールドに絞りましょう。

**description を活用する**: 各プロパティに `description` を付与すると、Gemini がフィールドの意図を正確に理解し、出力品質が向上します。

**キャッシュを検討する**: 同じ入力に対する構造化出力は再利用可能です。Context Caching と組み合わせることで、コストとレイテンシを削減できます。

**バッチ処理の活用**: 大量のデータを分類する場合は、Batch API との組み合わせで効率的に処理できます。

Structured Output を活用することで、Gemini API の出力をアプリケーションの型システムにシームレスに統合し、堅牢で保守性の高いコードを実現できます。
