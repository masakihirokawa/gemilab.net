## Gemini 3.1 Pro Custom Toolsとは何か

AIエージェントを開発していると、ある問題に直面した経験はないでしょうか。モデルに複数のツール（関数）を登録しているにもかかわらず、なぜかbashコマンドで処理しようとしてしまう。本来であれば専用の関数を呼び出すべき場面で、汎用的なbashツールに頼ってしまうのです。

Googleが2026年2月にリリースした **`gemini-3.1-pro-preview-customtools`** は、まさにこの課題を解決するために設計されたモデルです。通常のGemini 3.1 Proと同じ知能・コンテキスト長（100万トークン）・マルチモーダル能力を持ちながら、**ユーザーが登録したカスタムツールを優先的に選択する**ように調整されています。

この記事では、Custom Toolsモデルの仕組みから実装方法、通常モデルとの違いまでを詳しく解説します。

## 通常のGemini 3.1 Proとの違い

通常のGemini 3.1 Proでは、bashツールとカスタム関数が両方登録されている場合、モデルが状況に応じて最も効率的と判断した方を選択します。これは一般的なタスクには便利ですが、エージェント開発では問題になります。

Custom Toolsモデルでは以下の点が改善されています。

**ツール選択の優先順位が変わる。** カスタム関数が利用可能な場面では、bashへのフォールバックを抑制し、登録された関数を積極的に使用します。

**関数呼び出しの信頼性が向上する。** 引数の形式エラーや、呼び出し漏れが大幅に減少します。

**エージェントのワークフローが予測可能になる。** どのツールが呼ばれるかをより正確に予測できるため、デバッグやテストが容易になります。

なお、価格設定は通常のGemini 3.1 Proと同一です。追加コストなしで、より信頼性の高いエージェントを構築できます。

## カスタムツールを登録してエージェントを構築する

ここからが本題です。実際にカスタム関数を登録し、Custom Toolsモデルがどのようにツールを選択するかを確認しましょう。

### ステップ1: ツール関数の定義

以下の例では、天気情報の取得と、予定の確認という2つのカスタムツールを定義します。

```python
from google import genai
from google.genai import types

# ツール関数の定義
def get_weather(city: str) -> dict:
    """指定した都市の天気情報を取得する"""
    # 実際のAPIコールの代わりにモックデータを返す
    weather_data = {
        "東京": {"temp": 18, "condition": "晴れ", "humidity": 45},
        "大阪": {"temp": 20, "condition": "曇り", "humidity": 55},
        "札幌": {"temp": 8, "condition": "雪", "humidity": 70},
    }
    return weather_data.get(city, {"error": f"{city}のデータが見つかりません"})

def get_schedule(date: str) -> dict:
    """指定した日付の予定を取得する"""
    schedules = {
        "2026-03-22": [
            {"time": "10:00", "title": "チームミーティング"},
            {"time": "14:00", "title": "クライアントレビュー"},
        ],
    }
    return schedules.get(date, {"message": "予定はありません"})
```

### ステップ2: ツール定義をモデルに渡す

```python
# ツール定義の作成
weather_tool = types.FunctionDeclaration(
    name="get_weather",
    description="指定した都市の現在の天気情報（気温、天候、湿度）を取得する",
    parameters=types.Schema(
        type="OBJECT",
        properties={
            "city": types.Schema(
                type="STRING",
                description="天気を取得したい都市名（例: 東京、大阪）"
            ),
        },
        required=["city"],
    ),
)

schedule_tool = types.FunctionDeclaration(
    name="get_schedule",
    description="指定した日付のスケジュール・予定一覧を取得する",
    parameters=types.Schema(
        type="OBJECT",
        properties={
            "date": types.Schema(
                type="STRING",
                description="予定を確認したい日付（YYYY-MM-DD形式）"
            ),
        },
        required=["date"],
    ),
)

tools = types.Tool(function_declarations=[weather_tool, schedule_tool])
```

### ステップ3: エージェントの実行

```python
client = genai.Client(api_key="YOUR_API_KEY")

response = client.models.generate_content(
    model="gemini-3.1-pro-preview-customtools",
    contents="東京の天気を教えて、あと今日の予定も確認して",
    config=types.GenerateContentConfig(
        tools=[tools],
    ),
)

# レスポンスからツール呼び出しを処理
for part in response.candidates[0].content.parts:
    if part.function_call:
        fn_name = part.function_call.name
        fn_args = dict(part.function_call.args)
        print(f"呼び出されたツール: {fn_name}")
        print(f"引数: {fn_args}")

# 出力例:
# 呼び出されたツール: get_weather
# 引数: {'city': '東京'}
# 呼び出されたツール: get_schedule
# 引数: {'date': '2026-03-22'}
```

Custom Toolsモデルでは、「東京の天気」と「今日の予定」という2つの要求を正確に認識し、それぞれ対応するカスタム関数を呼び出します。通常モデルでは、こうした複合リクエストでbashコマンドにフォールバックするケースがありましたが、Custom Toolsモデルではカスタム関数が優先されます。

## マルチターン対話でのツール結果の返送

実際のエージェントでは、ツール呼び出し → 結果の取得 → モデルへの返送 → 最終応答生成というサイクルを回します。

```python
import json

def run_agent(user_message: str):
    client = genai.Client(api_key="YOUR_API_KEY")

    # 利用可能なツール関数のマッピング
    available_functions = {
        "get_weather": get_weather,
        "get_schedule": get_schedule,
    }

    messages = [types.Content(role="user", parts=[types.Part(text=user_message)])]

    # Step 1: 初回リクエスト
    response = client.models.generate_content(
        model="gemini-3.1-pro-preview-customtools",
        contents=messages,
        config=types.GenerateContentConfig(tools=[tools]),
    )

    # Step 2: ツール呼び出しの処理
    tool_results = []
    for part in response.candidates[0].content.parts:
        if part.function_call:
            fn_name = part.function_call.name
            fn_args = dict(part.function_call.args)
            result = available_functions[fn_name](**fn_args)
            tool_results.append(
                types.Part(
                    function_response=types.FunctionResponse(
                        name=fn_name,
                        response={"result": result},
                    )
                )
            )

    if tool_results:
        # Step 3: ツール結果をモデルに返送
        messages.append(response.candidates[0].content)
        messages.append(types.Content(role="user", parts=tool_results))

        final_response = client.models.generate_content(
            model="gemini-3.1-pro-preview-customtools",
            contents=messages,
            config=types.GenerateContentConfig(tools=[tools]),
        )
        return final_response.text
    else:
        return response.text

# 実行
answer = run_agent("東京の天気と今日の予定を教えて")
print(answer)
# 出力例:
# 東京の天気は晴れで、気温は18度、湿度は45%です。
# 今日の予定は以下の通りです：
# - 10:00 チームミーティング
# - 14:00 クライアントレビュー
```

## bashツールとカスタムツールの共存パターン

Custom Toolsモデルはbashツールを完全に無効化するわけではありません。カスタム関数でカバーされていないタスクでは、bashツールが引き続き利用されます。重要なのは **優先順位** です。

以下のようなケースでは、それぞれ適切なツールが選択されます。

```python
# bashも含むツール定義
bash_tool = types.Tool(
    code_execution=types.ToolCodeExecution()
)

custom_tools = types.Tool(
    function_declarations=[weather_tool, schedule_tool]
)

# 「東京の天気は？」→ get_weather が呼ばれる（カスタムツール優先）
# 「100の階乗を計算して」→ コード実行が使われる（カスタムツールに該当なし）
# 「天気を取得してCSVに保存して」→ get_weather + コード実行の組み合わせ
```

この優先順位の仕組みにより、カスタムツールで定義した処理は確実にカスタム関数経由で実行され、それ以外のタスクはbashやコード実行で柔軟に対応できます。

## 実践的なユースケース: カスタマーサポートエージェント

より実践的な例として、カスタマーサポート向けのエージェントを構築してみましょう。

```python
# サポート用ツールの定義
support_tools = types.Tool(
    function_declarations=[
        types.FunctionDeclaration(
            name="search_faq",
            description="FAQデータベースからキーワードに一致する回答を検索する",
            parameters=types.Schema(
                type="OBJECT",
                properties={
                    "query": types.Schema(type="STRING", description="検索キーワード"),
                },
                required=["query"],
            ),
        ),
        types.FunctionDeclaration(
            name="create_ticket",
            description="サポートチケットを新規作成する",
            parameters=types.Schema(
                type="OBJECT",
                properties={
                    "title": types.Schema(type="STRING", description="チケットのタイトル"),
                    "priority": types.Schema(
                        type="STRING",
                        description="優先度（low / medium / high）",
                        enum=["low", "medium", "high"],
                    ),
                    "description": types.Schema(type="STRING", description="問題の詳細"),
                },
                required=["title", "priority", "description"],
            ),
        ),
        types.FunctionDeclaration(
            name="check_order_status",
            description="注文番号から配送状況を確認する",
            parameters=types.Schema(
                type="OBJECT",
                properties={
                    "order_id": types.Schema(type="STRING", description="注文番号"),
                },
                required=["order_id"],
            ),
        ),
    ]
)
```

Custom Toolsモデルを使うことで、ユーザーの「注文が届かない」という問い合わせに対して、まず `check_order_status` でステータスを確認し、解決しない場合は `create_ticket` でチケットを起票する——というワークフローを高い信頼性で実現できます。

## よくある質問（FAQ）

### Q1: Custom Toolsモデルは通常のGemini 3.1 Proより高いですか？

いいえ、価格は同一です。入力100万トークンあたり1.25ドル、出力100万トークンあたり10ドルで、通常のGemini 3.1 Proと同じ料金体系です。追加コストなしにエージェントの関数呼び出し精度を向上できます。

### Q2: Custom Toolsモデルでもコード実行（Code Execution）は使えますか？

はい、使えます。Custom Toolsモデルはカスタム関数を「優先」するだけであり、コード実行を無効化するわけではありません。カスタムツールでカバーされていないタスク（計算、データ処理など）では、引き続きコード実行が利用されます。

### Q3: どのような場面でCustom Toolsモデルを選ぶべきですか？

複数のカスタム関数を登録するエージェント開発が主なユースケースです。特にbashツールとカスタム関数が混在する環境で、カスタム関数の呼び出し精度を高めたい場合に最適です。単純なテキスト生成やコード実行のみのタスクでは、通常のGemini 3.1 Proで十分です。

### Q4: Provisioned Throughput（PT）は利用できますか？

現時点では、`gemini-3.1-pro-preview-customtools` ではProvisioned Throughputはサポートされていません。大量のリクエストを処理する必要がある場合は、Googleのアカウントチームに相談してください。

## まとめ

Gemini 3.1 Pro Custom Toolsモデルは、AIエージェント開発において「ツール選択の信頼性」という根本的な課題に正面から取り組んだモデルです。通常モデルと同じ価格で、カスタム関数の呼び出し精度が劇的に向上するため、エージェント開発者にとっては非常に魅力的な選択肢と言えます。

bashへの不要なフォールバックに悩んでいた方は、ぜひモデル名を `gemini-3.1-pro-preview-customtools` に変更して試してみてください。エージェントの挙動がより予測可能で安定したものになるはずです。
