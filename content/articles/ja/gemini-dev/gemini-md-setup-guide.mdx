## はじめに

Gemini CLI でプロジェクトを効率的に管理するには、**GEMINI.md** というプロジェクト設定ファイルが欠かせません。

ところが「空白から GEMINI.md を書く」のは難しいもの。開発者の頭の中にある知識を整理し、分量のあるドキュメントにまとめるのは時間がかかります。

そこで登場するのが **2ステップアプローチ**：

```
Step 1: draft.md（メモレベル）を短く書く
         ↓
Step 2: Gemini CLI に読ませて、GEMINI.md を自動生成
```

本ガイドでは、この効率的な方法を解説します。

---

## GEMINI.md とは

### 役割

GEMINI.md は **Gemini CLI（Google の開発者向けAIツール）がプロジェクトの設定を自動理解するためのメタデータファイル** です。

以下の情報を構造化して提供します：

- **使用技術スタック** （言語、フレームワーク、ライブラリ）
- **開発方針** （アーキテクチャ、デザインパターン）
- **テスト戦略** （TDD/BDD の有無、カバレッジ）
- **設計意図の記録** （なぜこの選択をしたのか）

### AGENTS.md との違い

| 項目 | AGENTS.md | GEMINI.md |
|---|---|---|
| **対応ツール** | 複数（通用フォーマット） | Gemini CLI 専用 |
| **記述方式** | YAML | Markdown + Structured sections |
| **詳細度** | 中程度 | より詳細（Gemini特化） |
| **自動生成** | 手書き推奨 | AI生成推奨 |

## draft.md の書き方

### Step 1 概要

draft.md は、あなたの頭の中にあるプロジェクト情報を **箇条書き** で記録するメモファイルです。完璧さは不要。思いついたことから書き出します。

### 基本テンプレート

```markdown
# MyProject - draft.md

## プロジェクト概要
- 目的: [何をするアプリ/ツール？]
- ターゲット: [誰が使う？]
- コア機能: [最も重要な機能3つ]

## 技術スタック
- 言語: Node.js 20
- フレームワーク: Express.js
- データベース: PostgreSQL
- フロントエンド: React 19
- テスト: Vitest

## アーキテクチャ
- [大まかな層構造]
- [通信パターン]
- [データフロー]

## 開発方針
- TDD を採用している
- マイクロサービス志向
- API first

## テスト戦略
- ユニットテスト: 80% 以上
- 統合テスト: 主要フロー
- E2E テスト: Playwright

## トラブルシューティング
- よくあるエラー1: [症状と対処法]
- よくあるエラー2: [症状と対処法]

## 禁止事項
- [しないこと1]
- [しないこと2]
```

### 実例

```markdown
# PaymentAPI - draft.md

## プロジェクト概要
- 目的: SaaS向けのStripe統合決済API
- ターゲット: 中小オンラインストア
- コア機能:
  - サブスクリプション管理
  - 複数通貨対応
  - Webhook自動処理

## 技術スタック
- Node.js 20
- Express.js + TypeScript
- PostgreSQL (Prisma ORM)
- Redis (キャッシング)
- Docker Compose (本番）

## アーキテクチャ
- API層（Express）→ Service層 → Repository層（DB）
- イベント駆動（Stripe Webhook）
- キャッシング戦略: Redis に価格表をキャッシュ

## 開発方針
- TypeScript strict mode
- TDD重視（Jest で全テスト）
- 単一責任原則（各ファイル100行以下）

## テスト
- ユニット: 90%
- 統合: Stripe testmode で実施
- 本番環境での smoke test

## トラブルシューティング
- Stripe Webhook タイムアウト → Retry ロジック + DLQ
- 複数通貨でのレート計算ズレ → キャッシュ更新頻度見直し

## 禁止事項
- 本番 API キーを env ファイルに直記述（AWS Secrets Manager へ）
- Stripe Test API と Live API の混用
```

## @draft.md で Gemini に読ませる

### Step 2 概要

Gemini CLI では、`@draft.md` という記法を使うことで、ファイルをAIに渡すことができます。

以下のコマンドで、draft.md を基に GEMINI.md を自動生成させます：

### 実行方法

```bash
gemini ask "Read @draft.md and generate a complete GEMINI.md for this project. \
Include: technology stack, architecture decisions, coding standards, \
testing strategy, troubleshooting guide, and design rationale."
```

### 生成プロンプトの詳細版

```bash
gemini ask "
You are a project documentation expert.

Read the file @draft.md, which contains brief project notes.

Generate a comprehensive GEMINI.md file with these sections:
1. Project Overview
   - Clear purpose statement
   - Target users
   - Core features (3-5 items)

2. Technology Stack
   - Languages and versions
   - Frameworks and libraries
   - Database choice and rationale

3. Architecture
   - High-level diagram (text-based)
   - Layer structure
   - Data flow patterns
   - Communication protocols

4. Development Standards
   - Coding conventions
   - File structure
   - Naming patterns
   - TypeScript/Linting rules

5. Testing Strategy
   - Test types (unit/integration/e2e)
   - Coverage targets
   - Test frameworks used
   - CI/CD integration

6. Design Decisions
   - Why we chose this architecture
   - Trade-offs made
   - Performance considerations
   - Security considerations

7. Troubleshooting Guide
   - Common errors and solutions
   - Performance bottlenecks
   - Debugging tips

8. Prohibitions
   - What developers MUST NOT do
   - Anti-patterns to avoid

Output as a markdown file suitable for Gemini CLI documentation.
"
```

### 実際の出力例

```markdown
# PaymentAPI — GEMINI.md

## Project Overview

**Purpose:** Provide a secure, multi-currency payment processing API for SaaS platforms.

**Target Users:** Small to medium-sized online retailers needing PCI-compliant payment handling.

**Core Features:**
- Subscription management with flexible billing cycles
- Real-time multi-currency conversion
- Automatic Stripe Webhook processing
- Transaction history and reconciliation

## Technology Stack

**Runtime:** Node.js 20 LTS
**Language:** TypeScript 5 (strict mode)
**Web Framework:** Express.js 4
**Database:** PostgreSQL 15 with Prisma ORM
**Caching:** Redis 7
**Deployment:** Docker Compose (development), Kubernetes (production)

**Rationale:**
- TypeScript strict mode ensures compile-time safety
- Prisma abstracts DB operations and migrations
- Redis reduces DB load for frequently accessed data (pricing tables, cache)

...

## Testing Strategy

**Unit Tests (90% coverage):**
- Framework: Jest
- Target: Pure functions, service logic
- Run: `npm test`

**Integration Tests:**
- Stripe test mode API integration
- Database transaction rollback tests
- Webhook delivery verification
- Run: `npm run test:integration`

**Deployment Verification:**
- Smoke tests post-deployment
- Health check endpoints
- Sample payment flow E2E test

...
```

## AGENTS.md との組み合わせ

Gemini CLI は **AGENTS.md もサポート** しています。以下のように組み合わせるのが推奨です：

### 推奨構成

```
project/
├── draft.md          ← メモ（ステップ1で作成）
├── GEMINI.md         ← 詳細設定（ステップ2で生成）
├── AGENTS.md         ← 通用フォーマット（手書き）
├── README.md         ← 人間向けドキュメント
├── docs/
│   ├── architecture.md    ← アーキテクチャ詳細
│   ├── api-reference.md   ← API 仕様
│   └── design.md          ← 設計意図記録
└── src/
```

### 役割分担

| ファイル | 役割 | 作成方法 |
|---|---|---|
| **draft.md** | 最初のメモ | 手書き（箇条書き） |
| **GEMINI.md** | Gemini CLI の詳細設定 | AI生成（@draft.md から） |
| **AGENTS.md** | マルチAIツール対応 | 手書きまたはAI生成 |
| **README.md** | 人間向け公式ドキュメント | 手書き |
| **docs/design.md** | 設計意図（なぜ？） | 手書き |

## 運用のコツ

### draft.md の更新タイミング

draft.md は **生きているメモ** です。以下のタイミングで更新します：

```
新しいライブラリを追加した → draft.md を更新
新しいトラブルを発見した → draft.md に追加
設計方針が変わった → draft.md に記録
```

### 定期的な GEMINI.md 再生成

3ヶ月ごと、または重大な変更後に再生成することで、ドキュメントを最新に保ちます：

```bash
# 3ヶ月ごと
gemini ask "Read @draft.md and regenerate GEMINI.md..."
```

### draft.md と GEMINI.md の分離メリット

| メリット | 説明 |
|---|---|
| **保守性** | draft.md は短いので編集が楽 |
| **レビュー可能** | GEMINI.md を team で review して精度を上げる |
| **バージョン管理** | 変更前後の差分が明確 |
| **自動化** | 新人 onboarding 時に自動再生成できる |

## FAQ

### Q1: draft.md はどの程度の詳細さで書くべき？

**A:** **箇条書きで十分**。完全な文章は不要です。

```
✓ 良い draft.md
- TypeScript 使用
- Express.js + Prisma
- PostgreSQL
- Jest でテスト 80%
- イベント駆動アーキテクチャ

✗ 悪い draft.md（長すぎる）
当プロジェクトは、モダンな web アプリケーション開発のベストプラクティスを...
[長文]
```

### Q2: 自動生成された GEMINI.md に間違いがあった場合は？

**A:** 以下の2つの方法があります：

**方法1: 修正してコミット**
```bash
# 手動で間違いを修正
code GEMINI.md
git add GEMINI.md && git commit -m "Fix: GEMINI.md corrections"
```

**方法2: draft.md を修正して再生成**
```bash
# draft.md の該当箇所を修正してから
gemini ask "Regenerate GEMINI.md from @draft.md..."
```

方法2 の方が、将来の team members にも参考になります。

### Q3: draft.md と GEMINI.md、どちらを Git に commit すべき？

**A:** **両方** commit をお勧めします。

```bash
git add draft.md GEMINI.md
git commit -m "Update: project documentation from draft to GEMINI.md"
```

**理由：**
- draft.md は team の共有メモ（short, editable）
- GEMINI.md は公式仕様（long, reviewed）
- 両方があると、差分履歴が明確

## まとめ

**GEMINI.md を効率的に作るには：**

```
Step 1: draft.md（メモ）を 5〜10分で書く
        箇条書き、短い文で OK

Step 2: Gemini CLI で自動生成
        "Read @draft.md and generate GEMINI.md..."

Step 3: 確認・修正・commit
        Team で review してから Git に
```

このアプローチにより、**30分 → 5分** に短縮できます。
