## Plan Modeとは何か

Plan Modeは、AIが実際に作業を開始する前に**構造化された実行計画**を提示し、ユーザーが承認してから処理を進める仕組みです。2026年3月のアップデートでデフォルト有効（`--plan-mode on`）となり、すべてのGemini CLIユーザーが自動的に恩恵を受けられるようになりました。

### Plan Modeが重要な理由

従来のAIエージェントは「考えながら動く」スタイルで、途中経過が見えにくく、予期しない操作が発生することがありました。Plan Modeではこの問題を解決します。

```
ユーザー: "src/以下のすべてのTypeScriptファイルに型注釈を追加して"

[Gemini CLI – Plan Mode]
📋 実行計画を提示します:
  1. src/ ディレクトリを再帰的にスキャンして .ts/.tsx ファイルを列挙
  2. 各ファイルのAST解析で型注釈が欠けている箇所を特定
  3. 変更前にバックアップを作成（.backup/ に保存）
  4. 型注釈を追加（any は使用しない方針）
  5. 変更差分を表示してユーザーに確認を求める
  6. 承認後にファイルを更新

この計画を承認しますか？ [y/N]
```

この一時停止があることで、開発者はAIが何をしようとしているかを事前に把握し、意図しない変更を防ぐことができます。

## Plan Modeのセットアップ

### インストールとバージョン確認

まず、Gemini CLIが最新バージョンであることを確認します。

```bash
# インストール（未インストールの場合）
npm install -g @google/gemini-cli

# バージョン確認（v0.34.0以上が必要）
gemini --version
# 出力例: 0.34.0-preview.1

# Plan Modeが有効か確認
gemini config show | grep plan_mode
# 出力例: plan_mode: true
```

### 設定ファイルでのカスタマイズ

Plan Modeの動作は `~/.gemini/config.yaml` で詳細に設定できます。

```yaml
# ~/.gemini/config.yaml

plan_mode:
  enabled: true                    # デフォルト: true
  auto_approve_low_risk: false     # リスクの低い操作を自動承認するか
  preserve_on_compression: true   # チャット圧縮時にプランを保持するか（重要）
  max_plan_steps: 20               # 計画の最大ステップ数

sandbox:
  type: "gvisor"                   # "none" | "lxc" | "gvisor"
  allow_network: false             # サンドボックス内でのネット接続を許可するか
  workspace_mount: "/workspace"    # ワークスペースのマウントパス
```

### チャット圧縮時のプラン保持

長い会話では、コンテキスト節約のためにチャット履歴が圧縮されることがあります。2026年3月のアップデートでは、**承認済みプランがこの圧縮に耐えて保持される**ようになりました。

```bash
# 圧縮前後でプランが維持されることを確認
gemini chat --verbose

# コンソール出力例（圧縮発生時）
# [INFO] Context compression triggered (48k tokens → 12k tokens)
# [INFO] Approved plan preserved: "TypeScript annotation task (Step 3/6)"
# [PLAN] Resuming from step 3: Running AST analysis on auth/middleware.ts
```

## サンドボックスの仕組みと選択

Gemini CLIのサンドボックスは、AIエージェントが実行するコードやシェルコマンドをホストシステムから隔離します。2026年3月のアップデートで、**LXCコンテナ**と**gVisor（runsc）**の2つの新しいサンドボックスモードが追加されました。

### サンドボックスの比較

| 機能 | none | LXC | gVisor |
|------|------|-----|--------|
| ファイルシステム隔離 | ❌ | ✅ | ✅ |
| ネットワーク隔離 | ❌ | ✅ | ✅ |
| カーネル保護 | ❌ | △ | ✅ |
| 起動速度 | 即時 | 中 | 速い |
| オーバーヘッド | なし | 低 | 非常に低い |
| 推奨用途 | 開発・テスト | CI/CD | 本番・セキュリティ重視 |

### LXCサンドボックスの設定

LXC（Linux Containers）は、軽量なOSレベルの仮想化を提供します。

```bash
# LXCサンドボックスを有効化
gemini config set sandbox.type lxc

# LXCが正しく動作しているか確認
gemini sandbox status
# 出力例:
# Sandbox type: LXC (experimental)
# Container status: ready
# Network isolation: enabled
# Workspace: /workspace (read-write)

# LXCサンドボックス内でタスク実行
gemini run --sandbox lxc "package.jsonのdependenciesを最新バージョンに更新して"
```

### gVisorサンドボックスの設定

gVisorはGoogleが開発したユーザー空間カーネルで、コンテナのシステムコールをインターセプトしてホストカーネルを保護します。

```bash
# gVisorが利用可能か確認
which runsc || echo "gVisor (runsc) が見つかりません"

# gVisorのインストール（Debian/Ubuntu）
sudo apt-get install -y runsc

# Gemini CLIでgVisorを有効化
gemini config set sandbox.type gvisor

# 動作確認
gemini sandbox test
# 出力例:
# ✅ gVisor (runsc) sandbox: OK
# Security level: kernel-protected
# Syscall interception: active
```

### Pythonスクリプトでの自動化

Plan ModeはAPIから制御することも可能です。

```python
import subprocess
import json

def run_gemini_plan(task: str, sandbox: str = "gvisor") -> dict:
    """
    Gemini CLI Plan Modeを自動承認モードで実行する関数

    Args:
        task: AIに実行させるタスクの説明
        sandbox: サンドボックスタイプ ("none", "lxc", "gvisor")

    Returns:
        実行結果を含む辞書
    """
    cmd = [
        "gemini", "run",
        "--sandbox", sandbox,
        "--auto-approve",          # CI/CDでの自動承認
        "--output-format", "json", # 結果をJSON形式で出力
        task
    ]

    result = subprocess.run(
        cmd,
        capture_output=True,
        text=True,
        timeout=300  # 5分タイムアウト
    )

    if result.returncode == 0:
        return {
            "status": "success",
            "output": json.loads(result.stdout),
            "steps_completed": result.stdout.count("✅")
        }
    else:
        return {
            "status": "error",
            "error": result.stderr,
            "returncode": result.returncode
        }

# 使用例
result = run_gemini_plan(
    "src/utils/ 以下のすべての関数にJSDocコメントを追加して",
    sandbox="gvisor"
)

print(f"ステータス: {result['status']}")
# 出力例: ステータス: success
# 完了ステップ数: 5
```

## よくあるエラーと対処法

### エラー1: gVisor (runsc) が見つからない

```
Error: gVisor sandbox requested but 'runsc' not found in PATH
```

**対処法:**

```bash
# Ubuntuの場合
sudo apt-get install -y runsc

# macOSの場合（Homebrewは未サポート）
# Docker Desktop with gVisor を使用するか、LXCサンドボックスに切り替え
gemini config set sandbox.type lxc
```

### エラー2: チャット圧縮後にプランがリセットされる

古いバージョン（v0.33以前）では、チャット圧縮後にプランがリセットされることがありました。

```bash
# バージョンを確認・更新
gemini --version
npm install -g @google/gemini-cli@latest

# 設定でプラン保持を明示的に有効化
gemini config set plan_mode.preserve_on_compression true
```

### エラー3: LXCコンテナの起動失敗

```
Error: LXC container failed to start (permission denied)
```

**対処法:**

```bash
# ユーザーをlxdグループに追加
sudo usermod -aG lxd $USER
newgrp lxd

# LXDを初期化
sudo lxd init --auto

# 再試行
gemini sandbox test --type lxc
```

## セキュリティのベストプラクティス

AIエージェントを安全に運用するためのポイントをまとめます。

### 1. 最小権限の原則

サンドボックス内でのネットワークアクセスは、必要な場合のみ許可します。

```yaml
# ~/.gemini/config.yaml
sandbox:
  type: "gvisor"
  allow_network: false          # デフォルトはオフ
  allowed_hosts:                # 必要なホストのみ許可
    - "api.github.com"
    - "registry.npmjs.org"
  read_only_paths:              # 読み取り専用マウント
    - "/etc/ssl/certs"
```

### 2. CI/CDでの使用

```yaml
# .github/workflows/ai-refactor.yml
name: AI-assisted Refactoring

on:
  workflow_dispatch:
    inputs:
      task:
        description: 'Gemini CLIに実行させるタスク'
        required: true

jobs:
  ai-refactor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Gemini CLI
        run: npm install -g @google/gemini-cli

      - name: Install gVisor
        run: sudo apt-get install -y runsc

      - name: Run AI task in sandbox
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
        run: |
          gemini run \
            --sandbox gvisor \
            --auto-approve \
            "${{ github.event.inputs.task }}"
```

## FAQ

### Q1. Plan Modeを無効にすることはできますか？

はい、可能です。ただし本番環境での使用は推奨しません。

```bash
# Plan Modeを無効化
gemini config set plan_mode.enabled false

# または実行時に一時的に無効化
gemini run --no-plan-mode "簡単なタスク"
```

### Q2. gVisorはmacOSでも使えますか？

現時点（2026年3月）では、gVisorはLinux環境のみサポートされています。macOSでは、LXCサンドボックスまたはDocker Desktop経由でgVisorを利用するワークアラウンドがあります。最新のサポート状況は[Gemini CLI公式ドキュメント](https://geminicli.com/docs)を確認してください。

### Q3. Plan Modeで自動承認できますか？

CI/CD環境では `--auto-approve` フラグを使用できます。ただし、`auto_approve_low_risk: true` の設定も活用することで、リスクの低い操作（ファイル読み取り、コメント追加など）は自動承認し、ファイル削除・外部API呼び出しなどは手動確認するハイブリッド運用が可能です。

### Q4. サンドボックス内でのGPU使用は可能ですか？

gVisorはCUDAのパススルーをサポートしていますが、設定が必要です。LXCはネイティブにGPUパススルーをサポートしています。AIモデルのローカル推論を伴うタスクではLXCが有利です。

### Q5. Plan Modeはどの言語モデルで動きますか？

Gemini CLI v0.34以降では、Gemini 3.1 ProとGemini 3.1 Flash-Liteの両方でPlan Modeが動作します。Gemini 3.1 Proは複雑なマルチステッププランの生成に優れており、シンプルなタスクではGemini 3.1 Flash-Liteがコスト効率に優れています。
