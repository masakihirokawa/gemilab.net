## Gemini CLI 2026年3月アップデート — エンタープライズ開発の新標準

Gemini CLI は2026年3月、v0.33.0 から v0.34.0 へのメジャーアップデート、さらに v0.35.0-preview への先行プレビューを通じて、エンタープライズレベルの開発ワークフローを実現する3つの革新的な機能をもたらしました。Plan Mode のデフォルト有効化、gVisor/LXC ベースのセキュアサンドボックス、HTTP 認証による A2A リモートエージェント連携です。本記事では、各バージョンの進化と実装方法を徹底解説します。

## Plan Mode — 実行前の詳細計画生成

### Plan Mode とは

Plan Mode は、コード実行前に詳細な実行計画を生成し、ユーザーの確認を経た上で実行する安全性と透明性を重視したモードです。v0.34.0 からデフォルトで有効化されています。

### Plan Mode の基本動作

```bash
# インストール（v0.34.0以降）
brew install gemini-cli@latest
# または
gem install gemini-cli --version '~> 0.34.0'

# バージョン確認
gemini --version
# gemini-cli 0.34.0 (Plan Mode enabled by default)

# Plan Mode でコマンド実行
gemini run --script=deploy.rb --target=production

# 実行フロー:
# 1. スクリプト解析
# 2. 詳細な実行計画を生成（Plan Mode）
# 3. ユーザーに計画を提示
# 4. 確認後に実行
```

### Plan Mode での実行計画の例

```bash
$ gemini run --script=migrate-db.rb --target=production
```

## gVisor/LXC サンドボックス — セキュアな実行環境

### v0.34.0 でのサンドボックス導入

v0.34.0 は、gVisor（Google が開発したセキュアコンテナランタイム）と LXC（Linux Container）を統合し、Gemini CLI で実行されるすべてのスクリプトを隔離された環境で安全に実行します。

### サンドボックスの仕組み

```
ホストマシン
  │
  ├─ gVisor ユーザースペースカーネル
  │  └─ 各スクリプト実行用 LXC コンテナ
  │     ├─ コンテナ A（ファイルシステム隔離）
  │     ├─ コンテナ B（ネットワーク隔離）
  │     └─ コンテナ C（プロセス隔離）
  │
  └─ ホストシステムリソース（保護）
```

### サンドボックスの有効化と設定

```bash
# v0.34.0 ではデフォルトで有効
gemini run --script=untrusted-code.rb

# サンドボックスのステータス確認
gemini sandbox --status
# Output: gVisor runtime (LXC backend) - ACTIVE

# サンドボックスの詳細設定
gemini config sandbox.yml

# sandbox.yml の例:
---
runtime: gvisor
backend: lxc
security_level: strict
resource_limits:
  memory: 2G
  cpu: 2
  disk: 10G
  network_bandwidth: 100Mbps
isolation:
  filesystem: enabled
  network: enabled
  process: enabled
  ipc: enabled
```

### 外部リソースへのアクセス制御

```bash
# サンドボックス内から外部ネットワークにアクセス
gemini run --script=api-call.rb \
  --allow-network=https://api.example.com \
  --allow-network=https://api.third-party.com

# ファイルシステムのアクセス許可
gemini run --script=file-processor.rb \
  --allow-read=/data/input \
  --allow-write=/data/output \
  --deny-all-others
```

### セキュリティポリシーの定義

```ruby
# policy.rb: セキュリティポリシーの例

Gemini.define_sandbox_policy :api_integration do
  # ネットワークアクセス
  allow_network :https, hosts: [
    'api.example.com',
    'auth.service.com'
  ]

  # ファイルアクセス
  allow_read '/etc/config/api_keys.enc'
  allow_write '/var/log/api_calls.log'

  # 環境変数
  allow_env 'DATABASE_URL'
  allow_env 'API_KEY'

  # 禁止操作
  deny :system_call, 'rm -rf'
  deny :file_delete, '/etc'
  deny :network, 'http://localhost:6379'  # Redis への直接接続禁止
end

# 実行時にポリシー適用
gemini run --script=api.rb --policy=api_integration
```

## A2A リモートエージェント — HTTP認証による分散実行

### A2A (Agent-to-Agent) 認証の導入

v0.33.0 で導入された A2A HTTP 認証により、複数の Gemini エージェントが安全に相互通信できるようになりました。v0.34.0 でこの機能が安定化し、本番環境での利用が推奨されています。

### A2A の基本設定

```bash
# エージェント1（プライマリ）
gemini agent create \
  --name="primary-agent" \
  --port=9000 \
  --auth=http_bearer

# 出力:
# Agent Token: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
# Endpoint: https://primary-agent.local:9000

# エージェント2（サテライト）
gemini agent create \
  --name="satellite-agent" \
  --port=9001 \
  --upstream="https://primary-agent.local:9000" \
  --upstream-token="eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

### マルチエージェント分散タスク

```bash
# プライマリエージェントで複数のリモートタスクを調整
gemini run --script=distributed-build.rb \
  --agents=[
    "primary-agent@localhost:9000",
    "build-agent@ci.example.com:9001",
    "test-agent@ci.example.com:9002",
    "deploy-agent@prod.example.com:9003"
  ]

# distributed-build.rb の例:
Gemini.distributed_task do |task|
  task.coordinate_with :primary_agent do
    # タスク分配の調整
  end

  task.send_to :build_agent do
    # コンパイルタスク
    compile_source
  end

  task.send_to :test_agent do
    # テスト実行
    run_tests
  end

  task.send_to :deploy_agent do
    # デプロイ実行
    deploy_to_production
  end
end
```

### HTTP Bearer Token による認証

```bash
# トークンの生成
gemini auth generate-token \
  --agent="primary-agent" \
  --ttl="30d" \
  --scopes="read,write,execute"

# HTTP ヘッダーで認証
curl -H "Authorization: Bearer eyJhbGc..." \
  https://primary-agent.local:9000/execute \
  -X POST \
  -d @task.json

# Gemini CLI での認証設定
gemini config set auth.primary_token "eyJhbGc..."
gemini config set auth.method "http_bearer"
```

## ループ検知メカニズム — 無限ループの防止

### ループ検知とは

v0.34.0 では、スクリプト実行中の無限ループを自動検知し、実行を安全に停止するメカニズムが実装されました。これにより、予期しないハング状態からの復旧が自動化されます。

### ループ検知の設定

```bash
# デフォルトのループ検知設定
gemini config loop-detection.yml

# loop-detection.yml
---
enabled: true
detection_methods:
  - stack_trace_pattern_matching
  - execution_time_threshold
  - cpu_usage_anomaly

# スタック追跡パターンの検知
stack_trace:
  max_same_function_depth: 50
  timeout: 300s

# CPU 使用率の異常検知
cpu_anomaly:
  threshold: 95%  # 95% 以上が継続
  duration: 30s

# 自動停止時の動作
auto_stop:
  action: graceful_shutdown
  notify: true
  save_state: true
```

### 実行時のループ検知例

```bash
$ gemini run --script=broken-loop.rb

[00:00] スクリプト開始...
[00:05] 無限ループ検知...
[00:10] ⚠ ループ検知: function_A → function_B → function_A (深さ: 45)
[00:15] CPU使用率: 98%
[00:20] ✓ 自動停止実行中...

=== LOOP DETECTION REPORT ===
検知時刻: 2026-03-20 14:32:10
検知タイプ: Recursive Loop
関数: function_A, function_B
深さ: 45回
実行時間: 20秒
CPU使用率ピーク: 98%

=== スタックトレース ===
function_A (line 42)
  └─ function_B (line 58)
     └─ function_A (line 42)
        └─ [ループ繰り返し x 43]

=== 推奨アクション ===
1. line 42 のロジックを確認（終了条件が不足）
2. テスト環境でデバッグ実行
3. 最大反復回数を明示的に設定
```

### ループ検知からのリカバリ

```ruby
# loop-safe.rb: ループ検知対応スクリプト

Gemini.run do
  max_iterations = 1000
  iteration = 0

  while true
    iteration += 1

    # ループ検知レジストリに報告
    Gemini.loop_registry.checkpoint(
      function: __method__,
      iteration: iteration
    )

    # 終了条件のチェック
    break if condition_met?
    break if iteration > max_iterations

    process_item
  end
end
```

## v0.35.0-preview: 次世代機能プレビュー

### Subagent Tool Isolation

```bash
# v0.35.0-preview のインストール
gem install gemini-cli --pre --version '0.35.0-preview'

# Subagent ツール隔離の有効化
gemini config subagent-isolation.yml

# subagent-isolation.yml
---
isolation_level: strict
tool_groups:
  database:
    tools:
      - query
      - insert
      - update
    denied_access:
      - delete  # delete は明示的に禁止
      - truncate

  filesystem:
    tools:
      - read
      - append
    denied_access:
      - chmod
      - chown
```

### Proxy Routing — マルチエージェント最適化

```bash
# Proxy routing で複数エージェント間の最適ルーティング
gemini proxy-routing configure \
  --primary-agent="agent1@local:9000" \
  --fallback-agents=["agent2@backup1:9001", "agent3@backup2:9002"] \
  --load-balancing="round-robin"

# 実行時の自動ルーティング
gemini run --script=critical-task.rb \
  --routing-strategy=failover \
  --timeout-per-agent=30s
```

### Vim Mode — CLI の改善

```bash
# Vim mode を有効化
gemini config set ui.editor_mode "vim"

# または直接起動
gemini run --script=script.rb --editor=vim

# Vim mode での操作:
# - ESC で normal mode に切り替え
# - :w で保存
# - :e で別のファイル編集
# - :q で終了
```
