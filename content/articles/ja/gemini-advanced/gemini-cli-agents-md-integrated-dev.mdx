## はじめに

モダン開発では、複数の AI ツールが同時に動作するようになりました：

- **Gemini CLI** → コード生成・アーキテクチャ相談
- **Claude Code** → 深い思考・コードレビュー
- **Codex CLI** → ディレクトリ走査・一括生成

これら全ツールが同じプロジェクトで効果的に動作するには、共通の「約束ごと」が必要です。その約束ごとが **AGENTS.md** です。

本ガイドでは、Gemini CLI × AGENTS.md を中心とした **統合開発環境** の構築方法を解説します。

---

## AGENTS.md の設計思想

### 60,000リポジトリの採用実績

AGENTS.md は 2025年、Linux Foundation 傘下の Agentic AI Foundation（AAIF）により正式採択されました。現在、GitHub全体で **60,000以上のリポジトリ** がこのフォーマットを採用しています。

単なる「提案」ではなく、**事実上の業界標準** です。

### AIエージェントの4つの役割

AGENTS.md が定義する AI エージェント の役割：

| 役割 | 説明 | 実装 |
|---|---|---|
| **Constrain** | 「してはいけないこと」を定義 | constraints セクション |
| **Inform** | 設計意図と背景を提供 | technology_stack, design_philosophy |
| **Verify** | 制約違反をチェック | verification セクション |
| **Correct** | 自動的に問題を修正 | AI による auto-fix |

### Gemini CLI での活用

Gemini CLI は AGENTS.md を読み込み、以下の処理を自動化します：

```
AGENTS.md を読み込み
    ↓
constraints をチェック
    ↓
制約違反を検出
    ↓
修正案を提案 または自動修正
```

## AGENTS.md の実装パターン

### 基本構造

```yaml
---
# agents.md
# AI エージェント向けプロジェクト設定

technology_stack:
  runtime: "Node.js 20 LTS"
  language: "TypeScript 5 (strict)"
  frameworks:
    - "Express.js 4"
    - "React 19"
  database: "PostgreSQL 15"
  orm: "Prisma"
  testing: "Jest"

design_philosophy: |
  事業ロジックはサービス層に集約。
  UI層は表示に専念。
  データアクセスはリポジトリパターンで隔離。
  テスト駆動開発（TDD）を採用。

agents:
  - name: "Backend Developer"
    role: "API開発"
    responsibilities:
      - "src/services/ 配下のビジネスロジック実装"
      - "API エンドポイント設計"
      - "データベーススキーマ管理"
    constraints:
      - "TypeScript strict mode 必須"
      - "外部ライブラリ追加は事前承認必須"
      - "データベーススキーマ変更は Prisma migration で"
    verification:
      - "npm test が全て pass"
      - "npm run lint でエラーゼロ"
      - "TypeScript compile エラーなし（tsc --noEmit）"

  - name: "Frontend Developer"
    role: "UI/UX開発"
    responsibilities:
      - "React コンポーネント実装"
      - "状態管理"
      - "スタイリング"
    constraints:
      - "CSS は Tailwind CSS のみ"
      - "styled-components は禁止"
      - "API 呼び出しは src/services/ の Hook を経由"
    verification:
      - "npm test (React components)"
      - "Accessibility テスト（aXe）pass"

build:
  development: "npm run dev"
  test: "npm test"
  lint: "npm run lint"
  type_check: "tsc --noEmit"
  production: "npm run build && npm start"

prohibitions:
  - "本番 API キーを .env に直記述（AWS Secrets Manager へ）"
  - "node_modules の直接編集"
  - "データベーススキーマの自動マイグレーション"
  - "サードパーティ API キー の Git コミット"

testing_requirements:
  unit_coverage: "80% minimum"
  integration_coverage: "主要フロー"
  e2e_tests: "Playwright で重要ユーザーフロー"

design_decisions: |
  **なぜ Prisma？**
  - TypeScript型安全性
  - 自動マイグレーション
  - 多言語対応（SQL生成）

  **なぜ Express？**
  - 軽量で学習曲線が低い
  - ミドルウェアエコシステムが充実
  - REST API 構築に適している

  **なぜ React？**
  - コンポーネント指向で再利用性が高い
  - エコシステムが充実
  - チーム学習リソースが豊富
---
```

## 複数AIツール共存の設定

### ファイル構成

```
project/
├── AGENTS.md              ← 汎用フォーマット（Gemini CLI, Codex, Cursor）
├── GEMINI.md              ← Gemini CLI特化
├── CLAUDE.md              ← Claude Code特化
├── .cursorrules           ← Cursor特化
├── .copilot-instructions  ← GitHub Copilot特化
├── README.md              ← 人間向けドキュメント
└── docs/
    ├── architecture.md
    ├── api-reference.md
    └── design.md
```

### ツール別の読み込み順序

#### Gemini CLI
```
1. GEMINI.md → AGENTS.md の順で探索
2. ディレクトリ構造を走査して推測
```

#### Claude Code
```
1. CLAUDE.md （優先）
2. AGENTS.md （フォールバック）
```

#### Codex CLI
```
1. ディレクトリ全体を走査
2. AGENTS.md と README.md を参照
```

#### Cursor
```
1. .cursorrules
2. README.md
```

### 各ファイルの役割分担

| ファイル | 対象ツール | 詳細度 | 更新頻度 |
|---|---|---|---|
| AGENTS.md | 全ツール | 中程度 | 月1回 |
| GEMINI.md | Gemini CLI | 高い | 月1回 |
| CLAUDE.md | Claude Code | 高い | 月1回 |
| .cursorrules | Cursor | 中程度 | 三月1回 |
| .copilot-instructions | GitHub Copilot | 低い | 半年1回 |

## 実装例：エンタープライズプロジェクト

### プロジェクト要件

```
- 開発チーム: 10人（Backend 5, Frontend 3, DevOps 2）
- 製品: SaaS プラットフォーム
- 要件: 高可用性、セキュリティ重視、複数AI ツール対応
```

### AGENTS.md 実装（エンタープライズ版）

```yaml
---
# agents.md for SaaS Platform

project:
  name: "PaymentGateway SaaS"
  description: "Multi-tenant payment processing platform"
  version: "3.1.0"
  repository: "github.com/company/payment-gateway"

technology_stack:
  runtime: "Node.js 20 LTS"
  language: "TypeScript 5 (strict mode required)"
  backend:
    - "Express.js 4"
    - "Bull Queue (Job processing)"
    - "Winston (Logging)"
  frontend:
    - "React 19"
    - "Zustand (State management)"
    - "TanStack Query (Data fetching)"
  database:
    - "PostgreSQL 15 (primary)"
    - "Redis 7 (cache & queue)"
  infrastructure:
    - "Docker Compose (development)"
    - "Kubernetes (production)"
    - "GitHub Actions (CI/CD)"
  testing:
    - "Jest (unit)"
    - "Playwright (e2e)"
    - "k6 (load testing)"

design_philosophy: |
  # Core Principles

  1. **Domain-Driven Design**: Business logic in domain layer
  2. **Separation of Concerns**: Clear layer boundaries
  3. **Security First**: Defense in depth approach
  4. **Test-Driven Development**: Tests before code
  5. **Observability**: Logging, metrics, tracing enabled

  # Architectural Layers

  API → Controllers → Services → Repositories → Database

  Each layer has strict boundaries and responsibilities.

agents:
  # Backend Team
  - name: "Backend Team"
    role: "Core API and business logic"
    team_size: 5
    responsibilities:
      - "REST API design and implementation"
      - "Business logic in service layer"
      - "Database schema and migrations"
      - "Job queue management"
    constraints:
      - "TypeScript strict: true"
      - "API routes only in src/routes/"
      - "Business logic only in src/services/"
      - "Database access only via src/repositories/"
      - "No hardcoded secrets (AWS Secrets Manager)"
      - "All external API calls must have retry logic"
      - "Database migrations require review before production"
    verification:
      - "npm test -- --coverage (90%+ required)"
      - "npm run lint (0 errors)"
      - "tsc --noEmit (type check pass)"
      - "npm run type-check (full repo type check)"
      - "Security scan: npm audit (0 critical vulnerabilities)"
      - "Database migration dry-run: npm run migrate:dry-run"

  # Frontend Team
  - name: "Frontend Team"
    role: "UI/UX implementation"
    team_size: 3
    responsibilities:
      - "React component development"
      - "State management with Zustand"
      - "Data fetching with TanStack Query"
      - "Styling with Tailwind CSS"
    constraints:
      - "React functional components only"
      - "Tailwind CSS for styling (no CSS-in-JS)"
      - "All API calls through src/hooks/useApi*"
      - "Component files max 300 lines"
      - "No direct component coupling"
    verification:
      - "npm test -- --coverage (80%+ required)"
      - "Accessibility: axe-core tests pass"
      - "Visual regression: Percy tests pass"
      - "Bundle size: < 500KB (gzipped)"

  # DevOps Team
  - name: "DevOps Team"
    role: "Infrastructure and deployment"
    team_size: 2
    responsibilities:
      - "Docker/Kubernetes management"
      - "CI/CD pipeline maintenance"
      - "Monitoring and alerting"
      - "Security hardening"
    constraints:
      - "Infrastructure as Code (Terraform)"
      - "No manual deployments"
      - "All configs versioned in Git"
      - "Secrets in AWS Secrets Manager"
    verification:
      - "Terraform plan successful"
      - "All deployment checks green"
      - "Security scan: Snyk (0 critical)"

build:
  development: "npm run dev"
  testing: "npm test"
  lint: "npm run lint:fix"
  type_check: "tsc --noEmit"
  security_scan: "npm audit"
  production_build: "npm run build && docker build"
  deploy_staging: "npm run deploy:staging"
  deploy_production: "npm run deploy:prod"

prohibitions:
  - "Never commit .env files (use .env.example)"
  - "Never hardcode API keys or secrets"
  - "Never modify node_modules directly"
  - "Never run npm install in Docker (use ci)"
  - "Never push without passing all tests"
  - "Never deploy without security scan"
  - "Never use console.log in production (use logger)"
  - "Never mix TypeScript strict and non-strict"
  - "Never add dependencies without team review"
  - "Never bypass automated quality checks"

testing_requirements:
  unit_tests:
    coverage: "90%"
    framework: "Jest"
    location: "src/__tests__/"
  integration_tests:
    framework: "Jest (with test database)"
    coverage: "Main workflows"
  e2e_tests:
    framework: "Playwright"
    coverage: "Critical user paths"
  load_testing:
    framework: "k6"
    threshold: "1000 concurrent users"
  security_testing:
    - "OWASP Top 10 scan"
    - "Dependency vulnerability check"
    - "Secret scanning"

monitoring_and_observability:
  logs:
    tool: "Winston"
    destination: "CloudWatch"
  metrics:
    tool: "Prometheus"
    destination: "Grafana"
  tracing:
    tool: "Jaeger"
    destination: "Jaeger UI"
  alerts:
    tool: "PagerDuty"
    sla: "P1: 15min response"

design_decisions: |
  # Architectural Choices and Rationale

  ## Why Bull Queue?
  - Decouples request-response cycle
  - Handles background jobs reliably
  - Allows retry logic for failed tasks

  ## Why Zustand over Redux?
  - Minimal boilerplate
  - TypeScript-first design
  - Better performance for most use cases

  ## Why Kubernetes?
  - Auto-scaling capabilities
  - Self-healing pods
  - Zero-downtime deployments
  - Industry standard for enterprise

  ## Why PostgreSQL + Redis?
  - Separation of concerns (data vs. cache)
  - Redis handles high-frequency reads
  - PostgreSQL for transactional integrity

  ## Why GitHub Actions?
  - Native GitHub integration
  - Cost-effective for startups
  - Sufficient for current scale (upgradeable)

performance_targets:
  api_response_time: "< 200ms (p95)"
  page_load_time: "< 2s (LCP)"
  lighthouse_score: "> 90"
  database_query_time: "< 100ms (p95)"

security_requirements:
  authentication: "OAuth 2.0 + SAML"
  authorization: "RBAC with Casbin"
  encryption: "TLS 1.3 in transit, AES-256 at rest"
  compliance: "SOC 2 Type II, GDPR ready"
  audit_logging: "All data access logged"
  penetration_testing: "Quarterly"
  dependency_audit: "Weekly (Snyk)"
---
```

## ハーネスエンジニアリングの統合

### AI エージェント の品質担保フロー

```
┌───────────────────────────────────────┐
│ Step 1: Constrain（制約定義）         │
│ AGENTS.md で rules を定義            │
└───────────────────────────────────────┘
           ↓
┌───────────────────────────────────────┐
│ Step 2: Inform（情報提供）            │
│ コミットメッセージ・ドキュメント      │
└───────────────────────────────────────┘
           ↓
┌───────────────────────────────────────┐
│ Step 3: Verify（検証）                │
│ Lint, Type Check, Test               │
└───────────────────────────────────────┘
           ↓
┌───────────────────────────────────────┐
│ Step 4: Correct（修正）               │
│ AI による自動修正                    │
└───────────────────────────────────────┘
```

### Gemini CLI での検証コマンド

```bash
# AGENTS.md を基に自動レビュー
gemini ask "Review my code against @agents.md constraints"

# 自動修正
gemini ask "Fix violations of @agents.md according to the constraints"

# 検証
npm test && npm run lint && tsc --noEmit
```

## FAQ

### Q1: 複数AI ツールを共存させると、競合しないか？

**A:** AGENTS.md が「共通言語」として機能し、競合を防ぎます。

```
Gemini CLI が AGENTS.md を読む
    ↓ [ルール理解]
Claude Code が AGENTS.md を読む
    ↓ [同じルール]
Codex CLI が AGENTS.md を読む
    ↓ [同じルール]

全ツール が同じルールで動作 → 一貫性確保
```

### Q2: 各ツール専用ファイル（GEMINI.md, CLAUDE.md）まで作らないといけないか？

**A:** AGENTS.md だけで動作しますが、各ツール専用ファイルを作ると以下のメリットが得られます：

```
AGENTS.md → 汎用（全ツール対応、広く浅く）
GEMINI.md → Gemini 最適化（詳細度が高い）
CLAUDE.md → Claude 最適化（詳細度が高い）
```

導入初期は AGENTS.md だけで十分。成熟段階で各ツール専用ファイルを追加することをお勧めします。

### Q3: AGENTS.md のサイズが大きくなりすぎないか？

**A:** 1500行くらいまでなら問題ありません。それ以上になる場合は分割を検討：

```
AGENTS.md（ルート）
    ├─ /docs/agents-backend.md（Backend 専用）
    ├─ /docs/agents-frontend.md（Frontend 専用）
    └─ /docs/agents-devops.md（DevOps 専用）
```

各チームが自分のドメイン設定を持つことで、認知負荷を減らせます。
