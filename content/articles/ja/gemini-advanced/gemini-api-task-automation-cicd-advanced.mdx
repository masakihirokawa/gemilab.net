## はじめに — 手動実行から自動パイプラインへ

[前編](/articles/gemini-api/gemini-ai-daily-task-automation-guide)では、Gemini APIを使って日常業務を自動化する基本パターンを紹介しました。この後編では、それらのプロンプトを本番環境で安定運用するための実践テクニックに踏み込みます。

具体的には、GitHub ActionsにGemini APIを統合してPRワークフローを完全自動化する方法、構造化出力で応答を確実にパースする方法、そしてプロンプトの品質を体系的に改善するチューニング手法を解説します。

## GitHub ActionsにGemini APIを統合する

PR作成時にGeminiが自動でレビューコメントを付けるワークフローを構築しましょう。

### ワークフロー定義

```yaml
# .github/workflows/gemini-review.yml
name: Gemini Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: pip install google-genai

      - name: Run Gemini Review
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: python .github/scripts/gemini_review.py
```

### レビュースクリプト

```python
# .github/scripts/gemini_review.py
import os
import subprocess
import json
from google import genai

client = genai.Client()

# PRの差分を取得
diff = subprocess.run(
    ["git", "diff", "origin/main...HEAD", "--", ".", ":!package-lock.json"],
    capture_output=True, text=True
).stdout

if not diff.strip():
    print("No meaningful changes to review")
    exit(0)

# 構造化出力でレビューを生成
prompt = f"""以下のコード差分をレビューし、JSON形式で結果を返してください。

## 出力形式
{{
  "summary": "変更全体の概要（1〜2文）",
  "issues": [
    {{
      "severity": "critical|warning|suggestion",
      "file": "ファイルパス",
      "description": "指摘内容"
    }}
  ],
  "positive": ["良い点を1〜2つ"]
}}

## レビュー観点
- バグの可能性（severity: critical）
- パフォーマンスへの影響（severity: warning）
- 可読性・保守性の改善（severity: suggestion）

## diff
{diff[:15000]}
"""

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=prompt,
    config={"response_mime_type": "application/json"}
)

review = json.loads(response.text)

# PRにコメントとして投稿
comment_body = f"## Gemini Code Review\\n\\n"
comment_body += f"**概要**: {review['summary']}\\n\\n"

if review.get("issues"):
    comment_body += "### 指摘事項\\n"
    for issue in review["issues"]:
        icon = {"critical": "🔴", "warning": "🟡", "suggestion": "💡"}
        comment_body += f"- {icon.get(issue['severity'], '📝')} **{issue['file']}**: {issue['description']}\\n"

if review.get("positive"):
    comment_body += "\\n### 良い点\\n"
    for p in review["positive"]:
        comment_body += f"- ✅ {p}\\n"

# gh CLIでPRにコメント
pr_number = os.environ.get("GITHUB_REF", "").split("/")[2] if "pull" in os.environ.get("GITHUB_REF", "") else None
if pr_number:
    subprocess.run(
        ["gh", "pr", "comment", pr_number, "--body", comment_body],
        check=True
    )
    print(f"✅ Review posted to PR #{pr_number}")
# 出力: ✅ Review posted to PR #42
```

## 構造化出力（JSON Mode）で応答を確実にパースする

AIの応答をプログラムで処理する場合、自由形式のテキストでは解析が不安定になります。Gemini APIの**JSON Mode**を使えば、スキーマに沿った構造化データを確実に取得できます。

### response_mime_typeの指定

```python
from google import genai

client = genai.Client()

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="Node.js 22の主要な新機能を3つ挙げてください",
    config={
        "response_mime_type": "application/json",
        "response_schema": {
            "type": "object",
            "properties": {
                "features": {
                    "type": "array",
                    "items": {
                        "type": "object",
                        "properties": {
                            "name": {"type": "string"},
                            "description": {"type": "string"},
                            "impact": {"type": "string", "enum": ["high", "medium", "low"]}
                        },
                        "required": ["name", "description", "impact"]
                    }
                }
            },
            "required": ["features"]
        }
    }
)

data = json.loads(response.text)
for f in data["features"]:
    print(f"[{f['impact'].upper()}] {f['name']}: {f['description']}")
# 出力例:
# [HIGH] require(esm): CommonJSからESModulesを直接requireできるようになった
# [HIGH] V8エンジン13.0: パフォーマンス改善とメモリ使用量の削減
# [MEDIUM] テストランナー安定版: node:testがプロダクション対応に
```

### バッチ処理でコストを最適化

複数のタスクを一括処理する場合、非同期バッチを活用するとスループットが向上します。

```python
import asyncio
from google import genai

client = genai.Client()

async def analyze_file(filepath: str, content: str) -> dict:
    """1ファイルを分析して結果を返す"""
    prompt = f"以下のコードの品質スコア(1-10)と改善点をJSON形式で返してください:\n\n{content[:5000]}"

    response = client.models.generate_content(
        model="gemini-2.5-flash",
        contents=prompt,
        config={"response_mime_type": "application/json"}
    )
    return {"file": filepath, "analysis": json.loads(response.text)}

async def batch_review(files: dict[str, str]) -> list:
    """複数ファイルを並行レビュー"""
    tasks = [analyze_file(path, content) for path, content in files.items()]
    return await asyncio.gather(*tasks)

# 使用例
# results = asyncio.run(batch_review({"app.py": "...", "utils.py": "..."}))
```

## プロンプトチューニングの体系的アプローチ

プロンプトは一度書いたら終わりではなく、継続的に改善するものです。以下のフレームワークで品質を体系的に向上させましょう。

### Few-shotプロンプトで出力品質を安定させる

期待する出力の例を2〜3個添えることで、応答の形式と品質が大幅に安定します。

```python
prompt = """以下のgitコミットメッセージから、ユーザー向けリリースノートの1行を生成してください。

## 例
入力: "fix(auth): handle expired refresh tokens gracefully"
出力: "認証: リフレッシュトークンの期限切れ時にスムーズに再認証できるようになりました"

入力: "feat(search): add fuzzy matching support"
出力: "検索: あいまい検索に対応し、タイプミスがあっても目的の結果が見つかります"

入力: "perf(api): cache user preferences for 5min"
出力: "パフォーマンス: ユーザー設定の読み込みが高速化されました"

## 対象
入力: "fix(upload): retry failed chunk uploads with exponential backoff"
出力: """

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=prompt
)
print(response.text)
# 出力例: "アップロード: 途中で失敗したファイルのアップロードが自動的にリトライされるようになりました"
```

### System Instructionsで一貫した人格を設定する

すべてのリクエストに共通のコンテキストを与えるには、System Instructionsを活用します。

```python
response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="このPRをレビューしてください: ...",
    config={
        "system_instruction": """あなたはシニアソフトウェアエンジニアです。
コードレビューでは以下のスタイルを守ってください:
- 指摘は具体的に（ファイル名・行番号を含む）
- 改善案を必ず添える（問題の指摘だけで終わらない）
- 良い点も1つ以上褒める
- 敬語は使わず、フランクだが丁寧な口調で"""
    }
)
```

## よくある質問

### Q: GitHub ActionsでAPIキーを安全に管理するにはどうすればよいですか？

リポジトリの Settings → Secrets and variables → Actions に`GEMINI_API_KEY`を登録します。ワークフローファイル内では`${{ secrets.GEMINI_API_KEY }}`で参照できます。シークレットはログに出力されないよう自動的にマスクされます。

### Q: 差分が大きすぎてAPIのトークン制限を超える場合はどうすればよいですか？

差分を15,000文字程度にトリミングするか、変更ファイル単位で分割してバッチ処理します。Gemini 2.5 Flashのコンテキストウィンドウは100万トークンと非常に大きいため、通常のPRで上限に達することは稀ですが、モノレポの大規模リファクタリングでは分割を検討してください。

### Q: AIのレビュー結果をそのまま信用してよいですか？

AIのレビューは「下書き」として扱い、最終判断は人間が行うのがベストプラクティスです。特にセキュリティやビジネスロジックに関わる指摘は、必ず人間の目で確認してください。AIは定型的なパターン（未使用変数、エラーハンドリング漏れ）の検出に特に強く、人間はアーキテクチャ判断やビジネス要件との整合性に集中するという役割分担が効果的です。
