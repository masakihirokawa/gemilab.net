## はじめに — なぜ ADK × TypeScript なのか

AI エージェントの構築は、単一の LLM 呼び出しから**複数のエージェントが協調して動作するシステム**へと進化しています。Google が提供する **Agent Development Kit（ADK）** は、こうしたマルチエージェントシステムをコードファーストで構築するためのオープンソースフレームワークです。

従来は Python 版のみでしたが、2026年に入り **TypeScript 版（`@google/adk`）** が正式にリリースされました。Web アプリケーションや Node.js バックエンドを TypeScript で開発しているチームにとって、同じ言語でエージェントを構築できるメリットは計り知れません。

この記事では、ADK TypeScript を使って **SequentialAgent・ParallelAgent・LoopAgent** の3種類のワークフローエージェントを活用したマルチエージェントシステムを、ゼロから構築する方法を解説します。

**この記事で学べること:**

- ADK TypeScript のセットアップと基本概念
- `LlmAgent` にカスタムツール（Function Calling）を追加する方法
- `SequentialAgent`、`ParallelAgent`、`LoopAgent` の使い分け
- エージェント間でのステート（状態）の受け渡し
- MCP ツール連携とデプロイパターン
- コスト・パフォーマンスの最適化手法

**対象読者:** TypeScript での開発経験があり、Gemini API の基本を理解している中〜上級者

## 前提知識・環境構築

### 必要な環境

- **Node.js** 20 以上（LTS 推奨）
- **TypeScript** 5.x
- **Gemini API キー**（[Google AI Studio](https://aistudio.google.com/) で取得）

### プロジェクトの作成

ターミナルで以下のコマンドを実行し、プロジェクトを初期化します。

```bash
mkdir adk-multi-agent && cd adk-multi-agent

# パッケージ初期化
npm init -y
npm pkg set type="module"

# ADK と開発ツールのインストール
npm install @google/adk
npm install -D @google/adk-devtools typescript @types/node

# TypeScript 設定
npx tsc --init --module nodenext --moduleResolution nodenext --target es2022 --outDir dist

# API キーの設定
export GOOGLE_API_KEY="your-api-key-here"
```

### ディレクトリ構成

ADK プロジェクトの推奨構成は以下のとおりです。

```
adk-multi-agent/
├── agent.ts          # ルートエージェント（エントリーポイント）
```

## 概念と設計思想 — ADK のアーキテクチャ

### 3 つのエージェントタイプ

ADK には大きく分けて **LLM エージェント** と **ワークフローエージェント** の2系統があります。

**LlmAgent（LLM エージェント）:**
Gemini モデルを呼び出し、推論・ツール実行・対話を行う中核的なエージェントです。`instruction`（システムプロンプト）、`tools`（利用可能なツール）、`model`（使用モデル）を定義します。

**ワークフローエージェント（3種類）:**

| エージェント | 用途 | 実行パターン |
|---|---|---|
| `SequentialAgent` | 決まった順序で処理を実行 | A → B → C |
| `ParallelAgent` | 独立した処理を同時に実行 | A, B, C を並列 |
| `LoopAgent` | 条件を満たすまで反復 | (A → B) × N 回 |

### ステート管理

エージェント間のデータ受け渡しには **ステート（`context.state`）** を使います。`LlmAgent` の `outputKey` プロパティを指定すると、そのエージェントの最終応答がステートに自動保存されます。

```typescript
// writer エージェントの出力を "draft" キーに保存
const writerAgent = new LlmAgent({
  name: "writer",
  model: "gemini-2.5-flash",
  instruction: "与えられたデータに基づいてレポートを書いてください。",
  outputKey: "draft"
});

// 後続のエージェントで読み取り
// context.state["draft"] で取得可能
```

## ステップバイステップ実装

### Step 1: カスタムツールの定義

まず、エージェントが使用するツールを `FunctionTool` で定義します。ADK TypeScript では `zod` スキーマでパラメータを型安全に定義できます。

```typescript
// tools/search.ts
import { FunctionTool } from "@google/adk";
import { z } from "zod";

export const webSearchTool = new FunctionTool({
  name: "web_search",
  description: "指定されたキーワードで Web 検索を行い、結果を返す",
  parameters: z.object({
    query: z.string().describe("検索クエリ"),
    maxResults: z.number().default(5).describe("取得する結果の最大数")
  }),
  handler: async ({ query, maxResults }) => {
    // 実際の検索 API 呼び出し（ここでは例示）
    console.log(`検索中: ${query} (最大 ${maxResults} 件)`);

    // Google Custom Search API など実際の API を呼び出す
    const results = [
      { title: `${query} に関する最新情報`, url: "https://example.com/1", snippet: "最新の動向について..." },
      { title: `${query} 完全ガイド`, url: "https://example.com/2", snippet: "包括的な解説..." }
    ];

    return JSON.stringify(results.slice(0, maxResults));
  }
});
```

```typescript
// tools/analyzer.ts
import { FunctionTool } from "@google/adk";
import { z } from "zod";

export const sentimentAnalyzer = new FunctionTool({
  name: "analyze_sentiment",
  description: "テキストの感情分析を行い、ポジティブ/ネガティブ/ニュートラルを判定する",
  parameters: z.object({
    text: z.string().describe("分析対象のテキスト")
  }),
  handler: async ({ text }) => {
    // Gemini API を直接使って分析する場合の例
    const wordCount = text.split(/\s+/).length;
    const positiveWords = ["excellent", "great", "good", "素晴らしい", "良い"];
    const negativeWords = ["bad", "terrible", "poor", "悪い", "問題"];

    let score = 0;
    positiveWords.forEach(w => { if (text.includes(w)) score++; });
    negativeWords.forEach(w => { if (text.includes(w)) score--; });

    return JSON.stringify({
      sentiment: score > 0 ? "positive" : score < 0 ? "negative" : "neutral",
      confidence: Math.min(Math.abs(score) * 0.2 + 0.5, 1.0),
      wordCount
    });
  }
});
```

### Step 3: マルチエージェントパイプラインの構築

3つのワークフローエージェントを組み合わせて、本格的なパイプラインを構築します。

```typescript
// agent.ts（ルートエージェント）
import { SequentialAgent, ParallelAgent, LoopAgent, LlmAgent } from "@google/adk";
import { researcherAgent } from "./agents/researcher.js";
import { analystAgent } from "./agents/analyst.js";
import { writerAgent } from "./agents/writer.js";

// --- パターン1: SequentialAgent（順次実行）---
// リサーチ → 分析 → レポート作成の流れ
const sequentialPipeline = new SequentialAgent({
  name: "research_pipeline",
  description: "トピックの調査・分析・レポート作成を順次実行する",
  subAgents: [researcherAgent, analystAgent, writerAgent]
});

// --- パターン2: ParallelAgent（並列実行）---
// 複数の観点から同時にリサーチ
const marketResearcher = new LlmAgent({
  name: "market_researcher",
  model: "gemini-2.5-flash",
  instruction: "市場動向と競合分析の観点でリサーチしてください。",
  outputKey: "market_data"
});

const techResearcher = new LlmAgent({
  name: "tech_researcher",
  model: "gemini-2.5-flash",
  instruction: "技術的なトレンドとイノベーションの観点でリサーチしてください。",
  outputKey: "tech_data"
});

const parallelResearch = new ParallelAgent({
  name: "parallel_research",
  description: "市場と技術の両面から同時にリサーチする",
  subAgents: [marketResearcher, techResearcher]
});

// --- パターン3: LoopAgent（反復改善）---
// レビュー → 修正を品質基準を満たすまで繰り返す
const reviewerAgent = new LlmAgent({
  name: "reviewer",
  model: "gemini-2.5-pro",
  instruction: `ステートの "draft" を読み、以下の基準で評価してください:
- 正確性、論理的一貫性、読みやすさ
- 問題がなければ "APPROVED" と出力
- 問題があれば具体的な改善点を列挙
評価が完了したら結果を返してください。`,
  outputKey: "review_feedback"
});

const editorAgent = new LlmAgent({
  name: "editor",
  model: "gemini-2.5-flash",
  instruction: `"review_feedback" のフィードバックをもとに "draft" を改善してください。
"APPROVED" が含まれていれば、そのまま出力してください。`,
  outputKey: "draft"
});

const refinementLoop = new LoopAgent({
  name: "refinement_loop",
  description: "レビューと修正を繰り返してレポートの品質を向上させる",
  subAgents: [reviewerAgent, editorAgent],
  maxIterations: 3
});

// --- ルートエージェント: 全体を統合 ---
export const rootAgent = new SequentialAgent({
  name: "intelligent_report_generator",
  description: "並列リサーチ → 分析 → 執筆 → 反復改善の全工程を実行するマルチエージェントシステム",
  subAgents: [
    parallelResearch,  // 並列リサーチ
    analystAgent,      // 分析
    writerAgent,       // 執筆
    refinementLoop     // 反復改善
  ]
});
```

### Step 4: 実行とテスト

ADK DevTools を使って、ブラウザ上で対話的にテストできます。

```bash
# ターミナルで対話的に実行
npx @google/adk-devtools run agent.ts

# Web UI で実行（http://localhost:8000）
npx @google/adk-devtools web agent.ts
```

プログラムから実行する場合は、`Runner` と `InMemorySessionService` を使います。

```typescript
// run.ts
import { Runner, InMemorySessionService } from "@google/adk";
import { rootAgent } from "./agent.js";

async function main() {
  const sessionService = new InMemorySessionService();
  const runner = new Runner({
    agent: rootAgent,
    appName: "report-generator",
    sessionService
  });

  // セッション作成
  const session = await sessionService.createSession({
    appName: "report-generator"
  });

  // ユーザー入力を送信して実行
  const response = runner.runAsync({
    userId: "user-1",
    sessionId: session.id,
    newMessage: {
      role: "user",
      parts: [{ text: "Gemini API の最新動向についてレポートを作成してください" }]
    }
  });

  // ストリーミングでイベントを受信
  for await (const event of response) {
    if (event.content?.parts) {
      for (const part of event.content.parts) {
        if (part.text) {
          console.log(`[${event.author}] ${part.text}`);
        }
      }
    }
  }
}

main().catch(console.error);
```

## 応用パターン

### MCP ツール連携

ADK TypeScript は **Model Context Protocol（MCP）** との連携をサポートしています。既存の MCP サーバーをエージェントのツールとして接続できます。

```typescript
import { LlmAgent, MCPTool } from "@google/adk";

// MCP サーバーから取得したツールを使用
const mcpTool = new MCPTool({
  serverUrl: "http://localhost:3001",
  toolName: "database_query",
  description: "データベースに対して SQL クエリを実行する"
});

const dataAgent = new LlmAgent({
  name: "data_agent",
  model: "gemini-2.5-flash",
  instruction: "ユーザーの質問に基づいてデータベースをクエリし、結果を返してください。",
  tools: [mcpTool]
});
```

### エージェントをツールとして再利用

ADK では、あるエージェントを別のエージェントの **ツール** として使うことができます。これにより、モジュラーな設計が可能になります。

```typescript
import { LlmAgent, AgentTool } from "@google/adk";

// 翻訳専門エージェント
const translatorAgent = new LlmAgent({
  name: "translator",
  model: "gemini-2.5-flash",
  instruction: "与えられたテキストを日本語から英語に翻訳してください。"
});

// メインエージェントが翻訳エージェントをツールとして使用
const mainAgent = new LlmAgent({
  name: "main_assistant",
  model: "gemini-2.5-pro",
  instruction: "ユーザーのリクエストに応じて翻訳ツールを使ってください。",
  tools: [new AgentTool({ agent: translatorAgent })]
});
```

## トラブルシューティング

### よくあるエラーと対処法

**`GOOGLE_API_KEY is not set` エラー:**
環境変数が設定されていない場合に発生します。`.env` ファイルを使用する場合は `dotenv` パッケージを導入するか、ターミナルで直接 `export GOOGLE_API_KEY="..."` を実行してください。

**`Module not found` エラー:**
`package.json` に `"type": "module"` が設定されていること、`tsconfig.json` で `module` が `nodenext` になっていることを確認してください。インポートパスには `.js` 拡張子を明記する必要があります。

**LoopAgent が無限ループする:**
`maxIterations` を必ず設定してください。また、レビューエージェントの instruction に「条件を満たしたら APPROVED と出力する」など、明確な終了条件を記述することが重要です。

**ステートが後続エージェントに渡らない:**
`outputKey` を設定し忘れていないか確認してください。`outputKey` を指定しないと、エージェントの出力はステートに保存されません。

## コスト・パフォーマンス考慮事項

### モデルの使い分けでコストを最適化

マルチエージェントシステムでは、各エージェントに適切なモデルを割り当てることがコスト最適化の鍵です。

| 用途 | 推奨モデル | 理由 |
|---|---|---|
| 情報収集・検索 | `gemini-2.5-flash` | 高速・低コスト |
| 分析・推論 | `gemini-2.5-pro` | 高精度な推論が必要 |
| テキスト生成 | `gemini-2.5-flash` | 大量テキスト生成に適する |
| 品質レビュー | `gemini-2.5-pro` | 批判的思考が必要 |

### トークン消費の目安

SequentialAgent で 3 エージェントを直列実行する場合、各エージェントが約 1,000〜3,000 トークンを消費すると仮定すると、1 回のパイプライン実行で約 3,000〜9,000 トークンが必要です。ParallelAgent は並列実行のためレイテンシは短縮されますが、トークン消費量は変わりません。

### レイテンシ最適化のポイント

- `ParallelAgent` を活用して独立したタスクを同時実行する
- `LoopAgent` の `maxIterations` は必要最小限に設定する
- Flash モデルをメインに使い、Pro は判断が難しいタスクに限定する
- `outputKey` で必要なデータだけをステートに保存し、コンテキストの肥大化を防ぐ

## FAQ

### Q1: ADK TypeScript と Python 版の機能差はありますか？

コアとなるエージェントタイプ（`LlmAgent`、`SequentialAgent`、`ParallelAgent`、`LoopAgent`）は TypeScript 版でも完全にサポートされています。ただし、一部の実験的機能やサードパーティ連携は Python 版が先行しているケースがあります。2026年3月時点では、ほぼ同等の機能が利用可能です。

### Q2: Gemini 以外のモデル（Claude、GPT-4o など）も使えますか？

ADK は Gemini モデルに最適化されていますが、アーキテクチャ上は他の LLM プロバイダーへの拡張が可能です。ただし、Function Calling やストリーミングなどの機能は Gemini API に依存するため、完全な互換性は保証されません。

### Q3: 本番環境へのデプロイ方法は？

ADK エージェントは以下の方法でデプロイできます。（1）**Cloud Run** にコンテナとしてデプロイ、（2）**Vertex AI Agent Engine** でマネージドサービスとして運用、（3）独自の Node.js サーバー上で Express/Fastify と統合して運用。本番環境では Vertex AI Agent Engine がスケーラビリティとモニタリングの観点で推奨されます。

### Q4: エージェント間のエラーハンドリングはどうすればよいですか？

`SequentialAgent` 内のサブエージェントでエラーが発生した場合、デフォルトではパイプライン全体が停止します。堅牢なシステムを構築するには、各エージェントの `instruction` にエラー時の振る舞い（フォールバック値の生成、エラーメッセージの記録など）を明示的に記述してください。

### Q5: LoopAgent の終了条件はどう設計すべきですか？

`maxIterations` による上限設定は必須です。加えて、レビューエージェントが特定のキーワード（例: "APPROVED"）を出力した場合にループを終了するロジックを `instruction` に組み込むのがベストプラクティスです。ADK には組み込みの `exit_loop` ツールも用意されており、エージェントが自律的にループを終了できます。

## まとめ

Google ADK の TypeScript 版を使えば、Web 開発者にとって馴染み深い言語でマルチエージェント AI システムを構築できます。`SequentialAgent` で順次処理、`ParallelAgent` で並列処理、`LoopAgent` で反復改善という3つのパターンを組み合わせることで、単一の LLM では実現できない高度なワークフローを実装できます。

まずは ADK DevTools の Web UI で小さなエージェントから試し、段階的にマルチエージェント構成へと拡張していくのがおすすめです。Gemini API の [Function Calling](/articles/gemini-advanced/function-calling-guide) や [ストリーミング](/articles/gemini-api/streaming-and-chat) と組み合わせれば、さらに強力なシステムが構築できるでしょう。

ADK のエコシステムは急速に成長しており、[MCP 連携](/articles/gemini-dev/gemini-mcp-integration) や [Vertex AI デプロイ](/articles/gemini-api/vertex-ai-quickstart) など、今後もさらなる進化が期待されます。
