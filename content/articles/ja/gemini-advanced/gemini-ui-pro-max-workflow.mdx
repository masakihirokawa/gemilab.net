## はじめに

UI 設計の工程は、本来であれば複数のステップを踏むものです。要件整理 → ワイヤーフレーム → ローフィデリティ → ハイフィデリティという流れですが、この過程で何度も修正が入り、実装まで時間がかかってしまう——そのような課題を抱えたチームは多いでしょう。

**UI Pro Max** は、Gemini の高度なデザイン生成機能を活かし、この段階的な設計フローを大幅に加速させるスキルです。本記事では、実際のアプリ開発プロジェクトを通じて、その実践的な使用方法を解説します。

## UI Pro Max スキルの全体像

### スキルの核となる機能

UI Pro Max は、以下の3つのコア機能で構成されています。

1. **ワイヤーフレーム自動生成**：テキストベースの要件から、レイアウトスケッチを自動化
2. **デザイン提案・バリエーション生成**：複数の UI アプローチを AI が考案
3. **デザイントークン自動抽出**：色・フォント・スペーシング規則を機械可読フォーマットで出力

これらを組み合わせることで、デザイナーとエンジニアの共同作業がより円滑になります。

## 実践ワークフロー：段階的設計プロセスの詳細

### フェーズ 1：要件定義と情報設計

まずは、プロダクトの全体像を Gemini に提示します。ここで重要なのは、できるだけ**具体的で詳細な要件**を伝えることです。曖昧な要件は、AI の出力品質を低下させます。

**入力例（詳細版）：**
```
「オンラインショップの商品詳細ページを設計したいです。

【ビジネス要件】
- ターゲット：20代～40代の女性（月収30万以上）
- 目標：商品のカートイン率を現在の15%から25%に向上
- 競合分析：Amazon、楽天、Qoo10 を参考に

【掲載すべき要素】
優先度：高
- 商品画像（複数枚、Zoom機能付き）
- 商品名・説明（500字以内）
- 価格・割引情報（セール価格、割引率の表示）
- レーティング・レビュー（星5つ、件数表示）
- カートに追加ボタン

優先度：中
- サイズ・色の選択肢（プルダウン）
- 在庫状況表示
- 配送料・配送日数情報

優先度：低
- 購入者のレビュー一覧（スクロール可）
- 関連商品提案（横スクロールカルーセル）

【デバイス対応】
- スマートフォン（375px - 420px）
- タブレット（768px - 1024px）
- デスクトップ（1280px以上）

【デザイン要件】
- イメージ：モダンで信頼感あり、明るい雰囲気
- カラー：ブルー基調（Primary #2563EB）
- ブランド：高級感のある商品向け

【ユーザーフロー】
- 商品を見つける → 詳細を確認 → 購入決定 → カートに追加
- 所要時間：2-3分での意思決定を想定」
```

Gemini は要件を分析し、以下の提案を返します：
- **情報設計（IA）**：要素の優先度、グループ化、配置順序
- **レイアウト案**：複数案の提示（縦スクロール型 / タブ型 など）
- **ユーザーフロー**：ページ遷移パターン、各タッチポイントでの操作

**Gemini への追加質問例：**
- 「画像と商品情報、どちらを上に配置すべき？」
- 「レビュー一覧はすぐに表示すべき、それとも折り畳むべき？」
- 「モバイル表示では、どの情報を省略すべき？」

### フェーズ 2：ワイヤーフレーム生成と視覚化

Gemini に「上記要件のワイヤーフレームをテキストで表現してください」と依頼すれば、ASCII アート や図解形式で、画面レイアウトの案が返ってきます。

**Gemini への指示例：**
```
「商品詳細ページのワイヤーフレームを ASCII アート で表現してください。
スマートフォン版（375px）、タブレット版（768px）それぞれで。

各セクションについて：
1. セクション名
2. 高さ比率（ビューポートの何%）
3. 内部レイアウト（左右配置か？ 上下配置か？）
4. レスポンシブ時の変更ルール
を明記してください。」
```

**Gemini の出力例（スマートフォン版）：**
```
┌─────────────────────────────┐
│   ヘッダー（h: 12%）           │ ← ナビゲーション、検索
├─────────────────────────────┤
│                             │
│   商品画像（h: 40%）           │ ← Zoom可能、サムネイル付き
│                             │
├─────────────────────────────┤
│ 商品名（h: 8%）              │
├─────────────────────────────┤
│ 価格・割引（h: 6%）            │
├─────────────────────────────┤
│ レーティング（h: 6%）          │
├─────────────────────────────┤
│ サイズ・色選択（h: 12%）        │
├─────────────────────────────┤
│ 『カートに追加』（h: 8%）       │
├─────────────────────────────┤
│ 商品説明（h: スクロール）      │
└─────────────────────────────┘
```

その後、UI Pro Max スキルを実行することで、より詳細なワイヤーフレームが Figma レイアウトとして生成されます。

**出力される情報：**
- 各セクションのサイズ比率（ピクセル単位）
- 要素配置の論理的根拠（「なぜこの順序か」の説明）
- レスポンシブ時の変更ルール（375px 時は2列 → 768px では3列 など）
- グリッドシステム提案（8px / 12px / 16px ベース）

**Gemini への追加質問：**
- 「タブレット版での最適なレイアウトは？」
- 「スクロール距離を最小化するにはどう配置すべき？」
- 「ホワイトスペースの比率は？」

### フェーズ 3：ローフィデリティデザイン（グレースケール化）

スケッチの段階から、簡易的なグレースケール UI へと進みます。この段階では、**色や装飾を排除し、純粋なレイアウト・情報構造に集中**します。

**Gemini への指示例：**
```
「上記ワイヤーフレームをローフィデリティ UI に仕上げてください。

仕様：
- グレースケール（黒 #000、灰色 #666、白 #FFF のみ）
- 3階層の情報階層を反映（見出し：太い、本文：細い、補助情報：薄い）
- テキストは Lorem Ipsum でも構いません
- 約 20-30 コンポーネント（Button, Input, Card, Badge など）

提案してほしい内容：
1. 各セクションの視覚的階層
2. ホワイトスペースの配置
3. グリッドシステム（何px ベース？）
4. テキストの大きさ比率
5. ボタンのサイズ（推奨：最小 44px × 44px）」
```

**出力されるガイドライン例：**
```
グリッドシステム：8px ベース（推奨）

見出し階層：
- H1: 32px, Weight 700（商品名）
- H2: 24px, Weight 600（セクション見出し）
- H3: 18px, Weight 500（小見出し）

本文：
- Body: 16px, Weight 400, Line Height 1.5
- Small: 14px, Weight 400

ホワイトスペース：
- セクション間：32px（8px × 4）
- 要素内パディング：16px（8px × 2）
- リスト要素間：12px（8px × 1.5）
```

実際には、Figma 上で手動で作成することになりますが、Gemini の提案を参考にすれば、設計時間が大幅に短縮されます。

**実装のチェックリスト：**
- [ ] グリッドシステムを設定（Figma の Grid パネル）
- [ ] 色をグレースケール 3段階に限定
- [ ] テキストサイズの比率を設定（8px, 12px, 16px, 24px, 32px）
- [ ] タッチターゲットを 44px 以上に確保
- [ ] 複数デバイスでの表示確認（375px, 768px, 1280px）

### フェーズ 5：Figma への反映と実装準備

生成されたデザイン案を Figma に段階的に反映していきます。

**詳細なプロセス：**

**Step 1：デザイントークンの Figma Variables への登録**
```
Figma 操作：
1. 新規ファイルを作成
2. 右側パネル → 「Design」→「Variables」を開く
3. 「Create new collection」で「Colors」を作成
4. Gemini から提供された色トークンを登録
   - Primary: #2563EB
   - Secondary: #10B981
   - Neutral: 複数段階（50, 100, 400, 500, 700, 900）
5. 同様に「Typography」「Spacing」コレクションを作成
```

**Step 2：コンポーネントライブラリの構築**

Gemini の提案をベースに、段階的にコンポーネントを構築します：

```
基本コンポーネント層（優先度：高）：
├─ Button
│  ├─ Variant: Style (Primary, Secondary, Ghost)
│  ├─ Variant: Size (Small, Medium, Large)
│  └─ Variant: State (Default, Hover, Disabled)
├─ Input
│  ├─ Type: Text, Email, Password, Number
│  └─ State: Default, Focus, Error
├─ Badge / Label
├─ Avatar
└─ Rating

複合コンポーネント層（優先度：中）：
├─ Card
│  ├─ Image + Title + Description
│  ├─ Product Card（画像、価格、レーティング）
│  └─ Review Card（ユーザー画像、テキスト、星評価）
├─ FormGroup (Label + Input + Helper Text)
├─ PriceDisplay
│  ├─ Variant: Original / Sale Price
│  └─ Variant: Discount Badge 付き
└─ ImageCarousel
   └─ Thumbnail + Large Image
```

**Step 3：画面全体の組み立て**

```
商品詳細ページの構成：
1. Header Section
   - 商品画像（メイン） + サムネイル
   - Zoom, Slide 機能

2. Info Section
   - 商品名（H1）
   - レーティング + レビュー数
   - 価格表示（PriceDisplay コンポーネント使用）

3. Action Section
   - サイズ・色選択（FormGroup）
   - 『カートに追加』ボタン
   - 『ウィッシュリスト』ボタン

4. Description Section
   - 商品説明（タイポグラフィ：Body）
   - 仕様表（テーブル）

5. Reviews Section
   - レビュー一覧（ReviewCard コンポーネント）
   - 『さらに見る』ボタン

6. Related Products Section
   - 関連商品カルーセル（ProductCard コンポーネント）
```

Gemini が「このボタンコンポーネントは、primary / secondary / ghost の3 バリエーションを持つべき」と提案してくれるので、実装効率が大きく向上します。また、「ホバー状態では背景色が X% 明るくなる」といった具体的な指示も得られるため、微調整の時間が削減されます。

**Step 4：レスポンシブ対応の実装**

複数デバイスサイズでフレームを複製し、レスポンシブ対応を確認：

```
デバイスサイズ別フレーム作成：
├─ Mobile (375px / iPhone SE)
├─ Mobile L (414px / iPhone 13)
├─ Tablet (768px / iPad)
└─ Desktop (1280px / 27" Monitor)

確認項目：
- テキストリフロー：各サイズで読みやすいか
- 画像比率：縦横比が保たれているか
- ボタン位置：アクセスしやすいか
- スクロール距離：必要最小限か
```

## デザインシステムの構築と段階的スケーリング

UI Pro Max の強力な点は、**デザインシステム化** を視野に入れていることです。単一プロジェクト向けのデザイン生成にとどまらず、組織全体で再利用可能なシステムの基盤を構築できます。

### コンポーネント共通化の戦略

複数の画面を Gemini に送信し、「共通化できるコンポーネントを抽出してください」と依頼すれば、以下のような分類が返ってきます。

**Gemini への指示例：**
```
「以下の 3 画面を見て、共通化できるコンポーネントを抽出してください。

【提供物】
- 商品詳細ページ（スクリーンショット）
- カテゴリ一覧ページ（スクリーンショット）
- ユーザープロフィールページ（スクリーンショット）

【要求】
1. 3 画面で共通して使われている UI パターン
2. 各パターンの最小単位（アトミックデザイン的に）
3. バリエーション（色、サイズ、状態）の整理
4. 命名規則の提案（Button, ProductCard など）
5. 各コンポーネントの仕様書テンプレート」
```

**Gemini の出力例：**
```
【基本コンポーネント】
- Button (Primary, Secondary, Ghost) × (Small, Medium, Large)
- Input (Text, Email, Select) × (Default, Focus, Error)
- Badge (Success, Warning, Error)
- Avatar (Circle, Square) × (16px, 32px, 48px)

【複合コンポーネント】
- Card
  - ProductCard: Image + Title + Price + Rating
  - ReviewCard: Avatar + Name + Rating + Text
  - CategoryCard: Image + Name + Count
- FormGroup: Label + Input + HelperText

【パターン】
- PriceDisplay: Original × Sale (割引率 Badge 付き)
- RatingDisplay: Stars × Count
- ImageGallery: Main + Thumbnails
- ReviewSection: List + Filter + Pagination

【命名規則】
Component / Property / Value
例：Button / Style / Primary
例：ProductCard / Layout / Vertical
```

### トークン規則の文書化と運用

「このデザイントークンの命名規則の根拠を説明してください」と聞けば、なぜそのような階層・名前を付けたのかの論理が明確化されます。これにより、新しいチームメンバーへのオンボーディングが容易になります。

**トークン命名規則の例：**
```
色：
[Scope] / [Hue] / [Shade]
primary / blue / 500
primary / blue / 600
状態別：
primary / blue / default
primary / blue / hover
primary / blue / active
primary / blue / disabled

スペーシング：
[Size Name]
xs（4px）, sm（8px）, md（16px）, lg（24px）, xl（32px）

フォント：
[Role] / [Property]
heading / size-1 → 36px
heading / weight → 700
body / size → 16px
body / line-height → 1.5
```

### デザインシステムドキュメントの自動生成

Gemini に以下のように依頼することで、デザイナー・開発者向けのドキュメントを自動生成できます：

```
「上記コンポーネント一覧を基に、デザインシステムドキュメントを作成してください。

目次：
1. デザイントークン定義（色、フォント、スペーシング）
2. コンポーネント一覧（仕様書テンプレート）
3. 使用例（各コンポーネントの実装例）
4. よくある質問（FAQ）
5. 今後の拡張計画

形式：Markdown + コードブロック」
```

## 実装チームとの連携と自動化

### Dev Mode との連携ワークフロー

Figma の Dev Mode で、コンポーネントのサイズ・距離・フォント情報を自動抽出します。同時に、Gemini に「この CSS 実装に必要な design tokens を列挙してください」と依頼すれば、フロントエンド実装の準備が整います。

**Dev Mode での情報抽出例：**

| 情報種別 | 自動抽出内容 | 出力フォーマット |
|---------|----------|----------|
| 寸法 | 幅・高さ・マージン・パディング | px 値 |
| 色 | 背景色・テキスト色・ボーダー色 | HEX / RGB / HSL |
| タイポグラフィ | フォント名・サイズ・ウェイト・行間 | CSS プロパティ |
| エフェクト | シャドウ・ぼかし・透明度 | CSS filter / box-shadow |

**Figma Dev Mode の使用方法：**
```
1. Figma ファイルを開く
2. 右上の「Dev」タブをクリック
3. 対象コンポーネントを選択
4. 右側パネルに自動抽出された仕様が表示
5. 「Copy」ボタンで CSS / JSON をコピー
```

### CSS-in-JS への段階的変換

UI Pro Max が出力するトークン定義は、styled-components や Tailwind CSS への変換が可能です。

**Step 1：Gemini からのトークン定義（JSON）**
```json
{
  "colors": {
    "primary": {
      "500": "#2563EB",
      "600": "#1D4ED8"
    },
    "neutral": {
      "900": "#111827",
      "50": "#F9FAFB"
    }
  },
  "spacing": {
    "xs": "4px",
    "sm": "8px",
    "md": "16px"
  },
  "typography": {
    "heading": {
      "fontSize": "36px",
      "fontWeight": "700",
      "lineHeight": "1.3"
    }
  }
}
```

**Step 2：CSS-in-JS（styled-components 例）への変換**
```javascript
// tokens.js
export const tokens = {
  colors: {
    primary: {
      500: "#2563EB",
      600: "#1D4ED8",
    },
    neutral: {
      900: "#111827",
      50: "#F9FAFB",
    },
  },
  spacing: {
    xs: "4px",
    sm: "8px",
    md: "16px",
  },
  typography: {
    heading: {
      fontSize: "36px",
      fontWeight: "700",
      lineHeight: "1.3",
    },
  },
};

// Button.jsx
import styled from "styled-components";
import { tokens } from "./tokens";

export const StyledButton = styled.button`
  background-color: ${tokens.colors.primary[500]};
  padding: ${tokens.spacing.sm} ${tokens.spacing.md};
  font-size: ${tokens.typography.body.fontSize};

  &:hover {
    background-color: ${tokens.colors.primary[600]};
  }
`;
```

**Step 3：Tailwind CSS への変換**
```javascript
// tailwind.config.js
module.exports = {
  theme: {
    extend: {
      colors: {
        primary: {
          500: "#2563EB",
          600: "#1D4ED8",
        },
        neutral: {
          900: "#111827",
          50: "#F9FAFB",
        },
      },
      spacing: {
        xs: "4px",
        sm: "8px",
        md: "16px",
      },
    },
  },
};

// HTML/JSX での使用
<button className="bg-primary-500 hover:bg-primary-600 px-md py-sm">
  Click me
</button>
```

このデータを CSS 変数や Tailwind config に変換することで、設計と実装の乖離が最小化されます。さらに、Gemini に「この tokens.js を基に React コンポーネントテンプレートを生成してください」と依頼することで、実装の加速化が期待できます。

### Gemini への実装サポート依頼

```
「以下の design tokens を基に、React コンポーネントテンプレートを生成してください。

【要件】
- styled-components を使用
- TypeScript で型定義
- Props による色・サイズのバリエーション対応
- ホバー・アクティブ状態の自動適用

【出力形式】
- Button.jsx
- types/button.ts
- stories/Button.stories.tsx（Storybook）」
```

## ベストプラクティスと実装のコツ

### 1. 段階的フィードバックと反復

全体を一度に作るのではなく、各フェーズで Gemini にフィードバックを求めることが重要です。

**具体的な質問例：**
```
フェーズ 2（ワイヤーフレーム）:
「この配置では、ユーザーの視線の動きは最適ですか？
ファーストビューで最も重要な情報（商品名・価格）に視線が
向かうレイアウトになっていますか？」

フェーズ 4（ハイフィデリティ）:
「この配色、もっと落ち着いた・高級感のある印象にできますか？」
「このボタン配置は、モバイルビューで実用的ですか？
右手片手操作を想定して、アクセスしやすい位置ですか？」

実装前:
「このコンポーネント仕様は、React での実装を想定していますか？
Props や State の設計に問題ありませんか？」
```

**反復サイクルの目安：**
```
フェーズ 1: 2-3回のやり取り（要件確認、IA 検証）
フェーズ 2: 1-2回のやり取り（レイアウト最適化）
フェーズ 3: 1回（グレースケール確認）
フェーズ 4: 3-4回のやり取り（色・デテール調整）
フェーズ 5: 2-3回（実装視点からのレビュー）

合計所要時間：10-15回のやり取り = 約 4-6 時間
```

### 2. デザインシステムの早期確立

プロジェクト開始時に、Gemini と一緒にデザインシステムの基盤を構築しておくと、後々の修正が最小化されます。

**早期確立のメリット：**
- **複数プロジェクト間での再利用**：次のプロジェクトで同じシステムを使用可能
- **チーム全体の一貫性**：デザイナーが異なっても、ビジュアルは統一
- **実装効率の向上**：トークン化されたルール＝すぐに実装に着手可能
- **修正時間の削減**：「ボタンの色を全て変更」が 1 箇所の修正で完了

**システム確立の順序：**
```
1. デザイントークン定義（色、フォント、スペーシング）
2. 基本コンポーネント作成（Button, Input, Badge など）
3. 複合コンポーネント作成（Card, Form Group など）
4. ページテンプレート作成（ダッシュボード、詳細表示など）
5. ドキュメント整備（使用ガイド、FAQ、拡張方法）
```

### 3. 実装チームへの段階的な情報共有

デザイントークンとコンポーネント仕様を実装チームへ早期に共有することで、設計・実装の乖離を防げます。

**共有タイミング別の情報：**

| タイミング | 共有内容 | 形式 |
|----------|---------|------|
| フェーズ 2（ワイヤーフレーム） | ページ構造、データフロー | 図解、簡潔なドキュメント |
| フェーズ 4（ハイフィデリティ） | デザイントークン、コンポーネント仕様 | JSON、CSV、デザイン仕様書 |
| フェーズ 5（実装準備） | CSS / 実装例、アニメーション仕様 | コードサンプル、Storybook |

**Figma Dev Mode での共有例：**
```
1. Figma ファイルを「開く」権限で実装チームに共有
2. 「Dev」タブで各コンポーネントの寸法・色情報を確認可能
3. 実装チームが「Copy as CSS」で直接コードを生成
```

### 4. バージョン管理と修正ルール

デザインシステムが固まったら、バージョン管理を導入します：

```
v1.0 – 初期リリース（基本コンポーネント、トークン定義）
v1.1 – 小規模修正（色の微調整、フォントウェイト追加）
v2.0 – 大規模変更（コンポーネント削除、新しいアイコンセット）

修正ルール：
- 既存コンポーネントの小規模修正 → v1.x に含める
- 新しいコンポーネント追加 → v1.x に含める
- 既存コンポーネント削除・大規模変更 → v2.0 で実施
```

このルールにより、既存プロジェクトへの影響を最小化できます。

## まとめ

UI Pro Max は、単なる UI 生成ツールではなく、**デザインシステム思考を内包した、段階的・スケーラブルな UI 設計ワークフロー** を実現します。

このワークフローの実施により、以下が達成できます：

**効率面：**
- 要件整理から実装準備まで、一貫した流れが実現
- 従来の 50% 以下の時間でデザイン完成
- 修正・変更の際の作業時間が大幅削減

**品質面：**
- デザイン・実装チーム間の言語が統一される
- 曖昧さのない仕様書＝実装ズレが最小化
- アクセシビリティを最初から考慮したシステム

**スケーラビリティ：**
- デザインシステムが構築できれば、次のプロジェクトは数時間で完成
- チーム拡大時の新メンバー教育が容易
- 複数プロダクト間での一貫性が保証される

プロダクト開発に携わるなら、UI Pro Max の習熟は、**チーム全体の生産性向上に直結するスキル**です。ぜひ、次のプロジェクトで実践的に試してみてください。段階を重ねるごとに、プロセスがより洗練され、さらに高速・高品質なデザイン実現が期待できます。

LLMの仕組みや活用法についてより実践的な知識を身につけたい方には、「[大規模言語モデル入門](https://www.amazon.co.jp/dp/B0D1KLJPYJ/?tag=pinocchio-22)」（山田育矢 著）が参考になります。
