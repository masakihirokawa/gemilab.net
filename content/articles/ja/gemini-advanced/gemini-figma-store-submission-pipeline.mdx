アプリ開発の工程で最も時間を要する作業の1つが、App StoreやGoogle Playへの提出用アセット（スクリーンショット、プレビュー画像、アイコンのバリエーション、ローカライズ文字列）の準備です。デザイン変更のたびに手動で全プラットフォーム対応版を作成・エクスポートするのは現実的ではありません。

本記事では、**GeminiとFigma MCPを組み合わせ、デザイン変更から提出アセット生成までを自動パイプライン化する方法**を段階的に解説します。

## アーキテクチャ全体像

Gemini + Figma MCPのパイプラインは、以下の5ステップで構成されます。

```
Figmaデザイン変更（トリガー）
    ↓
Figma MCP + Gemini（設計仕様抽出）
    ↓
ローカライズ・多言語化処理
    ↓
マルチプラットフォーム化（iOS・Android）
    ↓
自動品質チェック＆エクスポート
```

Figma APIと異なり、**Figma MCPはGeminiの「思考」をリアルタイムに反映できます**。つまり、デザインファイルの構造を読み込みながら、AI的判断でスマートに最適な解像度・フォーマットを選択できるわけです。

参考記事：[Figma MCPとGeminiの連携基礎](/articles/gemini-advanced/figma-mcp-gemini)

## Figma MCPの設定とGeminiとの接続

### MCPサーバーの初期設定

まず、Figma MCPサーバーをセットアップします。

```json
{
  "mcpServers": {
    "figma": {
      "command": "node",
      "args": ["figma-mcp-server.js"],
      "env": {
        "FIGMA_ACCESS_TOKEN": "${FIGMA_TOKEN}",
        "FIGMA_FILE_ID": "your-file-id-here"
      }
    }
  }
}
```

`FIGMA_ACCESS_TOKEN`はFigmaの個人用アクセストークンで、Figma設定 → アカウント から取得できます。`FIGMA_FILE_ID`は、Figmaデザインファイルの共有URLに含まれています。

### Gemini APIとMCP統合

Figma MCPはModel Context Protocol (MCP)標準に対応しており、Gemini APIの`tool_choice`で直接呼び出せます。

```javascript
const Anthropic = require("@anthropic-ai/sdk");

const client = new Anthropic({
  apiKey: process.env.GEMINI_API_KEY,
});

async function extractStoreAssets(figmaFileId, platforms) {
  const message = await client.messages.create({
    model: "claude-3-5-sonnet-20241022",
    max_tokens: 4096,
    tools: [
      {
        name: "figma_read_file",
        description: "Read Figma file structure and component metadata",
        input_schema: {
          type: "object",
          properties: {
            file_id: { type: "string" },
            node_ids: {
              type: "array",
              items: { type: "string" },
            },
          },
          required: ["file_id"],
        },
      },
      {
        name: "figma_export",
        description: "Export artboards or components as images",
        input_schema: {
          type: "object",
          properties: {
            node_id: { type: "string" },
            format: { type: "string", enum: ["PNG", "JPG", "SVG", "PDF"] },
            scale: { type: "number" },
          },
          required: ["node_id", "format"],
        },
      },
    ],
    messages: [
      {
        role: "user",
        content: `You have access to a Figma design file. Your task:
1. Read the file structure
2. Identify all artboards labeled for store submission (e.g., "iOS Screenshot 1", "Android Screenshot")
3. For each platform in ${JSON.stringify(platforms)}, export at optimal resolution
4. Return metadata including dimensions, format, and suggested placement

Figma File ID: ${figmaFileId}

Proceed with file analysis.`,
      },
    ],
  });

  return message;
}

extractStoreAssets("your-file-id", ["iOS", "Android", "Web"]).then(
  (result) => console.log(result)
);
```

このコードは、Geminiに対して「Figmaファイルを読み込んで、ストア提出用アセットを自動判定しエクスポートせよ」という指示を出しています。Geminiは`figma_read_file`ツールでファイル構造を取得し、`figma_export`で対応フォーマットでエクスポートします。

## マルチプラットフォーム自動対応パイプライン

### プラットフォーム仕様を定義

App Store・Google Playでは、デバイスごとに異なるアセット仕様が要求されます。これを定義テーブルとして管理します。

```javascript
const storeSpecifications = {
  iOS: {
    iPhone: {
      sizes: [
        { width: 1170, height: 2532, scale: 3, name: "6.1-inch" },
        { width: 1284, height: 2778, scale: 3, name: "6.7-inch" },
      ],
      screenshotCount: 5,
      format: "PNG",
    },
    iPad: {
      sizes: [{ width: 2048, height: 2732, scale: 2, name: "12.9-inch" }],
      screenshotCount: 5,
      format: "PNG",
    },
  },
  Android: {
    phone: {
      sizes: [{ width: 1080, height: 1920, scale: 1, name: "phone" }],
      screenshotCount: 8,
      format: "JPG",
    },
    sevenInch: {
      sizes: [{ width: 1440, height: 2560, scale: 1, name: "7-inch" }],
      screenshotCount: 8,
      format: "JPG",
    },
    tenInch: {
      sizes: [{ width: 2560, height: 1600, scale: 1, name: "10-inch" }],
      screenshotCount: 8,
      format: "JPG",
    },
  },
};

async function generateAllAssets(figmaFileId) {
  // 各プラットフォーム仕様に従ってアセット生成
  for (const [platform, devices] of Object.entries(storeSpecifications)) {
    for (const [device, spec] of Object.entries(devices)) {
      // Gemini + Figma MCPを使用してアセット生成
      await generatePlatformAssets(figmaFileId, platform, device, spec);
    }
  }
}
```

このテーブルを基準にすることで、Geminiが「iPhone 6.1インチ用には1170×2532ピクセルで5枚のスクリーンショットが必要」という判定を自動的に行えます。

## ローカライズと多言語アセット生成

アセットの中には「スクリーンショット内のテキスト」も含まれます。国によって言語が異なるため、各言語版の自動生成が必須です。

Geminiはテキスト認識と翻訳を同時に処理できます。

```javascript
async function localizeAssets(
  figmaFileId,
  languages = ["ja", "en", "zh", "ko"]
) {
  const message = await client.messages.create({
    model: "claude-3-5-sonnet-20241022",
    max_tokens: 2000,
    messages: [
      {
        role: "user",
        content: `Analyze the Figma file and identify all UI text that should be localized.
For each text element:
1. Extract the current text (assumed to be in Japanese)
2. Translate to: ${languages.join(", ")}
3. Assess if visual re-layout is needed (e.g., German text is longer)

Figma File ID: ${figmaFileId}

Return a JSON structure mapping language codes to updated text and layout adjustments.`,
      },
    ],
  });

  return message;
}
```

Figma MCPの`figma_update_component_text`ツール（仮）を組み合わせることで、言語ごとに異なるテキストを持つバリエーションを自動作成できます。これにより、手動での多言語デザイン作成作業が大幅に削減されます。

参考記事：[Gemini APIで多言語スクリーンショット自動生成](/articles/gemini-api/gemini-api-screenshot-localization-automation)

## CI/CD パイプラインへの統合

### GitHubアクション例

提出アセット生成をCIパイプラインに組み込むことで、コミット時に自動実行できます。

```yaml
name: Generate Store Assets
on:
  push:
    branches:
      - main
    paths:
      - "design/figma-file-export/**"
      - ".github/workflows/store-assets.yml"

jobs:
  generate-assets:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 18

      - name: Install dependencies
        run: npm install

      - name: Generate store assets
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          FIGMA_TOKEN: ${{ secrets.FIGMA_TOKEN }}
          FIGMA_FILE_ID: ${{ secrets.FIGMA_FILE_ID }}
        run: node scripts/generate-store-assets.js

      - name: Validate asset dimensions
        run: node scripts/validate-asset-specs.js

      - name: Upload to artifact storage
        uses: actions/upload-artifact@v4
        with:
          name: store-assets
          path: output/store-assets/
          retention-days: 30

      - name: Create pull request
        uses: peter-evans/create-pull-request@v5
        with:
          commit-message: "chore: update store submission assets"
          title: "Store Assets Update"
          branch: assets-update-${{ github.run_number }}
```

このワークフローは、Figmaの設計ファイルが更新されるたびに自動実行され、生成されたアセットをArtifactストレージに保存します。

## 品質保証と自動チェック

### アセット検証スクリプト

生成されたアセットが仕様に適合しているか自動チェックすることは、リリース時の問題防止に不可欠です。

```javascript
async function validateAssets(outputDir) {
  const fs = require("fs").promises;
  const path = require("path");
  const sharp = require("sharp");

  const results = [];

  for (const platform of Object.keys(storeSpecifications)) {
    const platformDir = path.join(outputDir, platform);
    const files = await fs.readdir(platformDir);

    for (const file of files) {
      const filePath = path.join(platformDir, file);
      const metadata = await sharp(filePath).metadata();

      const spec = findMatchingSpec(platform, file);
      if (!spec) {
        results.push({
          file,
          status: "warning",
          message: `Could not find specification for ${file}`,
        });
        continue;
      }

      const isValid =
        metadata.width === spec.width && metadata.height === spec.height;

      results.push({
        file,
        status: isValid ? "pass" : "fail",
        expected: {
          width: spec.width,
          height: spec.height,
        },
        actual: {
          width: metadata.width,
          height: metadata.height,
        },
      });
    }
  }

  return results;
}
```

このバリデーションを通すことで、App Store/Google Playへの提出時に「サイズ不適合で却下」という問題を事前に防げます。

## よくある質問

**Q1: Figma MCPはFigma APIと何が違うのか？**

Figma APIは単なるデータ取得ツールですが、Figma MCPはGeminiの推論プロセスに統合されます。つまり、「このコンポーネントに最適なエクスポート解像度は？」という判断をAIが行えるので、手動設定が最小限で済みます。

**Q2: ローカライズ対応で文字が長くなった場合、レイアウトは自動調整される？**

Figmaのテキストボックスに「テキスト自動サイズ調整」設定があれば自動対応できます。複雑なレイアウト調整が必要な場合は、Geminiが判定して手動フローにエスカレートするよう設計すればよいでしょう。

**Q3: 毎日、複数回パイプラインを実行してもよいのか？**

はい。むしろ頻繁な実行をお勧めします。スクリーンショットが常に最新の設計状態を反映することで、ストア掲載情報の信頼性が高まります。ただし、Figma APIのレート制限（プランによって異なる）に注意してください。
