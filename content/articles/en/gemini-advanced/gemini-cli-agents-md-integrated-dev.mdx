## Introduction

Modern development now involves multiple AI tools operating simultaneously:

- **Gemini CLI** → Code generation, architecture consultation
- **Claude Code** → Deep reasoning, code review
- **Codex CLI** → Directory traversal, bulk generation

For all these tools to work effectively in the same project, they need a common "agreement." That agreement is **AGENTS.md**.

This guide shows how to build an **integrated development environment** centered on Gemini CLI × AGENTS.md.

---

## AGENTS.md Design Philosophy

### Adoption by 60,000+ Repositories

In 2025, the Agentic AI Foundation (AAIF) under the Linux Foundation formally adopted AGENTS.md. Today, **over 60,000 repositories** on GitHub use this format.

It's no longer a "proposal"—it's a **de facto industry standard**.

### Four Roles of AI Agents

AGENTS.md defines four core responsibilities:

| Role | Description | Implementation |
|---|---|---|
| **Constrain** | Define what NOT to do | constraints section |
| **Inform** | Share design intent & context | technology_stack, design_philosophy |
| **Verify** | Check for constraint violations | verification section |
| **Correct** | Fix issues automatically | AI auto-fix |

### Gemini CLI Integration

Gemini CLI reads AGENTS.md and automates:

```
Read AGENTS.md
    ↓
Check constraints
    ↓
Detect violations
    ↓
Propose or auto-fix solutions
```

## AGENTS.md Implementation Patterns

### Core Structure

```yaml
---
# agents.md - AI Agent Configuration

technology_stack:
  runtime: "Node.js 20 LTS"
  language: "TypeScript 5 (strict)"
  frameworks:
    - "Express.js 4"
    - "React 19"
  database: "PostgreSQL 15"
  orm: "Prisma"
  testing: "Jest"

design_philosophy: |
  Business logic lives in service layer.
  UI layer handles presentation only.
  Data access isolated via repository pattern.
  Test-driven development (TDD) required.

agents:
  - name: "Backend Developer"
    role: "API development"
    responsibilities:
      - "Business logic in src/services/"
      - "API endpoint design"
      - "Database schema management"
    constraints:
      - "TypeScript strict mode mandatory"
      - "External libraries require pre-approval"
      - "Schema changes via Prisma migration only"
    verification:
      - "npm test passes all"
      - "npm run lint with 0 errors"
      - "tsc --noEmit produces no errors"

  - name: "Frontend Developer"
    role: "UI/UX development"
    responsibilities:
      - "React component implementation"
      - "State management"
      - "Styling"
    constraints:
      - "Tailwind CSS only"
      - "No styled-components"
      - "API calls via src/services/ hooks"
    verification:
      - "npm test for React components"
      - "aXe accessibility tests pass"

build:
  development: "npm run dev"
  test: "npm test"
  lint: "npm run lint"
  type_check: "tsc --noEmit"
  production: "npm run build && npm start"

prohibitions:
  - "No hardcoded prod API keys (use AWS Secrets Manager)"
  - "No direct node_modules editing"
  - "No auto database schema migrations"
  - "No third-party API keys in Git"

testing_requirements:
  unit_coverage: "80% minimum"
  integration_coverage: "Main workflows"
  e2e_tests: "Playwright for critical paths"

design_decisions: |
  **Why Prisma?**
  - TypeScript type safety
  - Automatic migrations
  - Multi-dialect support

  **Why Express?**
  - Lightweight, gentle learning curve
  - Rich middleware ecosystem
  - Perfect for REST APIs

  **Why React?**
  - Component-oriented, reusable
  - Mature ecosystem
  - Abundant team learning resources
---
```

## Multi-AI Tool Coexistence Configuration

### File Layout

```
project/
├── AGENTS.md              ← Universal (Gemini, Codex, Cursor)
├── GEMINI.md              ← Gemini CLI optimized
├── CLAUDE.md              ← Claude Code optimized
├── .cursorrules           ← Cursor optimized
├── .copilot-instructions  ← GitHub Copilot optimized
├── README.md              ← Human documentation
└── docs/
    ├── architecture.md
    ├── api-reference.md
    └── design.md
```

### Tool-Specific Load Order

#### Gemini CLI
```
1. Look for GEMINI.md first
2. Fall back to AGENTS.md
3. Traverse directory structure for inference
```

#### Claude Code
```
1. Prioritize CLAUDE.md
2. Fall back to AGENTS.md
```

#### Codex CLI
```
1. Traverse entire directory
2. Reference AGENTS.md and README.md
```

#### Cursor
```
1. Read .cursorrules
2. Reference README.md
```

### Role Division

| File | Tool | Detail Level | Update Frequency |
|---|---|---|---|
| AGENTS.md | All | Moderate | Monthly |
| GEMINI.md | Gemini CLI | High | Monthly |
| CLAUDE.md | Claude Code | High | Monthly |
| .cursorrules | Cursor | Moderate | Quarterly |
| .copilot-instructions | GitHub Copilot | Low | Semi-annually |

## Harness Engineering Integration

### Quality Assurance Flow

```
┌───────────────────────────────────────┐
│ Step 1: Constrain (Define rules)     │
│ AGENTS.md specifies constraints      │
└───────────────────────────────────────┘
           ↓
┌───────────────────────────────────────┐
│ Step 2: Inform (Share context)        │
│ Commit messages & documentation       │
└───────────────────────────────────────┘
           ↓
┌───────────────────────────────────────┐
│ Step 3: Verify (Check constraints)   │
│ Lint, Type Check, Tests              │
└───────────────────────────────────────┘
           ↓
┌───────────────────────────────────────┐
│ Step 4: Correct (Auto-fix)            │
│ AI auto-correction                   │
└───────────────────────────────────────┘
```

### Gemini CLI Verification Commands

```bash
# Auto-review against constraints
gemini ask "Review my code against @agents.md constraints"

# Auto-fix violations
gemini ask "Fix violations of @agents.md constraints"

# Verify
npm test && npm run lint && tsc --noEmit
```

## FAQ

### Q1: Won't multiple AI tools conflict?

**A:** AGENTS.md acts as a "common language," preventing conflicts:

```
Gemini CLI reads AGENTS.md
    ↓ [understands rules]
Claude Code reads AGENTS.md
    ↓ [same rules]
Codex CLI reads AGENTS.md
    ↓ [same rules]

All tools operate consistently → unified behavior
```

### Q2: Do I really need tool-specific files (GEMINI.md, CLAUDE.md)?

**A:** AGENTS.md alone is sufficient, but tool-specific files offer benefits:

```
AGENTS.md → Universal (all tools, broad & shallow)
GEMINI.md → Gemini optimized (more detailed)
CLAUDE.md → Claude optimized (more detailed)
```

Start with AGENTS.md only. Add tool-specific files as you mature.

### Q3: Won't AGENTS.md become too large?

**A:** Up to ~1500 lines is fine. For larger projects, split:

```
AGENTS.md (root)
    ├─ /docs/agents-backend.md
    ├─ /docs/agents-frontend.md
    └─ /docs/agents-devops.md
```

Each team manages its domain configuration, reducing cognitive load.
