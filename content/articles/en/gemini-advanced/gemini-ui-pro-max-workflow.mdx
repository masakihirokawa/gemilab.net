## Introduction

UI design traditionally involves multiple stages: requirements → wireframes → low-fidelity → high-fidelity. But iteration loops eat time, and implementation handoff often brings surprises.

**UI Pro Max** accelerates this entire workflow using Gemini's advanced design generation. This article walks through real-world application, showing how to compress weeks of design work into days.

## Understanding UI Pro Max

### Core Capabilities

UI Pro Max combines three essential functions:

1. **Wireframe auto-generation** — Transform requirements into layout sketches
2. **Design variations** — Multiple UI approaches from AI
3. **Design token extraction** — Automatic color, font, spacing definitions in machine-readable format

Together, these enable seamless designer-developer collaboration.

## Practical Workflow: Five-Phase Design Process

### Phase 1: Requirements and Information Architecture

Feed Gemini **specific, detailed requirements**. Vague input produces mediocre output.

**Detailed input example:**
```
"We're designing an e-commerce product detail page.

[Business Requirements]
- Target: 20–40 year-old women (income $30k+/month)
- Goal: Increase cart conversion from 15% to 25%
- References: Amazon, Shopify, luxury fashion sites

[Elements to Include]
High priority:
- Product images (multiple, with zoom)
- Product name & description (under 500 chars)
- Price & discounts (show savings percentage)
- Ratings & review count (5-star + number)
- Add-to-cart button

Medium priority:
- Size/color dropdowns
- Stock status
- Shipping info & estimated delivery

Low priority:
- Review carousel
- Related products carousel

[Device Support]
- Mobile: 375px–420px
- Tablet: 768px–1024px
- Desktop: 1280px+

[Design Brief]
- Vibe: Modern, trustworthy, bright
- Primary color: Blue #2563EB
- Audience: Luxury product positioning

[User Flow]
Find product → Review details → Make decision → Add to cart
Expected time: 2–3 minutes"
```

Gemini returns:
- **Information Architecture (IA)** — Element priority, grouping, order
- **Layout options** — Multiple approaches (vertical scroll, tabbed, etc.)
- **User flows** — Navigation patterns and touchpoints

**Follow-up questions:**
- "Image or product info first?"
- "Collapse reviews or show inline?"
- "What information can we drop on mobile?"

### Phase 2: Wireframe Generation and Visualization

Ask Gemini: "Express this wireframe as ASCII art, both mobile (375px) and tablet (768px). For each section, note its height ratio, internal layout, and responsive rules."

**Gemini's output example (mobile):**
```
┌─────────────────────────────┐
│   Header (12%)              │ ← Navigation, search
├─────────────────────────────┤
│                             │
│   Product image (40%)       │ ← Zoomable, thumbnails
│                             │
├─────────────────────────────┤
│ Product name (8%)           │
├─────────────────────────────┤
│ Price & discount (6%)       │
├─────────────────────────────┤
│ Rating (6%)                 │
├─────────────────────────────┤
│ Size/color options (12%)    │
├─────────────────────────────┤
│ "Add to cart" button (8%)   │
├─────────────────────────────┤
│ Description (scrollable)    │
└─────────────────────────────┘
```

Then UI Pro Max generates detailed specs:
- Pixel heights for each section
- Why this order matters
- Responsive breakpoint rules
- Grid system recommendation (8px, 12px, or 16px base)

**Follow-up questions:**
- "How should tablet differ?"
- "Minimize scroll distance?"
- "Ideal whitespace ratio?"

### Phase 3: Low-Fidelity Design (Grayscale)

Move from sketch to simplified grayscale UI—focus on **layout and information hierarchy only**. No color or decoration yet.

**Prompt to Gemini:**
```
"Convert the wireframe to low-fidelity UI.

Specifications:
- Grayscale only (black #000, gray #666, white #FFF)
- Three information levels (heading: bold, body: normal, secondary: light)
- ~20–30 components (buttons, inputs, cards, badges)

Provide:
1. Visual hierarchy per section
2. Whitespace placement
3. Grid system (what base unit?)
4. Text size ratios
5. Button size (min 44px × 44px)"
```

**Output guidelines:**
```
Grid: 8px base

Typography:
- H1: 32px, Weight 700 (product name)
- H2: 24px, Weight 600 (section headers)
- H3: 18px, Weight 500 (subheadings)

Body:
- Standard: 16px, Weight 400, Line height 1.5
- Small: 14px, Weight 400

Spacing:
- Between sections: 32px (8px × 4)
- Element padding: 16px (8px × 2)
- List items: 12px (8px × 1.5)
```

**Figma checklist:**
- [ ] Set up 8px grid
- [ ] Limit colors to 3 grayscale values
- [ ] Establish text size scale (8, 12, 16, 24, 32px)
- [ ] Ensure touch targets are ≥44px
- [ ] Test across devices (375px, 768px, 1280px)

### Phase 4: High-Fidelity and Design Tokens

Introduce design tokens—variables for color, size, and spacing that can be shared across tools.

**Design token example (JSON):**
```json
{
  "colors": {
    "primary": {
      "50": "#EFF6FF",
      "500": "#2563EB",
      "700": "#1D4ED8"
    },
    "neutral": {
      "50": "#F9FAFB",
      "900": "#111827"
    },
    "status": {
      "success": "#10B981",
      "error": "#EF4444"
    }
  },
  "typography": {
    "heading-1": {
      "fontSize": "36px",
      "fontWeight": "700",
      "lineHeight": "1.3"
    },
    "body": {
      "fontSize": "16px",
      "fontWeight": "400",
      "lineHeight": "1.5"
    }
  },
  "spacing": {
    "xs": "4px",
    "sm": "8px",
    "md": "16px",
    "lg": "24px"
  }
}
```

**Prompt to Gemini:**
```
"Create a complete high-fidelity design for this product page.

Specs:
- Brand colors: Blue #2563EB / Green #10B981
- Font: Inter or Roboto (multilingual)
- Target: 20–40 women (sophisticated, trust-focused)
- References: Apple, Notion, Stripe

Deliver:
1. Full design token definition (JSON)
2. High-fidelity screen descriptions
3. Color usage rules
4. Typography details (letter-spacing, etc.)
5. Interaction specs (hover, tap states)"
```

**Gemini output includes:**
- Complete token definition
- Light/dark mode variants
- Component-specific rules
- Animation & transition specs
- Accessibility considerations

### Phase 5: Reflect in Figma and Prepare Handoff

Systematically bring AI-generated designs into Figma.

**Step 1: Register tokens in Figma Variables**
```
1. Create new Figma file
2. Right panel → Design → Variables
3. Create "Colors" collection
4. Add tokens from Gemini (Primary, Secondary, Neutral, Status)
5. Create "Typography" and "Spacing" collections the same way
```

**Step 2: Build component library**

```
Core components (high priority):
├─ Button (Variants: Primary, Secondary, Ghost × Small, Medium, Large × Default, Hover, Disabled)
├─ Input (Types: Text, Email, Number × States: Default, Focus, Error)
├─ Card
│  ├─ ProductCard (image, price, rating)
│  ├─ ReviewCard (avatar, text, stars)
├─ Badge
└─ Rating stars

Composite components (medium priority):
├─ FormGroup (label + input + helper)
├─ PriceDisplay
├─ ImageCarousel
└─ ReviewSection
```

**Step 3: Assemble full page**

```
Product detail structure:
1. Header → Images (main + thumbnails, zoom enabled)
2. Info → Name, rating, price
3. Actions → Size/color selectors, buttons
4. Description → Details, specs
5. Reviews → Review list, pagination
6. Related → Product carousel
```

**Step 4: Responsive implementation**

Create frames for each breakpoint:
```
├─ Mobile (375px)
├─ Tablet (768px)
└─ Desktop (1280px)

Check:
- Text reflow at each size
- Image aspect ratios
- Button accessibility
- Scroll distance
```

## Building and Scaling Design Systems

UI Pro Max's real power is **design system construction**—reusable across multiple products.

### Component Extraction Strategy

Send multiple screens and ask: "Extract reusable components across all three screens. Organize by atomic design principles."

**Gemini's response:**
```
[Base components]
- Button (Primary, Secondary, Ghost) × (Small, Medium, Large)
- Input (Text, Email, Select) × (Default, Focus, Error)
- Badge (Success, Warning, Error)
- Avatar (Circle, Square) × sizes

[Composite components]
- ProductCard: Image + Title + Price + Rating
- ReviewCard: Avatar + Name + Rating + Text
- FormGroup: Label + Input + HelperText
- PriceDisplay: Original × Sale price with discount badge
```

### Token Naming Convention

Ask Gemini to explain your naming rules. This helps new team members understand *why* the system is structured this way.

**Convention example:**
```
Colors:
[Scope] / [Hue] / [Shade]
primary / blue / 500
primary / blue / hover
primary / blue / active

Spacing:
xs (4px), sm (8px), md (16px), lg (24px), xl (32px)

Typography:
heading / size-1 → 36px
heading / weight → 700
body / size → 16px
```


## Implementation Team Handoff

### Dev Mode Integration

Figma's Dev Mode auto-extracts measurements (sizes, spacing, fonts).

**Info extraction table:**

| Type | Content | Format |
|------|---------|--------|
| Dimensions | Width, height, margins, padding | px values |
| Color | Background, text, borders | HEX / RGB |
| Typography | Font, size, weight, line-height | CSS properties |
| Effects | Shadows, blur, opacity | CSS filter |

**How to use Dev Mode:**
```
1. Open Figma file
2. Click "Dev" tab (top right)
3. Select component
4. View extracted specs in right panel
5. Click "Copy" to copy as CSS/JSON
```

### CSS-in-JS Transformation

**From Gemini tokens (JSON) → styled-components:**

```javascript
// tokens.js
export const tokens = {
  colors: {
    primary: { 500: "#2563EB", 600: "#1D4ED8" },
    neutral: { 50: "#F9FAFB", 900: "#111827" }
  },
  spacing: { xs: "4px", sm: "8px", md: "16px" }
};

// Button.jsx
import styled from "styled-components";
import { tokens } from "./tokens";

export const StyledButton = styled.button`
  background-color: ${tokens.colors.primary[500]};
  padding: ${tokens.spacing.sm} ${tokens.spacing.md};

  &:hover {
    background-color: ${tokens.colors.primary[600]};
  }
`;
```

**Or Tailwind config:**
```javascript
// tailwind.config.js
module.exports = {
  theme: {
    extend: {
      colors: {
        primary: { 500: "#2563EB", 600: "#1D4ED8" }
      },
      spacing: { xs: "4px", sm: "8px", md: "16px" }
    }
  }
};
```

## Best Practices

### 1. Iterative Feedback

Don't build everything at once. After each phase, ask Gemini specific questions:

**Phase 2:** "Does visual flow prioritize key information?"
**Phase 4:** "Can we make this more sophisticated?"
**Pre-implementation:** "Will this component structure work in React?"

### 2. Design System Early

Establish tokens and component rules from day one. This pays off across future projects.

**Order:**
1. Define tokens (color, font, spacing)
2. Build core components
3. Assemble composite components
4. Create page templates
5. Document and iterate

### 3. Staged Handoff to Developers

Share information progressively:

| When | What | Format |
|------|------|--------|
| Phase 2 | Page structure, data flow | Diagrams, brief docs |
| Phase 4 | Tokens, specs | JSON, design doc |
| Phase 5 | CSS examples, animations | Code samples, Storybook |

### 4. Version Control

```
v1.0 – Initial release (core components, tokens)
v1.1 – Minor tweaks (color adjustments, new icons)
v2.0 – Major overhaul (removed components, redesigned)

Rule: Small changes → v1.x; Major changes → v2.0
```

This prevents breaking existing projects.
