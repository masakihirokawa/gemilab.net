## Introduction

Code review is one of the most important practices in software development, yet it's also one of the biggest bottlenecks. Reviewers are busy, context-switching is expensive, and PRs can sit in limbo for days. What if you could get instant, automated feedback the moment a pull request is opened?

With Gemini API's exceptional code comprehension capabilities and massive context window, you can build a bot that automatically reviews every PR in your repository. In this guide, we'll walk through creating a fully functional AI code review bot using Gemini 2.5 Pro and GitHub Actions — from zero to production-ready in about 30 minutes.

**What you'll learn:**

- How to analyze code diffs with Gemini API
- Setting up a GitHub Actions workflow that triggers on PR events
- Building a Python script that posts review comments directly to pull requests

## Architecture Overview

The AI code review bot follows a straightforward pipeline:

1. **A PR is opened or updated** → GitHub Actions workflow triggers
2. **Fetch the diff** → Pull changed files and patches via the GitHub API
3. **Analyze with Gemini** → Send the diff to Gemini API with a review-focused prompt
4. **Post comments** → Format the results and post them as a PR comment

```
┌──────────┐     ┌────────────────┐     ┌────────────┐     ┌────────────┐
│  PR Open  │────▶│ GitHub Actions  │────▶│ Gemini API  │────▶│ PR Comment  │
│  Update   │     │   Workflow      │     │  Analysis   │     │   Posted    │
└──────────┘     └────────────────┘     └────────────┘     └────────────┘
```

## Step 1: Building the Core Review Script

Let's start with the Python script that sends code diffs to Gemini and generates review feedback.

```python
# review_bot.py
import os
import json
from google import genai
from github import Github

# Load credentials from environment variables
GEMINI_API_KEY = os.environ["GEMINI_API_KEY"]
GITHUB_TOKEN = os.environ["GITHUB_TOKEN"]
REPO_NAME = os.environ["GITHUB_REPOSITORY"]
PR_NUMBER = int(os.environ["PR_NUMBER"])

# Initialize clients
client = genai.Client(api_key=GEMINI_API_KEY)

gh = Github(GITHUB_TOKEN)
repo = gh.get_repo(REPO_NAME)
pr = repo.get_pull(PR_NUMBER)


def get_pr_diff() -> str:
    """Fetch changed files and their diffs from the PR."""
    files = pr.get_files()
    diff_text = ""
    for f in files:
        # Skip binary files
        if f.patch is None:
            continue
        diff_text += f"### {f.filename}\n"
        diff_text += f"Status: {f.status}\n"
        diff_text += f"```diff\n{f.patch}\n```\n\n"
    return diff_text


def review_with_gemini(diff: str) -> str:
    """Send the diff to Gemini API for code review."""
    prompt = f"""You are an experienced senior software engineer.
Review the following pull request diff thoroughly.

## Review Criteria
1. **Bugs & Logic Errors**: Potential bugs, edge cases, off-by-one errors
2. **Security**: SQL injection, XSS, authentication gaps, secret exposure
3. **Performance**: N+1 queries, unnecessary loops, memory leaks
4. **Readability**: Variable naming, function decomposition, comments
5. **Best Practices**: Language-specific idioms, design patterns

## Output Format
Respond in this JSON format:
{{
  "summary": "Overall assessment (1-2 sentences)",
  "issues": [
    {{
      "severity": "critical|warning|suggestion",
      "file": "filename",
      "description": "Description of the issue"
    }}
  ],
  "positive": ["1-2 things done well"]
}}

If there are no issues, return an empty array for issues.

## PR Diff
{diff}
"""

    response = client.models.generate_content(
        model="gemini-2.5-pro",
        contents=prompt,
        config={
            "response_mime_type": "application/json",
            "temperature": 0.2,  # Low temperature for consistent reviews
        },
    )
    return response.text


def format_review_comment(review_json: str) -> str:
    """Format the review result as a Markdown comment."""
    review = json.loads(review_json)

    comment = "## 🤖 AI Code Review by Gemini\n\n"
    comment += f"**Summary**: {review['summary']}\n\n"

    # Positive feedback
    if review.get("positive"):
        comment += "### ✅ What's Good\n"
        for point in review["positive"]:
            comment += f"- {point}\n"
        comment += "\n"

    # Issues found
    if review.get("issues"):
        severity_icons = {
            "critical": "🔴",
            "warning": "🟡",
            "suggestion": "🔵",
        }
        comment += "### 📝 Issues Found\n\n"
        for issue in review["issues"]:
            icon = severity_icons.get(issue["severity"], "⚪")
            comment += f"{icon} **[{issue['severity'].upper()}]** "
            comment += f"`{issue['file']}`\n"
            comment += f"> {issue['description']}\n\n"
    else:
        comment += "### 📝 Issues Found\nNone — the code looks great! 🎉\n\n"

    comment += "---\n"
    comment += "*Powered by Gemini 2.5 Pro — [Gemini Lab](https://gemilab.net)*"
    return comment


def main():
    print(f"Reviewing PR #{PR_NUMBER} in {REPO_NAME}...")

    # Fetch the diff
    diff = get_pr_diff()
    if not diff.strip():
        print("No reviewable changes found.")
        return

    # Truncate if the diff is extremely large
    max_chars = 100_000  # Gemini 2.5 Pro supports 1M tokens
    if len(diff) > max_chars:
        diff = diff[:max_chars] + "\n\n(... truncated due to size)"

    # Run the review
    review_json = review_with_gemini(diff)
    print(f"Review result: {review_json[:200]}...")

    # Format and post the comment
    comment = format_review_comment(review_json)
    pr.create_issue_comment(comment)
    print("Review comment posted successfully!")


if __name__ == "__main__":
    main()
```

### Key Design Decisions

- **Structured Output**: Setting `response_mime_type: "application/json"` ensures Gemini returns well-formed JSON every time
- **Low Temperature**: Code review demands accuracy over creativity, so we use `temperature: 0.2`
- **Size Guard**: A character limit prevents failures on unusually large PRs

## Step 2: Setting Up the GitHub Actions Workflow

Create `.github/workflows/ai-review.yml` in your repository:

```yaml
# .github/workflows/ai-review.yml
name: AI Code Review

on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read
  pull-requests: write

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: pip install google-genai PyGithub

      - name: Run AI Review
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          GITHUB_REPOSITORY: ${{ github.repository }}
          PR_NUMBER: ${{ github.event.pull_request.number }}
        run: python review_bot.py
```

### Configuring Secrets

Navigate to your repository's **Settings → Secrets and variables → Actions** and add:

| Secret Name | Value |
|---|---|
| `GEMINI_API_KEY` | Your API key from Google AI Studio |

`GITHUB_TOKEN` is automatically provided by GitHub Actions — no manual setup needed.

## Step 3: Improving Review Quality with Prompt Tuning

Once the basic bot is running, you can fine-tune the prompts to improve review quality.

### Adding Project-Specific Rules

```python
# Load project-specific review rules from .ai-review-rules.md
def load_custom_rules() -> str:
    rules_path = ".ai-review-rules.md"
    if os.path.exists(rules_path):
        with open(rules_path, "r") as f:
            return f.read()
    return ""

# Append to the prompt
custom_rules = load_custom_rules()
if custom_rules:
    prompt += f"\n## Project-Specific Rules\n{custom_rules}\n"
```

Example `.ai-review-rules.md`:

```markdown
# Review Rules
- TypeScript: Do not use `any` type — use `unknown` or proper generics
- All API endpoints must include input validation
- No console.log in production code — use a structured logger
- Test file changes must include corresponding test cases
```

### File-Type-Specific Review Criteria

```python
def get_file_context(filename: str) -> str:
    """Return additional review context based on file type."""
    if filename.endswith((".ts", ".tsx")):
        return "TypeScript: Focus on type safety, null checks, proper async/await usage"
    elif filename.endswith(".py"):
        return "Python: Focus on type hints, exception handling, PEP 8 compliance"
    elif filename.endswith((".yml", ".yaml")):
        return "YAML config: Check indentation, env var leaks, security settings"
    elif filename.endswith("Dockerfile"):
        return "Docker: Check for multi-stage builds, non-root user, layer cache optimization"
    return ""
```

## Step 4: Rate Limiting and Error Handling

For production stability, implement proper retry logic and error handling.

```python
import time
from google.api_core import exceptions as google_exceptions


def review_with_retry(diff: str, max_retries: int = 3) -> str:
    """Call Gemini API with exponential backoff retry."""
    for attempt in range(max_retries):
        try:
            return review_with_gemini(diff)
        except google_exceptions.ResourceExhausted:
            # Rate limited
            wait_time = 2 ** attempt * 10  # 10, 20, 40 seconds
            print(f"Rate limited. Retrying in {wait_time}s... ({attempt + 1}/{max_retries})")
            time.sleep(wait_time)
        except google_exceptions.InvalidArgument as e:
            # Prompt too long
            print(f"Input too large: {e}")
            if len(diff) > 10000:
                diff = diff[:len(diff) // 2] + "\n\n(... truncated)"
                continue
            raise
        except Exception as e:
            print(f"Unexpected error: {e}")
            if attempt < max_retries - 1:
                time.sleep(5)
                continue
            raise
    raise RuntimeError("Max retries exceeded for Gemini API call")
```

## Frequently Asked Questions

### Q1: How long does a review take?

With Gemini 2.5 Pro, a typical PR with a few hundred lines of changes takes about 10–30 seconds to review. Including GitHub Actions startup time, you'll see the comment within 1–2 minutes. Even large PRs with thousands of lines work fine thanks to Gemini's 1 million token context window.

### Q2: How much does it cost?

Gemini 2.5 Pro pricing is approximately $1.25 per million input tokens and $10 per million output tokens (as of March 2026). A typical review uses about 2,000–10,000 input tokens and 500–1,000 output tokens, costing roughly $0.01–$0.05 per review. Even with 100 reviews per month, you're looking at just a few dollars.

### Q3: Is it safe to send proprietary code to Gemini API?

Under Google's API terms of service, data sent via the API is not used for model training. For highly sensitive projects, consider using Vertex AI with VPC Service Controls or private endpoints. You can learn more in our guide on [using Gemini in production with Vertex AI](/en/articles/gemini-dev/vertex-ai-gemini-production).

### Q4: Does this replace human code review?

AI review **complements** human review — it doesn't replace it. Architectural decisions, business logic correctness, and system design still require human judgment. By letting AI handle the routine checks (coding standards, security patterns, common bugs), your human reviewers can focus on higher-level concerns that truly need their expertise.
