## What Is Plan Mode?

Plan Mode is a structured workflow feature that makes Gemini CLI present a **step-by-step execution plan** before taking any action. The developer reviews and approves the plan, then Gemini proceeds.

As of the March 2026 update, Plan Mode is **enabled by default** — no configuration required.

### Why This Matters

Without Plan Mode, AI agents operate reactively: they think and act simultaneously, which can lead to hard-to-predict behavior. Plan Mode solves this by creating a deliberate pause between intent and execution.

```
User: "Add TypeScript type annotations to all files in src/"

[Gemini CLI – Plan Mode]
📋 Proposed execution plan:
  1. Recursively scan src/ for .ts and .tsx files
  2. Parse AST to identify missing type annotations
  3. Create backups in .backup/ before modifying files
  4. Add type annotations (no `any` types)
  5. Display diff and ask for confirmation
  6. Apply changes after approval

Approve this plan? [y/N]
```

This transparency gives developers full control over what the AI does — no surprises, no accidental deletions.

## Setting Up Plan Mode

### Installation and Version Check

Make sure you're on Gemini CLI v0.34.0 or later for full Plan Mode and sandbox support.

```bash
# Install Gemini CLI
npm install -g @google/gemini-cli

# Confirm version (v0.34.0+ required)
gemini --version
# Expected output: 0.34.0-preview.1

# Confirm Plan Mode is active
gemini config show | grep plan_mode
# Expected output: plan_mode: true
```

### Configuring Plan Mode

You can customize Plan Mode behavior in `~/.gemini/config.yaml`:

```yaml
# ~/.gemini/config.yaml

plan_mode:
  enabled: true                    # Default: true
  auto_approve_low_risk: false     # Auto-approve low-risk operations
  preserve_on_compression: true   # Keep approved plans through chat compression
  max_plan_steps: 20               # Cap on plan steps

sandbox:
  type: "gvisor"                   # "none" | "lxc" | "gvisor"
  allow_network: false             # Block network access inside sandbox
  workspace_mount: "/workspace"    # Mount point for your project
```

### Plan Preservation Through Chat Compression

Long sessions trigger context compression to save tokens. In older versions, this would reset any approved plan mid-task. **This is now fixed.** Approved plans persist through compression.

```bash
# Run a verbose session to see compression behavior
gemini chat --verbose

# When compression occurs, you'll see:
# [INFO] Context compression triggered (48k tokens → 12k tokens)
# [INFO] Approved plan preserved: "TypeScript annotation task (Step 3/6)"
# [PLAN] Resuming from step 3: Running AST analysis on auth/middleware.ts
```

## Understanding Sandbox Options

The sandbox isolates code and shell commands that Gemini CLI executes, preventing unintended access to your host system. The March 2026 update added two new options: **LXC (experimental)** and **native gVisor (runsc)**.

### Sandbox Comparison

| Feature | none | LXC | gVisor |
|---------|------|-----|--------|
| Filesystem isolation | ❌ | ✅ | ✅ |
| Network isolation | ❌ | ✅ | ✅ |
| Kernel protection | ❌ | Partial | ✅ |
| Startup time | Instant | Medium | Fast |
| Overhead | None | Low | Very low |
| Best for | Dev/Testing | CI/CD | Production / High security |

### LXC Sandbox Configuration

LXC provides OS-level virtualization with minimal overhead.

```bash
# Enable LXC sandbox
gemini config set sandbox.type lxc

# Verify sandbox status
gemini sandbox status
# Output:
# Sandbox type: LXC (experimental)
# Container status: ready
# Network isolation: enabled
# Workspace: /workspace (read-write)

# Run a task inside LXC
gemini run --sandbox lxc "Update all dependencies in package.json to their latest versions"
```

### gVisor Sandbox Configuration

gVisor is a user-space kernel developed by Google that intercepts container syscalls, providing strong isolation without full virtualization overhead.

```bash
# Check if gVisor is installed
which runsc || echo "gVisor (runsc) not found"

# Install gVisor on Debian/Ubuntu
sudo apt-get install -y runsc

# Enable gVisor in Gemini CLI
gemini config set sandbox.type gvisor

# Verify the setup
gemini sandbox test
# Output:
# ✅ gVisor (runsc) sandbox: OK
# Security level: kernel-protected
# Syscall interception: active
```

### Automating with Python

For CI/CD pipelines, you can drive Plan Mode programmatically:

```python
import subprocess
import json

def run_gemini_plan(task: str, sandbox: str = "gvisor") -> dict:
    """
    Run a Gemini CLI task in Plan Mode with auto-approval.

    Args:
        task: Natural language description of the task
        sandbox: Sandbox type ("none", "lxc", or "gvisor")

    Returns:
        Dict with status, output, and steps completed

    Expected output (success):
        {"status": "success", "steps_completed": 5, "output": {...}}
    """
    cmd = [
        "gemini", "run",
        "--sandbox", sandbox,
        "--auto-approve",           # Skip manual approval (for CI)
        "--output-format", "json",
        task
    ]

    result = subprocess.run(
        cmd,
        capture_output=True,
        text=True,
        timeout=300  # 5-minute timeout
    )

    if result.returncode == 0:
        return {
            "status": "success",
            "output": json.loads(result.stdout),
            "steps_completed": result.stdout.count("✅")
        }
    else:
        return {
            "status": "error",
            "error": result.stderr,
            "returncode": result.returncode
        }

# Example usage
result = run_gemini_plan(
    "Add JSDoc comments to all functions in src/utils/",
    sandbox="gvisor"
)

print(f"Status: {result['status']}")
# Output: Status: success
```

## Common Errors and Fixes

### Error 1: gVisor not found

```
Error: gVisor sandbox requested but 'runsc' not found in PATH
```

**Fix:**

```bash
# Ubuntu/Debian
sudo apt-get install -y runsc

# macOS: gVisor is Linux-only; switch to LXC instead
gemini config set sandbox.type lxc
```

### Error 2: Plan resets after chat compression

This was a bug in v0.33 and earlier. Update to the latest version:

```bash
npm install -g @google/gemini-cli@latest

# Explicitly enable plan preservation
gemini config set plan_mode.preserve_on_compression true
```

### Error 3: LXC container fails to start

```
Error: LXC container failed to start (permission denied)
```

**Fix:**

```bash
# Add your user to the lxd group
sudo usermod -aG lxd $USER
newgrp lxd

# Initialize LXD
sudo lxd init --auto

# Test the sandbox
gemini sandbox test --type lxc
```

## Security Best Practices

### 1. Least-Privilege Networking

Only allow the specific hosts your task needs:

```yaml
# ~/.gemini/config.yaml
sandbox:
  type: "gvisor"
  allow_network: false
  allowed_hosts:
    - "api.github.com"
    - "registry.npmjs.org"
  read_only_paths:
    - "/etc/ssl/certs"
```

### 2. CI/CD Integration

```yaml
# .github/workflows/ai-refactor.yml
name: AI-assisted Refactoring

on:
  workflow_dispatch:
    inputs:
      task:
        description: 'Task for Gemini CLI'
        required: true

jobs:
  ai-refactor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Gemini CLI
        run: npm install -g @google/gemini-cli

      - name: Install gVisor
        run: sudo apt-get install -y runsc

      - name: Run AI task in sandbox
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
        run: |
          gemini run \
            --sandbox gvisor \
            --auto-approve \
            "${{ github.event.inputs.task }}"
```

## FAQ

### Q1. Can I disable Plan Mode?

Yes. Though we recommend keeping it on for production workflows:

```bash
# Disable globally
gemini config set plan_mode.enabled false

# Disable for a single run
gemini run --no-plan-mode "Quick task"
```

### Q2. Does gVisor work on macOS?

Not natively — gVisor is Linux-only as of March 2026. On macOS, use LXC sandbox or run via Docker Desktop with gVisor support. Check the [Gemini CLI docs](https://geminicli.com/docs) for the latest platform support.

### Q3. Can I auto-approve in CI/CD?

Yes, use `--auto-approve`. For more granular control, set `auto_approve_low_risk: true` in your config — this automatically approves file reads and comment additions while still prompting for file deletions or external API calls.

### Q4. Can sandboxes access GPUs?

gVisor has experimental CUDA passthrough support, but it requires manual configuration. LXC supports GPU passthrough natively and is the better choice for AI inference tasks running locally.

### Q5. Which model does Plan Mode use?

Plan Mode works with both Gemini 3.1 Pro and Gemini 3.1 Flash-Lite. Gemini 3.1 Pro generates more detailed multi-step plans and handles complex migrations better. For simple, high-volume tasks, Flash-Lite offers better cost efficiency without sacrificing plan quality.
