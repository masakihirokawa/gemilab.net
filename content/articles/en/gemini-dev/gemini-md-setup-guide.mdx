## Introduction

Managing projects with Gemini CLI requires a **GEMINI.md** configuration file. But writing it from scratch is daunting.

Here's a better approach—a **2-step workflow** that saves time:

```
Step 1: Write draft.md (rough notes)
         ↓
Step 2: Have Gemini CLI auto-generate GEMINI.md
```

This guide walks you through this efficient process.

---

## What is GEMINI.md?

### Purpose

GEMINI.md is a **structured metadata file that helps Gemini CLI automatically understand your project's configuration**. It provides:

- **Technology stack** (languages, frameworks, libraries)
- **Development philosophy** (architecture, patterns)
- **Testing strategy** (TDD/BDD approaches, coverage goals)
- **Design rationale** (the "why" behind decisions)

### How It Differs from AGENTS.md

| Aspect | AGENTS.md | GEMINI.md |
|---|---|---|
| **Supported Tools** | Multiple (universal format) | Gemini CLI only |
| **Format** | YAML | Markdown + Structured sections |
| **Detail Level** | Moderate | More detailed (Gemini-optimized) |
| **Authoring** | Hand-written | AI-generated (recommended) |

## Writing draft.md (Step 1)

### Overview

draft.md is a **quick note file** where you bullet-point project information. Perfection isn't required. Just capture what's in your head.

### Basic Template

```markdown
# MyProject - draft.md

## Project Overview
- Purpose: [What does this app/tool do?]
- Target Users: [Who will use it?]
- Core Features: [3 most important features]

## Technology Stack
- Language: Node.js 20
- Framework: Express.js
- Database: PostgreSQL
- Frontend: React 19
- Testing: Vitest

## Architecture
- [Rough layer structure]
- [Communication patterns]
- [Data flow overview]

## Development Philosophy
- TDD adopted
- Microservices-oriented
- API-first approach

## Testing Strategy
- Unit tests: 80%+ coverage
- Integration tests: main workflows
- E2E tests: Playwright

## Troubleshooting
- Common error 1: [Symptom and fix]
- Common error 2: [Symptom and fix]

## Prohibitions
- [Don't do this 1]
- [Don't do this 2]
```

### Real Example

```markdown
# PaymentAPI - draft.md

## Project Overview
- Purpose: Stripe integration payment API for SaaS
- Target Users: Small online retailers
- Core Features:
  - Subscription management
  - Multi-currency support
  - Webhook auto-processing

## Technology Stack
- Node.js 20
- Express.js + TypeScript
- PostgreSQL (Prisma ORM)
- Redis (caching)
- Docker Compose (prod)

## Architecture
- API → Service → Repository layers
- Event-driven (Stripe Webhooks)
- Caching: Redis for price tables

## Development Philosophy
- TypeScript strict mode
- TDD-first (Jest)
- Single responsibility (files < 100 lines)

## Testing
- Unit: 90% coverage
- Integration: Stripe testmode
- Production: smoke tests

## Troubleshooting
- Stripe Webhook timeout → Retry logic + DLQ
- Multi-currency rounding drift → Cache refresh frequency

## Prohibitions
- Never hardcode prod API keys (use AWS Secrets Manager)
- Never mix Stripe Test and Live APIs
```

## Using @draft.md with Gemini (Step 2)

### Overview

Gemini CLI supports the `@draft.md` syntax, allowing you to pass files to AI. Use this to auto-generate GEMINI.md:

### Command Syntax

```bash
gemini ask "Read @draft.md and generate a complete GEMINI.md for this project. \
Include: technology stack, architecture decisions, coding standards, \
testing strategy, troubleshooting guide, and design rationale."
```

### Detailed Generation Prompt

```bash
gemini ask "
You are a project documentation expert.

Read @draft.md, which contains brief project notes.

Generate a comprehensive GEMINI.md with these sections:

1. Project Overview
   - Clear purpose
   - Target users
   - Core features (3-5)

2. Technology Stack
   - Languages & versions
   - Frameworks & libraries
   - Database choice & rationale

3. Architecture
   - Text-based diagram
   - Layer structure
   - Data flows
   - Protocols

4. Development Standards
   - Coding conventions
   - File structure
   - Naming patterns
   - TypeScript/Linting rules

5. Testing Strategy
   - Test types
   - Coverage targets
   - Frameworks
   - CI/CD integration

6. Design Decisions
   - Why this architecture?
   - Trade-offs
   - Performance considerations
   - Security considerations

7. Troubleshooting
   - Common errors & solutions
   - Bottlenecks
   - Debug tips

8. Prohibitions
   - What devs MUST NOT do
   - Anti-patterns

Output as a markdown file for Gemini CLI.
"
```

## Combining with AGENTS.md

Gemini CLI also supports **AGENTS.md**. Here's the recommended structure:

### Recommended Layout

```
project/
├── draft.md          ← Notes (Step 1)
├── GEMINI.md         ← Detailed config (Step 2, AI-generated)
├── AGENTS.md         ← Universal format (hand-written)
├── README.md         ← Human-friendly docs
├── docs/
│   ├── architecture.md    ← Architecture details
│   ├── api-reference.md   ← API specs
│   └── design.md          ← Design rationale
└── src/
```

### Role Division

| File | Role | Creation |
|---|---|---|
| **draft.md** | Initial notes | Hand-written (bullet points) |
| **GEMINI.md** | Gemini CLI config | AI-generated from @draft.md |
| **AGENTS.md** | Multi-tool support | Hand-written or AI-generated |
| **README.md** | Public documentation | Hand-written |
| **docs/design.md** | Design "why" | Hand-written |

## Operational Best Practices

### When to Update draft.md

draft.md is a **living document**. Update it when:

```
You add a new library → Update draft.md
You discover a new issue → Add to draft.md
Architecture changes → Update draft.md
```

### Periodic GEMINI.md Regeneration

Regenerate every 3 months or after major changes:

```bash
# Quarterly refresh
gemini ask "Read @draft.md and regenerate GEMINI.md..."
```

### Benefits of Separation

| Benefit | Explanation |
|---|---|
| **Maintainability** | draft.md is short and easy to edit |
| **Reviewability** | Team can review GEMINI.md for accuracy |
| **Version Control** | Changes are clear in git diffs |
| **Automation** | Auto-regenerate for new team members |

## FAQ

### Q1: How detailed should draft.md be?

**A:** **Bullet points are enough.** Full sentences aren't needed.

```
✓ Good draft.md
- TypeScript enabled
- Express.js + Prisma
- PostgreSQL
- Jest with 80% target
- Event-driven architecture

✗ Bad draft.md (too long)
This project implements modern web application development best practices...
[long narrative]
```

### Q2: What if the auto-generated GEMINI.md has errors?

**A:** Two options:

**Option 1: Fix and commit**
```bash
# Manually correct errors
code GEMINI.md
git add GEMINI.md && git commit -m "Fix: GEMINI.md corrections"
```

**Option 2: Fix draft.md and regenerate**
```bash
# Update draft.md first, then
gemini ask "Regenerate GEMINI.md from @draft.md..."
```

Option 2 is better for team continuity.

### Q3: Should I commit both draft.md and GEMINI.md?

**A:** **Yes, commit both.**

```bash
git add draft.md GEMINI.md
git commit -m "Update: project documentation from draft to GEMINI.md"
```

**Why:**
- draft.md = team shared notes (short, editable)
- GEMINI.md = official spec (long, reviewed)
- Both in history = clear change tracking
