# Complete Guide to Gemini 3's `thinking_level` Parameter: Control Reasoning Depth

With the release of **Gemini 3** in early 2026, Google introduced a powerful new API parameter — **`thinking_level`** — that lets developers directly control how deeply the model "thinks" before generating a response. This guide covers everything you need to know: what it is, how it works, when to use each setting, and how to implement it with real Python code examples.

---

## What Is `thinking_level`?

Before generating a response, Gemini 3 internally analyzes problems through a multi-step reasoning process. The `thinking_level` parameter controls the **maximum depth** of this internal thinking process.

Importantly, Gemini 3 treats these levels as **relative guidelines for reasoning depth** — not as strict token guarantees. The actual amount of reasoning varies based on the complexity of the problem at hand.

### Available Values

| Value | Reasoning Depth | Primary Use Case |
|---|---|---|
| `"low"` | Shallow (fast) | Simple queries, latency-sensitive apps |
| `"medium"` | Moderate (balanced) | General tasks — recommended default |
| `"high"` | Deep (high accuracy) | Complex reasoning, math, coding |

---

## Why Does `thinking_level` Matter?

### The Cost-Latency Trade-off

Internal reasoning is powerful, but it increases token consumption and response latency. By tuning `thinking_level` appropriately, you can:

- **Reduce costs**: Use `"low"` for simple tasks to avoid burning unnecessary tokens
- **Improve speed**: `"low"` skips complex reasoning chains for faster responses
- **Maximize accuracy**: Use `"high"` for hard problems to unlock the model's full potential

### PhD-Level Reasoning Performance

Even Gemini 3 Flash, when configured with `thinking_level: "high"`, achieves state-of-the-art scores on PhD-level reasoning benchmarks:
- **GPQA Diamond**: 90.4%
- **Humanity's Last Exam**: 33.7% (without tools)

These results rival much larger frontier models, making Gemini 3 Flash an extremely cost-effective choice for demanding tasks.

## Code Example: Basic Usage of `thinking_level`

The following Python example demonstrates how to call the Gemini 3 API and dynamically switch `thinking_level` based on task complexity.

```python
import google.generativeai as genai
import os
import time

# Configure your API key
genai.configure(api_key=os.environ["GEMINI_API_KEY"])

def ask_gemini(prompt: str, thinking_level: str = "medium") -> dict:
    """
    Send a prompt to Gemini 3 with a specified thinking_level.

    Args:
        prompt: The user's question or instruction
        thinking_level: Reasoning depth — "low", "medium", or "high"

    Returns:
        A dict containing the response text, thinking level used, and elapsed time
    """
    model = genai.GenerativeModel(
        model_name="gemini-3-flash",
        generation_config={
            "thinking_level": thinking_level,  # Control reasoning depth here
        }
    )

    start = time.time()
    response = model.generate_content(prompt)
    elapsed = time.time() - start

    return {
        "text": response.text,
        "thinking_level": thinking_level,
        "elapsed_sec": round(elapsed, 2),
    }


# ── Test Runs ──────────────────────────────────────────────────────────────
if __name__ == "__main__":
    # Simple question → "low" is sufficient
    simple_q = "What is the capital of Japan?"
    result_low = ask_gemini(simple_q, thinking_level="low")
    print(f"[LOW]    {result_low['elapsed_sec']}s → {result_low['text'][:80]}")

    # Complex math problem → "high" for maximum accuracy
    hard_q = "Find the sum of all integers from 1 to 1000 that are divisible by neither 3 nor 5."
    result_high = ask_gemini(hard_q, thinking_level="high")
    print(f"[HIGH]   {result_high['elapsed_sec']}s → {result_high['text'][:80]}")

# Expected output:
# [LOW]    0.41s → Tokyo is the capital of Japan...
# [HIGH]   2.93s → To solve this, we use inclusion-exclusion: ...
```

## Integrating with Thought Signatures

Alongside `thinking_level`, Gemini 3 introduced **Thought Signatures** — encrypted representations of the model's internal reasoning chain. By passing these signatures back in subsequent API calls, Gemini 3 maintains continuity of reasoning across a multi-turn conversation.

This combination is especially powerful for **complex, multi-step agentic workflows**.

```python
import google.generativeai as genai
import os

genai.configure(api_key=os.environ["GEMINI_API_KEY"])

model = genai.GenerativeModel(
    model_name="gemini-3-flash",
    generation_config={
        "thinking_level": "high",  # Use deep reasoning for coherent multi-turn chats
    }
)

chat = model.start_chat()

# Turn 1: Ask the initial question
response1 = chat.send_message(
    "Explain Shor's algorithm for quantum computing."
)

# Thought Signatures are automatically carried forward by the SDK.
# Turn 2: The model retains context from Turn 1's reasoning chain.
response2 = chat.send_message(
    "Now implement that algorithm as Python pseudocode."
)

print(response2.text)

# Expected output:
# Coherent Python pseudocode that directly follows from the Turn 1 explanation
```

### Key Notes on Thought Signatures

- Thought Signatures are only valid **within the same model session**
- They **cannot** be transferred across different model versions
- Supported only in **Gemini 3 and later** (not available in Gemini 2.x)

## Recommended `thinking_level` by Use Case

### Use `"low"` when:

- Handling casual chitchat in a customer service chatbot
- Running simple translation or summarization tasks
- Streaming real-time responses where sub-second latency is required
- Building mobile apps with strict response time requirements

### Use `"medium"` (default recommendation) when:

- Creating general business documents or reports
- Performing light code reviews or bug triage
- Generating FAQ responses or knowledge base answers
- Writing marketing or social media content

### Use `"high"` when:

- Solving mathematical proofs or complex calculations
- Designing or implementing sophisticated algorithms
- Performing expert-level analysis in legal, medical, or scientific domains
- Running multi-step agentic tasks requiring deep reasoning

## Cost Comparison by Level

`thinking_level` directly affects token consumption. The table below shows approximate relative values (actual usage varies by prompt and problem complexity):

| Level | Relative Token Usage | Relative Latency |
|---|---|---|
| `"low"` | ×1 (baseline) | ×1 (baseline) |
| `"medium"` | ×2–3 | ×2–3 |
| `"high"` | ×5–10 | ×5–10 |

**Cost optimization tip:** Pre-classify tasks by complexity and dynamically select the appropriate level.

```python
def smart_ask(prompt: str) -> str:
    """
    Automatically selects the optimal thinking_level based on prompt keywords.
    """
    complex_keywords = ["algorithm", "proof", "implement", "optimize", "calculate", "design"]
    simple_keywords = ["translate", "summarize", "hello", "what is", "when is"]

    if any(kw in prompt.lower() for kw in complex_keywords):
        level = "high"
    elif any(kw in prompt.lower() for kw in simple_keywords):
        level = "low"
    else:
        level = "medium"

    result = ask_gemini(prompt, thinking_level=level)
    print(f"→ Selected thinking_level='{level}' ({result['elapsed_sec']}s)")
    return result["text"]

# Expected output:
# → Selected thinking_level='high' (3.05s)
```

## FAQ

### Q1. What happens if I don't set `thinking_level`?

If you omit `thinking_level`, Gemini 3 will internally assess the task complexity and automatically select an appropriate reasoning depth. However, explicitly setting the parameter is recommended when you need predictable cost or latency characteristics.

### Q2. Is `thinking_level` supported in Gemini 2.x?

No. The `thinking_level` parameter is only supported in **Gemini 3 and later models**. Passing it to Gemini 2.x will either be silently ignored or trigger an API error. Use Gemini 3 Flash or Gemini 3 Pro to access this feature.

### Q3. Does `thinking_level="high"` guarantee accurate answers?

`"high"` maximizes reasoning depth but does **not guarantee 100% accuracy**. For tasks requiring up-to-date information, combine it with `grounding_with_google_search`. For precise calculations, integrate a code interpreter tool. Always validate critical outputs programmatically.
