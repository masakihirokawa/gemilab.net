## Introduction

When running Gemini API-powered applications in production, you'll inevitably encounter network failures, transient server errors, and rate limits. While manually retrying a request is fine during development, production systems need robust error handling to maintain a smooth user experience.

This guide walks through the major error types you'll encounter with the Gemini API, then shows you how to implement **exponential backoff retries**, **circuit breaker patterns**, and **model fallback strategies** — all with working Python code you can adapt for your own projects.

**What you'll learn:**

- How to classify Gemini API errors and decide which ones to retry
- Implementing exponential backoff with jitter correctly
- Using the circuit breaker pattern to prevent cascading failures
- Designing model fallback for high availability
- Logging and monitoring best practices for production

**Who this is for:** Developers with Gemini API experience who are preparing to deploy to production environments.

## Understanding Gemini API Error Types

Gemini API errors map to standard HTTP status codes and fall into two main categories. Knowing which errors are retryable — and which aren't — is the foundation of good error handling.

### Retryable Errors (Transient Failures)

| Status Code | Meaning | Common Cause |
|---|---|---|
| 429 | Too Many Requests | Rate limit exceeded |
| 500 | Internal Server Error | Temporary server-side issue |
| 503 | Service Unavailable | Temporary service overload |

These errors are temporary. Waiting and retrying usually resolves them.

### Non-Retryable Errors (Configuration Issues)

| Status Code | Meaning | Common Cause |
|---|---|---|
| 400 | Bad Request | Malformed request body |
| 401 / 403 | Unauthorized / Forbidden | Invalid API key or insufficient permissions |
| 404 | Not Found | Non-existent model or endpoint |

Retrying these won't help — you need to fix the underlying configuration or request. For authentication-specific issues, check out our [Gemini API Authentication Errors FAQ](/en/articles/gemini-api/gemini-api-authentication-errors-faq).

## Implementing Exponential Backoff with Retry

For retryable errors, **exponential backoff** is the gold standard. The idea is simple: wait longer between each retry attempt, and add random jitter to prevent the "thundering herd" problem where many clients retry simultaneously.

```python
import time
import random
import google.generativeai as genai
from google.api_core import exceptions

genai.configure(api_key="YOUR_API_KEY")

def call_gemini_with_retry(
    prompt: str,
    model_name: str = "gemini-3-flash",
    max_retries: int = 5,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
) -> str:
    """Call Gemini API with exponential backoff retry."""
    model = genai.GenerativeModel(model_name)

    for attempt in range(max_retries + 1):
        try:
            response = model.generate_content(prompt)
            return response.text

        except exceptions.ResourceExhausted as e:
            # 429: Rate limited — retryable
            if attempt == max_retries:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay * 0.5)
            wait_time = delay + jitter
            print(f"[Retry {attempt + 1}/{max_retries}] Rate limited. "
                  f"Waiting {wait_time:.1f}s...")
            time.sleep(wait_time)

        except exceptions.InternalServerError as e:
            # 500: Server error — retryable
            if attempt == max_retries:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay * 0.3)
            wait_time = delay + jitter
            print(f"[Retry {attempt + 1}/{max_retries}] Server error. "
                  f"Waiting {wait_time:.1f}s...")
            time.sleep(wait_time)

        except exceptions.ServiceUnavailable as e:
            # 503: Service down — retryable
            if attempt == max_retries:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay * 0.3)
            wait_time = delay + jitter
            print(f"[Retry {attempt + 1}/{max_retries}] Service unavailable. "
                  f"Waiting {wait_time:.1f}s...")
            time.sleep(wait_time)

        except (exceptions.InvalidArgument, exceptions.PermissionDenied) as e:
            # 400, 403: Not retryable — raise immediately
            raise

# Usage
try:
    result = call_gemini_with_retry("Explain the benefits of list comprehensions in Python")
    print(result)
except Exception as e:
    print(f"All retries exhausted: {e}")

# Expected output:
# List comprehensions offer several benefits...
# (If transient error occurs, retries automatically until success)
```

### Key Implementation Details

**Why jitter matters:** Without jitter, hundreds of clients hitting a rate limit at the same time will all retry at the exact same moment, overwhelming the server again. Adding randomness spreads out the retry attempts.

**Capping the maximum delay:** Exponential growth can lead to absurdly long wait times. The `max_delay` parameter (60 seconds in this example) puts a ceiling on how long any single retry wait can be.

**Limiting retry attempts:** Infinite retries are dangerous. Cap at around 5 attempts, then propagate the error to the caller for appropriate handling.

## The Circuit Breaker Pattern

Retries alone can't handle prolonged outages. If the API is down for an extended period, retrying every request wastes resources and degrades your application's overall responsiveness.

A **circuit breaker** monitors failure rates and temporarily blocks requests when consecutive failures exceed a threshold. After a cooldown period, it allows a few test requests through to check if the service has recovered — just like an electrical circuit breaker.

```python
import time
from enum import Enum
from dataclasses import dataclass, field

class CircuitState(Enum):
    CLOSED = "closed"       # Normal: requests flow through
    OPEN = "open"           # Tripped: requests are blocked
    HALF_OPEN = "half_open" # Testing: limited requests allowed

@dataclass
class CircuitBreaker:
    """Circuit breaker for Gemini API calls."""
    failure_threshold: int = 5        # Consecutive failures to trip
    recovery_timeout: float = 30.0    # Seconds before testing recovery
    success_threshold: int = 2        # Successes needed to close

    state: CircuitState = field(default=CircuitState.CLOSED)
    failure_count: int = field(default=0)
    success_count: int = field(default=0)
    last_failure_time: float = field(default=0.0)

    def can_execute(self) -> bool:
        """Check whether a request should be allowed."""
        if self.state == CircuitState.CLOSED:
            return True
        elif self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time >= self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                self.success_count = 0
                print("[Circuit] OPEN → HALF_OPEN: Testing recovery...")
                return True
            return False
        else:  # HALF_OPEN
            return True

    def record_success(self):
        """Record a successful API call."""
        if self.state == CircuitState.HALF_OPEN:
            self.success_count += 1
            if self.success_count >= self.success_threshold:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                print("[Circuit] HALF_OPEN → CLOSED: Service recovered")
        else:
            self.failure_count = 0

    def record_failure(self):
        """Record a failed API call."""
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.state == CircuitState.HALF_OPEN:
            self.state = CircuitState.OPEN
            print("[Circuit] HALF_OPEN → OPEN: Recovery failed")
        elif self.failure_count >= self.failure_threshold:
            self.state = CircuitState.OPEN
            print(f"[Circuit] CLOSED → OPEN: {self.failure_count} consecutive failures")

# Usage
circuit = CircuitBreaker(failure_threshold=3, recovery_timeout=30)

def safe_gemini_call(prompt: str) -> str:
    """API call with circuit breaker protection."""
    if not circuit.can_execute():
        raise RuntimeError("Circuit OPEN: API requests temporarily suspended")

    try:
        result = call_gemini_with_retry(prompt, max_retries=2)
        circuit.record_success()
        return result
    except Exception as e:
        circuit.record_failure()
        raise

# Expected behavior:
# Normal → Passes requests through
# 3 consecutive failures → Circuit OPEN (blocks for 30s)
# After 30s → HALF_OPEN (allows test requests)
# 2 successes → CLOSED (fully recovered)
```

## Model Fallback Strategy

The Gemini API offers multiple models, which means you can fall back to an alternative when your primary model is unavailable. For example, if Gemini 3 Pro hits an issue, switching to Gemini 3 Flash maintains service continuity — the output quality may differ slightly, but your users stay unblocked.

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class ModelConfig:
    name: str
    priority: int  # Lower number = higher priority

# Prioritized model list
FALLBACK_MODELS = [
    ModelConfig(name="gemini-3-pro", priority=1),
    ModelConfig(name="gemini-3-flash", priority=2),
    ModelConfig(name="gemini-2.5-flash", priority=3),
]

def call_with_fallback(
    prompt: str,
    models: list[ModelConfig] = FALLBACK_MODELS,
) -> tuple[str, str]:
    """Call Gemini API with automatic model fallback.

    Returns:
        tuple: (response text, model name used)
    """
    errors: list[str] = []

    for model_config in sorted(models, key=lambda m: m.priority):
        try:
            print(f"[Fallback] Trying model: {model_config.name}")
            result = call_gemini_with_retry(
                prompt,
                model_name=model_config.name,
                max_retries=2,  # Fewer retries per model in fallback mode
            )
            return result, model_config.name

        except Exception as e:
            errors.append(f"{model_config.name}: {e}")
            print(f"[Fallback] {model_config.name} failed: {e}")
            continue

    raise RuntimeError(
        f"All models failed:\n" + "\n".join(errors)
    )

# Usage
try:
    text, used_model = call_with_fallback("Explain the basics of quantum computing")
    print(f"Responded with: {used_model}")
    print(text)
except RuntimeError as e:
    print(f"Service unavailable: {e}")

# Expected output:
# [Fallback] Trying model: gemini-3-pro
# [Fallback] gemini-3-pro failed: 503 Service Unavailable
# [Fallback] Trying model: gemini-3-flash
# Responded with: gemini-3-flash
# Quantum computing is...
```

## Logging and Monitoring in Production

Writing error handling code is only half the battle. In production, you need visibility into **what's happening and how often** — that's where structured logging and monitoring come in.

### Structured Logging Implementation

```python
import json
import logging
from datetime import datetime, timezone, timedelta

# UTC timezone
UTC = timezone.utc

# Set up structured logging
logger = logging.getLogger("gemini_api")
logger.setLevel(logging.INFO)
handler = logging.StreamHandler()
handler.setFormatter(logging.Formatter("%(message)s"))
logger.addHandler(handler)

def log_api_call(
    model: str,
    prompt_tokens: int,
    status: str,
    latency_ms: float,
    error: Optional[str] = None,
    retry_count: int = 0,
):
    """Log API call results as structured JSON."""
    log_entry = {
        "timestamp": datetime.now(UTC).isoformat(),
        "service": "gemini_api",
        "model": model,
        "prompt_tokens": prompt_tokens,
        "status": status,         # "success", "retry", "error"
        "latency_ms": round(latency_ms, 2),
        "retry_count": retry_count,
    }
    if error:
        log_entry["error"] = error

    logger.info(json.dumps(log_entry, ensure_ascii=False))

# Example output:
# {"timestamp": "2026-03-24T11:15:00+00:00", "service": "gemini_api",
#  "model": "gemini-3-flash", "prompt_tokens": 128, "status": "success",
#  "latency_ms": 342.5, "retry_count": 0}
```

### Key Metrics to Monitor

For a production Gemini API deployment, keep a close eye on these metrics through your monitoring dashboard:

- **Error rate**: Percentage of failed requests. Alert if it exceeds 5%.
- **Retry rate**: How often retries are needed. Frequent retries signal you're approaching rate limits.
- **Latency P95/P99**: The 95th and 99th percentile response times — these reveal tail latency issues.
- **Circuit breaker state transitions**: How often the circuit opens and how long it stays open.
- **Fallback activation rate**: How frequently alternative models are used instead of the primary.

## FAQ

### Q1: How many retries should I configure?

**3 to 5 retries** is the sweet spot for most use cases. Too few and you won't survive transient hiccups; too many and your users wait too long. A good approach is to differentiate by error type — 5 retries for rate limits (429), 3 for server errors (500/503).

### Q2: What's a good recovery_timeout for the circuit breaker?

**30 to 60 seconds** is a reasonable starting point. Most Gemini API server-side issues resolve within a few minutes, so testing recovery after 30 seconds strikes a good balance. Tune this based on your actual traffic patterns and SLA requirements.

### Q3: How should I handle quality differences when using a fallback model?

Include the model name in your response metadata so your UI can display a notice like "Currently responding with an alternative model." For quality-critical operations like code generation, you might choose to return an error instead of falling back to a less capable model.

### Q4: How do I implement retry patterns with async code?

In Python, the `tenacity` library pairs beautifully with `asyncio`. Its `@retry` decorator lets you declaratively define exponential backoff and retry conditions, and it supports async functions natively.

### Q5: What happens if an error occurs mid-stream during a streaming response?

For streaming errors, the ideal approach is to detect the failure, discard the partial response, and restart the stream from the beginning. The Gemini API doesn't support resuming a stream from a midpoint, so the practical flow is: **detect error → retry the full streaming request → replace partial output with the new stream**.
