## Introduction — Why Keyword Search Falls Short

Traditional keyword-based search has a fundamental limitation: it can't understand meaning. A search for "tips for walking your dog safely" won't surface an article titled "Pet outdoor safety precautions" — even though they're about the same thing.

Semantic search solves this by converting text into vectors (arrays of numbers) that capture meaning, then calculating similarity based on the distance between those vectors. Gemini's Embedding API generates high-quality vectors in 768 or 3072 dimensions, with excellent multilingual support out of the box.

In this guide, we'll build a production-grade semantic search engine using **Gemini's Embedding API** and **PostgreSQL pgvector**, walking through every step from vector generation to deployment.

### What You'll Learn

- How to use the Gemini Embedding API (`text-embedding-004` / `gemini-embedding-exp-03-07`)
- Setting up PostgreSQL with the pgvector extension and choosing the right index strategy
- Batch embedding generation and efficient database storage
- Building a semantic search API with FastAPI
- Performance tuning with HNSW and IVFFlat indexes

## Generating Vectors with the Gemini Embedding API

Gemini's Embedding API transforms text into high-quality vectors. The `task_type` parameter lets you optimize vectors for specific use cases like retrieval, classification, or clustering.

```python
# embed_documents.py — Generate text embeddings with Gemini
from google import genai
import numpy as np

# Initialize client
client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

def generate_embedding(text: str, task_type: str = "RETRIEVAL_DOCUMENT") -> list[float]:
    """
    Generate an embedding vector from text.

    Args:
        text: The text to vectorize
        task_type: "RETRIEVAL_DOCUMENT" (for storage) or "RETRIEVAL_QUERY" (for search)

    Returns:
        A list of 768 floating-point numbers
    """
    result = client.models.embed_content(
        model="text-embedding-004",
        contents=text,
        config={
            "task_type": task_type,
            "output_dimensionality": 768  # Configurable: 256 to 3072
        }
    )
    return result.embeddings[0].values

# Usage example
doc_text = "PostgreSQL is a powerful open-source relational database system"
vector = generate_embedding(doc_text)
print(f"Vector dimensions: {len(vector)}")  # Output: Vector dimensions: 768
print(f"First 5 elements: {vector[:5]}")
# Example output: First 5 elements: [0.0234, -0.0156, 0.0412, -0.0089, 0.0301]
```

### Choosing the Right task_type

| task_type | Purpose | When to Use |
|-----------|---------|-------------|
| `RETRIEVAL_DOCUMENT` | Document storage | When saving vectors to your database |
| `RETRIEVAL_QUERY` | Search queries | When vectorizing user search input |
| `SEMANTIC_SIMILARITY` | Similarity measurement | When comparing the semantic distance between two texts |
| `CLASSIFICATION` | Text classification | When categorizing documents |
| `CLUSTERING` | Grouping | When organizing documents into groups |

> **Key insight**: Using `RETRIEVAL_DOCUMENT` for storage and `RETRIEVAL_QUERY` for search queries significantly improves retrieval accuracy compared to using a single task type for both.

## Batch Processing for Large-Scale Document Vectorization

In production environments, you'll typically need to process thousands or even millions of documents. Gemini's batch embedding feature makes this efficient.

```python
# batch_embed.py — Batch vectorize and store documents
import psycopg2
from google import genai
import time

client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

def batch_embed_and_store(documents: list[dict], batch_size: int = 100):
    """
    Batch vectorize documents and store them in PostgreSQL.

    Args:
        documents: List of {"title": "...", "content": "..."} dicts
        batch_size: Number of documents per API call
    """
    conn = psycopg2.connect(
        host="localhost",
        database="semantic_search",
        user="postgres",
        password="your_password"
    )
    cur = conn.cursor()

    for i in range(0, len(documents), batch_size):
        batch = documents[i:i + batch_size]
        texts = [doc["content"] for doc in batch]

        # Generate batch embeddings
        result = client.models.embed_content(
            model="text-embedding-004",
            contents=texts,
            config={
                "task_type": "RETRIEVAL_DOCUMENT",
                "output_dimensionality": 768
            }
        )

        # Store in database
        for doc, emb in zip(batch, result.embeddings):
            cur.execute(
                """
                INSERT INTO documents (title, content, embedding)
                VALUES (%s, %s, %s::vector)
                """,
                (doc["title"], doc["content"], str(emb.values))
            )

        conn.commit()
        print(f"  Batch {i // batch_size + 1}: {len(batch)} documents stored")

        # Rate limit protection (assuming 1,500 RPM)
        time.sleep(0.5)

    cur.close()
    conn.close()
    print(f"All {len(documents)} documents vectorized and stored successfully")

# Usage example
sample_docs = [
    {"title": "Intro to PostgreSQL", "content": "PostgreSQL is the world's most advanced open source database..."},
    {"title": "What is Vector Search", "content": "Vector search retrieves results based on semantic similarity..."},
    {"title": "Python API Development", "content": "FastAPI lets you build high-performance APIs in Python..."},
]
batch_embed_and_store(sample_docs)
# Output:
#   Batch 1: 3 documents stored
# All 3 documents vectorized and stored successfully
```

## pgvector Index Strategies — HNSW vs IVFFlat

The performance of your vector search depends heavily on which index type you choose. pgvector offers two primary options, each with distinct trade-offs.

### HNSW (Hierarchical Navigable Small World)

Delivers high recall and consistent performance. This should be your **default choice for production workloads**.

```sql
-- Create an HNSW index (recommended settings)
CREATE INDEX idx_documents_embedding_hnsw
ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 200);

-- Tune search-time parameters
SET hnsw.ef_search = 100;  -- Higher = better recall, slower speed
```

### IVFFlat (Inverted File with Flat Compression)

Faster to build and more memory-efficient, making it suitable for very large datasets. However, recall may be lower than HNSW.

```sql
-- Create an IVFFlat index
-- Recommended lists value: sqrt(row_count) to row_count/1000
CREATE INDEX idx_documents_embedding_ivfflat
ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- Tune probe count for search
SET ivfflat.probes = 10;  -- Higher = better recall, slower speed
```

### Performance Comparison

| Metric | HNSW | IVFFlat |
|--------|------|---------|
| Search latency (100K docs) | ~5ms | ~3ms |
| Recall@10 | 99.2% | 95.8% |
| Index build time | Longer | Shorter |
| Memory usage | Higher | Lower |
| Recommended scale | Up to 1M docs | 1M+ docs |

> **Recommendation**: For datasets under one million documents, choose HNSW. The difference in recall directly impacts user experience.

## Building a Semantic Search API with FastAPI

Now let's bring everything together into a production-ready search API using FastAPI.

```python
# api.py — FastAPI semantic search service
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field
import psycopg2
from psycopg2.extras import RealDictCursor
from google import genai
from contextlib import asynccontextmanager

# --- Configuration ---
GEMINI_API_KEY = "YOUR_GEMINI_API_KEY"
DB_CONFIG = {
    "host": "localhost",
    "database": "semantic_search",
    "user": "postgres",
    "password": "your_password"
}

client = genai.Client(api_key=GEMINI_API_KEY)

@asynccontextmanager
async def lifespan(app: FastAPI):
    """Verify DB connection on startup"""
    conn = psycopg2.connect(**DB_CONFIG)
    conn.close()
    print("DB connection verified")
    yield

app = FastAPI(
    title="Semantic Search API",
    description="Semantic search powered by Gemini Embeddings + pgvector",
    version="1.0.0",
    lifespan=lifespan
)

class SearchRequest(BaseModel):
    query: str = Field(..., min_length=1, max_length=1000, description="Search query")
    top_k: int = Field(default=5, ge=1, le=50, description="Number of results to return")
    similarity_threshold: float = Field(
        default=0.7, ge=0.0, le=1.0,
        description="Minimum cosine similarity threshold"
    )

class SearchResult(BaseModel):
    id: int
    title: str
    content: str
    similarity: float
    metadata: dict

@app.post("/search", response_model=list[SearchResult])
async def semantic_search(request: SearchRequest):
    """Execute a semantic search query"""
    try:
        # Vectorize the query (note: RETRIEVAL_QUERY task type)
        result = client.models.embed_content(
            model="text-embedding-004",
            contents=request.query,
            config={
                "task_type": "RETRIEVAL_QUERY",
                "output_dimensionality": 768
            }
        )
        query_vector = result.embeddings[0].values

        # Run cosine similarity search with pgvector
        conn = psycopg2.connect(**DB_CONFIG, cursor_factory=RealDictCursor)
        cur = conn.cursor()

        cur.execute(
            """
            SELECT
                id, title, content, metadata,
                1 - (embedding <=> %s::vector) AS similarity
            FROM documents
            WHERE 1 - (embedding <=> %s::vector) >= %s
            ORDER BY embedding <=> %s::vector
            LIMIT %s
            """,
            (
                str(query_vector),
                str(query_vector),
                request.similarity_threshold,
                str(query_vector),
                request.top_k
            )
        )

        results = cur.fetchall()
        cur.close()
        conn.close()

        return [
            SearchResult(
                id=row["id"],
                title=row["title"],
                content=row["content"][:500],  # Truncate for preview
                similarity=round(float(row["similarity"]), 4),
                metadata=row["metadata"] or {}
            )
            for row in results
        ]

    except Exception as e:
        raise HTTPException(status_code=500, detail=f"Search error: {str(e)}")

@app.post("/index")
async def index_document(title: str, content: str):
    """Add a new document to the search index"""
    try:
        result = client.models.embed_content(
            model="text-embedding-004",
            contents=content,
            config={
                "task_type": "RETRIEVAL_DOCUMENT",
                "output_dimensionality": 768
            }
        )
        embedding = result.embeddings[0].values

        conn = psycopg2.connect(**DB_CONFIG)
        cur = conn.cursor()
        cur.execute(
            """
            INSERT INTO documents (title, content, embedding)
            VALUES (%s, %s, %s::vector)
            RETURNING id
            """,
            (title, content, str(embedding))
        )
        doc_id = cur.fetchone()[0]
        conn.commit()
        cur.close()
        conn.close()

        return {"id": doc_id, "message": "Document added to the index"}

    except Exception as e:
        raise HTTPException(status_code=500, detail=f"Indexing error: {str(e)}")

# Run: uvicorn api:app --host 0.0.0.0 --port 8000
# Output: INFO: Uvicorn running on http://0.0.0.0:8000
```

## Production Optimization Techniques

### 1. Connection Pooling

In production, you should use a connection pool instead of opening and closing database connections per request.

```python
# connection_pool.py — Use psycopg2's connection pool
from psycopg2 import pool

# Create the pool at application startup
connection_pool = pool.ThreadedConnectionPool(
    minconn=5,
    maxconn=20,
    host="localhost",
    database="semantic_search",
    user="postgres",
    password="your_password"
)

def get_connection():
    return connection_pool.getconn()

def release_connection(conn):
    connection_pool.putconn(conn)
```

### 2. Embedding Cache with Redis

To avoid redundant API calls for repeated queries, cache embeddings in Redis.

```python
# embedding_cache.py — Redis-based embedding cache
import redis
import json
import hashlib

redis_client = redis.Redis(host="localhost", port=6379, db=0)
CACHE_TTL = 3600  # 1 hour

def get_cached_embedding(text: str, task_type: str) -> list[float] | None:
    """Retrieve a cached embedding"""
    key = f"emb:{hashlib.sha256(f'{text}:{task_type}'.encode()).hexdigest()}"
    cached = redis_client.get(key)
    if cached:
        return json.loads(cached)
    return None

def cache_embedding(text: str, task_type: str, embedding: list[float]):
    """Store an embedding in the cache"""
    key = f"emb:{hashlib.sha256(f'{text}:{task_type}'.encode()).hexdigest()}"
    redis_client.setex(key, CACHE_TTL, json.dumps(embedding))
```

### 3. Hybrid Search (Semantic + Full-Text)

Combining semantic search with PostgreSQL's built-in full-text search delivers higher relevance through a technique called Reciprocal Rank Fusion (RRF).

```sql
-- Add a full-text search column
ALTER TABLE documents ADD COLUMN tsv tsvector
    GENERATED ALWAYS AS (to_tsvector('english', content)) STORED;
CREATE INDEX idx_documents_tsv ON documents USING gin(tsv);

-- Hybrid search query using Reciprocal Rank Fusion (RRF)
WITH semantic AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $1::vector) AS rank_s
    FROM documents
    ORDER BY embedding <=> $1::vector
    LIMIT 20
),
fulltext AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(tsv, query) DESC) AS rank_f
    FROM documents, plainto_tsquery('english', $2) query
    WHERE tsv @@ query
    LIMIT 20
)
SELECT d.id, d.title, d.content,
    COALESCE(1.0 / (60 + s.rank_s), 0) +
    COALESCE(1.0 / (60 + f.rank_f), 0) AS rrf_score
FROM documents d
LEFT JOIN semantic s ON d.id = s.id
LEFT JOIN fulltext f ON d.id = f.id
WHERE s.id IS NOT NULL OR f.id IS NOT NULL
ORDER BY rrf_score DESC
LIMIT 10;
```

## Frequently Asked Questions

### Q1: Should I use text-embedding-004 or gemini-embedding-exp-03-07?

`text-embedding-004` is the stable release, producing 768-dimensional vectors. It's the recommended choice for production workloads. `gemini-embedding-exp-03-07` is experimental, capable of generating up to 3072-dimensional vectors with potentially higher accuracy, but its API may change without notice. If stability matters most, stick with `text-embedding-004`. If you need maximum precision and can tolerate breaking changes, consider the experimental model.

### Q2: Should I use 768 or 3072 dimensions?

768 dimensions offer an excellent balance between storage costs, search speed, and accuracy for most use cases. 3072 dimensions improve precision but require roughly 4x the storage and slower searches. For datasets exceeding one million documents, 768 dimensions is strongly recommended to keep performance manageable.

### Q3: Should I use pgvector or a dedicated vector database like Pinecone or Weaviate?

If you already run PostgreSQL, pgvector is often the pragmatic choice — it adds vector search without introducing new infrastructure. However, for datasets in the billions, multi-tenant architectures, or when you need managed scaling, a dedicated vector database may be worth the operational complexity.

### Q4: How often should I re-embed my documents?

Only re-embed when a document's content changes. If Gemini releases a new embedding model version, consider a full re-embedding pass. Mixing vectors from different model versions in the same table will degrade search quality, so always re-embed in bulk.

### Q5: How do I measure semantic search quality?

The two most common metrics are Recall@K (how often the correct result appears in the top K results) and MRR (Mean Reciprocal Rank). Create a benchmark dataset of query-answer pairs and run evaluations regularly, especially after changing your embedding model or index parameters.
