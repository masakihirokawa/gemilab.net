## What Is the thinking_level Parameter?

Starting with the Gemini 3 series, Google introduced the `thinking_level` parameter that lets you control how deeply the model reasons before generating a response. By configuring this parameter appropriately, you can fine-tune the balance between response quality and API cost or latency.

By default, Gemini 3 models operate at `HIGH` (dynamic thinking mode), performing thorough reasoning for complex prompts. However, not every request needs maximum reasoning depth. Using `HIGH` for simple translation or summarization tasks wastes money, while using `LOW` for mathematical proofs or complex code generation will degrade quality.

Mastering `thinking_level` is an essential skill for running Gemini API efficiently in production environments.

## Supported Models and Available Levels

The `thinking_level` parameter is available on Gemini 3 and later models. Since available levels differ by model, let's start with the compatibility matrix.

| Model | Available Levels | Default | Notes |
|-------|-----------------|---------|-------|
| Gemini 3.1 Pro | LOW, MEDIUM, HIGH | HIGH | HIGH activates Deep Think Mini |
| Gemini 3 Pro | LOW, MEDIUM, HIGH | HIGH | Migration to 3.1 Pro recommended |
| Gemini 3.1 Flash | MINIMAL, LOW, MEDIUM, HIGH | HIGH | MINIMAL is Flash-exclusive |
| Gemini 3 Flash | MINIMAL, LOW, MEDIUM, HIGH | HIGH | Same as above |

When you specify `HIGH` with Gemini 3.1 Pro, **Deep Think Mini** — a specialized reasoning mode — kicks in. This engine achieved 77.1% on the ARC-AGI-2 benchmark and excels particularly at complex tasks like mathematics, logic, and code generation.

## Basic Implementation with the Python SDK

Let's walk through how to set `thinking_level` using the Google AI Python SDK.

```python
from google import genai
from google.genai import types

# Initialize the client
client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

# Send a request with thinking_level set to MEDIUM
response = client.models.generate_content(
    model="gemini-3.1-pro",
    contents="Write a Python implementation of binary search",
    config=types.GenerateContentConfig(
        thinking_config=types.ThinkingConfig(
            thinking_level="MEDIUM"
        )
    )
)

# Get the response text
print(response.text)

# Check thinking token count (useful for cost estimation)
if response.usage_metadata:
    print(f"Input tokens: {response.usage_metadata.prompt_token_count}")
    print(f"Output tokens: {response.usage_metadata.candidates_token_count}")
    print(f"Thinking tokens: {response.usage_metadata.thoughts_token_count}")
```

Expected output:

```
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1

Input tokens: 14
```

The `thoughts_token_count` field tells you how much the model "thought" internally, which is invaluable for cost estimation.

## JavaScript / TypeScript Implementation

In Node.js environments, use the `@google/genai` package.

```javascript
import { GoogleGenAI } from "@google/genai";

const ai = new GoogleGenAI({ apiKey: "YOUR_GEMINI_API_KEY" });

async function generateWithThinking(prompt, level = "MEDIUM") {
  const response = await ai.models.generateContent({
    model: "gemini-3.1-flash",
    contents: prompt,
    config: {
      thinkingConfig: {
        thinkingLevel: level,  // "MINIMAL" | "LOW" | "MEDIUM" | "HIGH"
      },
    },
  });

  console.log(response.text);

  // Check token usage
  if (response.usageMetadata) {
    console.log(`Thinking tokens: ${response.usageMetadata.thoughtsTokenCount}`);
  }

  return response.text;
}

// Usage examples
await generateWithThinking("Write a function to convert JSON to CSV", "LOW");
await generateWithThinking("Design a recursive type inference algorithm", "HIGH");
```

## Level Characteristics and Use Cases

Each level has distinct characteristics, and choosing the right one for each task type is crucial.

### MINIMAL (Flash Models Only)

Performs almost no internal reasoning, generating responses at maximum speed and minimum cost. Ideal for simple tasks where reasoning isn't needed.

**Best for**: Text formatting, boilerplate generation, simple translations, keyword extraction

### LOW

Performs minimal reasoning — enough for fact-checking and simple logical judgments.

**Best for**: Summarization, sentiment analysis, FAQ responses, template-based code generation

### MEDIUM

The standard reasoning depth. Offers the best quality-to-cost ratio for most business use cases.

**Best for**: Technical documentation, moderately complex code generation, data analysis reports, comparative evaluations

### HIGH (Default)

Maximum reasoning depth with Deep Think Mini (on 3.1 Pro). The model builds reasoning step by step through complex problems.

**Best for**: Mathematical proofs, algorithm design, complex bug identification, architecture design, logic puzzles

## Dynamic Routing Based on Task Type

In production environments, dynamically switching `thinking_level` based on request content is highly effective. Here's an implementation pattern.

```python
from google import genai
from google.genai import types

client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

# Thinking level mapping by task type
TASK_ROUTING = {
    "translate": "LOW",
    "summarize": "LOW",
    "extract": "MINIMAL",
    "code_simple": "MEDIUM",
    "code_complex": "HIGH",
    "analysis": "MEDIUM",
    "reasoning": "HIGH",
    "math": "HIGH",
}

def classify_task(prompt: str) -> str:
    """Classify task type based on prompt content"""
    prompt_lower = prompt.lower()
    if any(w in prompt_lower for w in ["translate", "translation", "convert to"]):
        return "translate"
    if any(w in prompt_lower for w in ["summarize", "summary", "tldr"]):
        return "summarize"
    if any(w in prompt_lower for w in ["extract", "pull out", "parse"]):
        return "extract"
    if any(w in prompt_lower for w in ["prove", "proof", "math", "calculate"]):
        return "math"
    if any(w in prompt_lower for w in ["design", "architect", "algorithm"]):
        return "reasoning"
    if any(w in prompt_lower for w in ["code", "implement", "function", "write a"]):
        return "code_simple"
    return "analysis"

def smart_generate(prompt: str, model: str = "gemini-3.1-flash") -> str:
    """Generate response with optimal thinking_level for the task"""
    task_type = classify_task(prompt)
    level = TASK_ROUTING.get(task_type, "MEDIUM")

    response = client.models.generate_content(
        model=model,
        contents=prompt,
        config=types.GenerateContentConfig(
            thinking_config=types.ThinkingConfig(
                thinking_level=level
            )
        )
    )

    # Log output (send to monitoring system in production)
    thoughts = response.usage_metadata.thoughts_token_count or 0
    print(f"[{task_type}] level={level}, thoughts={thoughts} tokens")

    return response.text

# Usage examples
result = smart_generate("Translate this to French: Hello World")
# → [translate] level=LOW, thoughts=12 tokens

result = smart_generate("Prove the time complexity of merge sort in Big O notation")
# → [math] level=HIGH, thoughts=450 tokens
```

This approach keeps costs low for simple tasks with `LOW` while ensuring quality for complex tasks with `HIGH`.

## Cost Comparison and Optimization Tips

Your `thinking_level` choice directly impacts API costs. Thinking tokens are counted separately from output tokens, so setting `HIGH` increases costs due to the additional thinking tokens consumed.

Here's a comparison across levels for the same prompt (a moderately complex code generation task):

```python
import time
from google import genai
from google.genai import types

client = genai.Client(api_key="YOUR_GEMINI_API_KEY")

prompt = "Implement an async HTTP client in Python with retry logic and rate limiting"

levels = ["LOW", "MEDIUM", "HIGH"]

for level in levels:
    start = time.time()
    response = client.models.generate_content(
        model="gemini-3.1-flash",
        contents=prompt,
        config=types.GenerateContentConfig(
            thinking_config=types.ThinkingConfig(
                thinking_level=level
            )
        )
    )
    elapsed = time.time() - start
    meta = response.usage_metadata
    print(f"Level: {level}")
    print(f"  Latency: {elapsed:.2f}s")
    print(f"  Thinking tokens: {meta.thoughts_token_count}")
    print(f"  Output tokens: {meta.candidates_token_count}")
    print()

# Expected output:
# Level: LOW
#   Latency: 1.20s
#   Thinking tokens: 30
#   Output tokens: 280
#
# Level: MEDIUM
#   Latency: 2.15s
#   Thinking tokens: 150
#   Output tokens: 350
#
# Level: HIGH
#   Latency: 4.80s
#   Thinking tokens: 520
#   Output tokens: 480
```

In this example, moving from `LOW` to `HIGH` roughly quadruples latency and increases thinking tokens by about 17x. However, `HIGH` tends to produce more thorough output with better error handling and edge case coverage.

For cost optimization, start by visualizing the `thinking_level` distribution across all your requests and check what percentage uses `HIGH`. In most cases, 60–70% of requests can achieve sufficient quality with `LOW` or `MEDIUM`.

## Verifying Reasoning with Thought Signatures

Deep Think Mini, activated at the `HIGH` level on Gemini 3.1 Pro, includes a feature called **Thought Signatures**. These are encrypted, tamper-proof records of each reasoning step, allowing you to verify that the AI's reasoning process hasn't been altered.

```python
response = client.models.generate_content(
    model="gemini-3.1-pro",
    contents="Prove the computational complexity of Grover's algorithm in quantum computing",
    config=types.GenerateContentConfig(
        thinking_config=types.ThinkingConfig(
            thinking_level="HIGH"
        )
    )
)

# Inspect the thinking process
for part in response.candidates[0].content.parts:
    if part.thought:
        print("[Thinking]", part.text[:200], "...")
    else:
        print("[Response]", part.text[:200], "...")
```

Thought Signatures are particularly valuable in healthcare, legal, and financial domains where transparency in AI reasoning is essential. Because each reasoning step is recorded, you can trace back "why the model reached this conclusion" after the fact.

## Frequently Asked Questions

### Q1: What's the default when thinking_level isn't specified?

For all Gemini 3 series models, the default is `HIGH`. When not explicitly specified, the model operates at maximum reasoning depth. To optimize costs, we recommend explicitly setting the level based on task complexity.

### Q2: Can I use thinking_level with Gemini 2.5 models?

The `thinking_level` parameter is recommended for Gemini 3 and later. Gemini 2.5 Pro/Flash used `thinking_budget` (token-count based) for reasoning control. Starting with the 3 series, Google transitioned to the more intuitive `thinking_level`. When migrating existing code, map your `thinking_budget` values to the corresponding levels.

### Q3: How much quality difference is there between HIGH and MEDIUM?

It depends on the task type. For simple Q&A and summarization, there's virtually no difference. However, for mathematical reasoning, multi-step logic problems, and complex code generation, `HIGH` produces clearly superior results. A practical approach is to start with `MEDIUM` and switch to `HIGH` only when quality falls short.

### Q4: Can I combine it with streaming?

Yes, `thinking_level` works with streaming mode. The thinking portion (`thought: true`) is delivered first, followed by the response text. For more on combining streaming with thinking_level, check out the [Gemini API Streaming × Function Calling Implementation Guide](/articles/gemini-api/gemini-api-streaming-function-calling-guide).

### Q5: What's the difference between MINIMAL and LOW on Flash models?

`MINIMAL` performs almost no reasoning, generating responses based on token prediction. `LOW` performs minimal reasoning steps (fact-checking, simple logical judgments). For format conversion and template output where reasoning is fundamentally unnecessary, `MINIMAL` minimizes costs.
