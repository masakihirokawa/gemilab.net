## Introduction: What Are Thought Signatures?

With the launch of **Gemini 3**, Google introduced a powerful new mechanism called **Thought Signatures** to the Gemini API. These are encrypted representations of the model's internal reasoning process, designed to preserve reasoning continuity across multi-turn agentic workflows.

Before Gemini 3, the API operated in a purely stateless fashion — each request was independent, and the model had no way to carry forward *why* it made a particular decision in a previous turn. Thought Signatures solve this by providing a cryptographic anchor that ties successive reasoning steps together.

## How Thought Signatures Work

### The Core Concept

When Gemini 3 processes a request, it performs internal reasoning (Thinking) before generating a response. Part of this reasoning is serialized into an encrypted **Thought Signature** that is returned alongside the response. Passing this signature back in the next API call tells the model to continue its reasoning chain from where it left off.

```
User Message
  → Gemini 3 Internal Reasoning (Thinking)
    → Reasoning serialized as Thought Signature (encrypted)
      → Response generated
        → Response contains: Text + Thought Signature
```

On the next turn, you send the Thought Signature back as part of the conversation history, and Gemini 3 uses it to maintain coherent, high-quality reasoning across the entire interaction.

### Why It's Mandatory in Gemini 3

For agentic workflows that use Function Calling, returning Thought Signatures is **enforced** by the Gemini 3 API. Omitting them will result in a **4xx error**. This is an intentional design choice to guarantee reasoning integrity in complex, multi-step tasks.

From the official documentation:

> *"If you receive a thought signature in a model response, pass it back in the conversation history when you make the next request."*

## Step-by-Step Implementation

### Case 1: Simple Multi-Turn Conversation (Automatic)

When you use the SDK's `ChatSession`, Thought Signatures are managed for you automatically. No extra work required.

```python
import os
from google import genai

client = genai.Client(api_key=os.environ["GOOGLE_API_KEY"])

# ChatSession handles Thought Signatures automatically
chat = client.chats.create(model="gemini-3.1-pro-preview")

# Turn 1
response1 = chat.send_message("Explain Python decorators.")
print("Turn 1:", response1.text[:200])
# Expected output:
# Turn 1: Python decorators are a form of syntactic sugar that allow you to
# modify or extend the behavior of functions or classes without changing their source code...

# Turn 2 — reasoning context from Turn 1 is automatically preserved
response2 = chat.send_message("Show me a practical example.")
print("Turn 2:", response2.text[:200])
# Expected output:
# Turn 2: Building on the decorator concept I just described, here's a practical
# example using a timing decorator...
```

Use `ChatSession` whenever possible. It abstracts away all Thought Signature management.

### Case 2: Manual Function Calling Implementation

If you need full control over the agentic loop — for example, to route tool calls to external services — you'll manage conversation history manually. This is where Thought Signatures require explicit handling.

```python
import os
from google import genai
from google.genai import types

client = genai.Client(api_key=os.environ["GOOGLE_API_KEY"])

# Tool definition: retrieve weather data
weather_tool = types.Tool(
    function_declarations=[
        types.FunctionDeclaration(
            name="get_weather",
            description="Get the current weather for a given city",
            parameters=types.Schema(
                type="object",
                properties={
                    "city": types.Schema(
                        type="string",
                        description="City name (e.g., Tokyo, New York)"
                    )
                },
                required=["city"]
            )
        )
    ]
)

# Simulated weather function
def get_weather(city: str) -> dict:
    data = {
        "Tokyo": {"temp_c": 18, "condition": "Sunny", "humidity": 45},
        "New York": {"temp_c": 12, "condition": "Cloudy", "humidity": 65},
    }
    return data.get(city, {"temp_c": 15, "condition": "Unknown", "humidity": 50})

# Initial request
contents = [
    types.Content(
        role="user",
        parts=[types.Part(text="What's the weather like in Tokyo and New York today?")]
    )
]

response = client.models.generate_content(
    model="gemini-3.1-pro-preview",
    contents=contents,
    config=types.GenerateContentConfig(tools=[weather_tool])
)

# ✅ Critical: save ALL parts as-is — this preserves the Thought Signatures
contents.append(
    types.Content(role="model", parts=response.candidates[0].content.parts)
)

# Process function calls
function_response_parts = []
for part in response.candidates[0].content.parts:
    if part.function_call:
        fc = part.function_call
        print(f"Tool call: {fc.name}({dict(fc.args)})")
        # Expected output:
        # Tool call: get_weather({'city': 'Tokyo'})
        # Tool call: get_weather({'city': 'New York'})

        result = get_weather(**fc.args)
        function_response_parts.append(
            types.Part(
                function_response=types.FunctionResponse(
                    name=fc.name,
                    response=result
                )
            )
        )

# Add tool results and get final response
if function_response_parts:
    contents.append(
        types.Content(role="user", parts=function_response_parts)
    )

    final_response = client.models.generate_content(
        model="gemini-3.1-pro-preview",
        contents=contents,
        config=types.GenerateContentConfig(tools=[weather_tool])
    )

    print("Final answer:", final_response.text)
    # Expected output:
    # Final answer: Here's today's weather update:
    # - Tokyo: 18°C, sunny with 45% humidity — a great day to be outside!
    # - New York: 12°C, cloudy with 65% humidity — might want to bring a jacket.
```

**Key rule**: Always store `response.candidates[0].content.parts` as-is when appending to the `model` role in your conversation history. Never cherry-pick individual parts or reconstruct them, as this will drop the Thought Signatures.

### Case 3: Parallel Function Calls

When Gemini 3 issues multiple function calls simultaneously (parallel tool use), only the **first function call part** in the response contains the Thought Signature. All subsequent parts do not. This makes it even more important to save the full `parts` array rather than filtering.

```python
# Parallel function calls — save the entire parts array
model_parts = response.candidates[0].content.parts

# ✅ Correct: preserve all parts including the Thought Signature in part[0]
contents.append(types.Content(role="model", parts=model_parts))

# ❌ Wrong: filtering to only function_call parts loses the Thought Signature
# fc_only = [p for p in model_parts if p.function_call]
# contents.append(types.Content(role="model", parts=fc_only))  # Will cause 400 error!
```

## Common Errors and Fixes

### Error 1: 400 Bad Request (INVALID_ARGUMENT)

**Cause**: Thought Signature missing or corrupted in the conversation history.

**Fix**: Ensure you're storing `response.candidates[0].content.parts` unmodified as the `model` role entry.

```python
# ❌ Will break: reconstructing response as plain text only
contents.append(types.Content(role="model", parts=[types.Part(text=response.text)]))

# ✅ Correct: use the original parts
contents.append(types.Content(role="model", parts=response.candidates[0].content.parts))
```

### Error 2: Thought Signatures Not Present in Response

**Cause**: `thinking_level` is set to `none`, which disables internal reasoning and therefore produces no Thought Signatures.

**Fix**: Either omit `thinking_level` (defaults to `high`) or set it to `low` or above for agentic use cases. See the [Gemini 3 Thinking Level Guide](/en/articles/gemini-api/gemini-3-thinking-level-guide) for details.

### Error 3: Degraded Response Quality in Long Workflows

**Cause**: As the agentic loop grows, the conversation history (including Thought Signatures) approaches the context window limit.

**Fix**: Use [Context Caching](/en/articles/gemini-api/context-caching-guide) to offload fixed content like system prompts and reduce the effective token count of each request.

## Advanced: Production Agent Design with Thought Signatures

### Recommended Architecture

Here are the key principles for building production-ready agents on Gemini 3:

1. **Default to `ChatSession`** — Let the SDK manage Thought Signatures wherever possible. Manual management introduces risk.
2. **If manual, never mutate `content.parts`** — Store and return the raw array from every model response.
3. **Leverage parallel function calls** — Gemini 3 supports simultaneous tool invocations, allowing your agent to gather multiple data points in a single round-trip.
4. **Combine with Interactions API** — For complex session management at scale, the [Gemini Interactions API](/en/articles/gemini-api/gemini-interactions-api-guide) provides a higher-level stateful abstraction on top of Thought Signatures.

### Debugging Thought Signatures

Thought Signatures are opaque to users but you can verify their presence in your agentic loop:

```python
# Debug: check for Thought Signatures in the response
for i, part in enumerate(response.candidates[0].content.parts):
    has_thought = hasattr(part, 'thought') and part.thought
    has_fc = part.function_call is not None
    has_text = bool(part.text)
    print(f"Part {i}: thought_signature={has_thought}, function_call={has_fc}, text={has_text}")
    # Expected output for a thinking + function call response:
    # Part 0: thought_signature=True, function_call=True, text=False
    # Part 1: thought_signature=False, function_call=True, text=False
```

If `thought_signature=False` for all parts when you expect reasoning, check your `thinking_level` setting.

## FAQ

**Q1. Can I read the content of a Thought Signature?**

No. Thought Signatures are encrypted and opaque by design. They are intended to be passed back to the model as-is — you should treat them as an internal token that the model uses, not as data you can inspect or parse.

**Q2. Are Thought Signatures required for Gemini 2.5 models?**

No. Thought Signatures are a **Gemini 3 feature**. If you're still using `gemini-2.5-pro` or `gemini-2.5-flash`, this does not apply. But if you're planning a migration to Gemini 3, audit your Function Calling code early.

**Q3. How do Thought Signatures affect token costs?**

Thought Signatures consume internal "thinking tokens," which are billed separately from input/output tokens in Gemini 3. Review the [Gemini API Cost Optimization guide](/en/articles/gemini-api/gemini-api-cost-optimization) for strategies to manage thinking token usage with the `thinking_budget` parameter.

**Q4. What about the REST API? Do I need to do anything special?**

Yes. When using the REST API directly, ensure that the `content.parts` array from each model response is included verbatim in the `contents` field of your next request under the `model` role. Do not strip, reorder, or transform individual parts.

**Q5. Can I disable Thought Signatures to reduce latency?**

You can reduce thinking depth by setting `thinking_level: "low"`, which lowers latency while still generating Thought Signatures. Setting `thinking_level: "none"` disables thinking entirely and eliminates Thought Signatures — but this is not recommended for Function Calling workflows in Gemini 3, as it may cause validation errors.

## Summary

Thought Signatures represent a foundational shift in how Gemini 3 handles agentic workflows — moving from stateless to stateful reasoning across multiple API calls.

- **What they are**: Encrypted representations of the model's internal reasoning, returned with each response
- **When they matter**: Multi-turn Gemini 3 workflows using Function Calling
- **Implementation rule**: Always store and return `response.candidates[0].content.parts` as-is
- **Easiest path**: Use `ChatSession` and let the SDK handle everything automatically

For developers migrating from Gemini 2.5 to Gemini 3, Thought Signature handling is the most common source of breaking changes. Audit your code early. For building full-featured agents with session management, explore the [Gemini Interactions API](/en/articles/gemini-api/gemini-interactions-api-guide) as a natural next step.
