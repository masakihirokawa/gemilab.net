## Introduction

Finding the right information buried in company documents and knowledge bases is harder than it should be. Traditional keyword search falls short when people use different terms for the same concept, or when the answer lives across multiple documents.

**Semantic search** solves this by converting text into numerical vectors that capture meaning, then finding results based on conceptual similarity rather than exact keyword matches. A query like "how do I request time off" will match documents about "PTO policy" and "vacation procedures" — even without shared keywords.

In this guide, you'll build a complete document search system using **Gemini's Embedding API** for vectorization and **ChromaDB** as a lightweight vector database, all in Python. By the end, you'll have a working prototype that you can extend into a full RAG (Retrieval-Augmented Generation) pipeline.

## How Semantic Search Works with Gemini Embeddings

### The Concept Behind Semantic Search

Semantic search encodes text as high-dimensional vectors where similar meanings cluster together in vector space. When a user submits a query, the system converts it into the same vector space and finds the closest matches using distance metrics like cosine similarity.

### Why Gemini's Embedding API Stands Out

Google's **text-embedding-004** model (along with the multimodal **gemini-embedding-exp-03-07**) offers several advantages for building search systems:

- **768-dimensional vectors**: Rich enough to capture nuanced meaning in text
- **Strong multilingual support**: Handles English, Japanese, and dozens of other languages natively
- **Task type specification**: Separate modes for `RETRIEVAL_DOCUMENT` (indexing) and `RETRIEVAL_QUERY` (searching) that improve retrieval accuracy
- **Generous input limits**: Up to 2,048 tokens per input

### Why ChromaDB

ChromaDB is a Python-native vector database that prioritizes developer experience:

- **Zero-config setup**: Just `pip install chromadb` — no Docker, no PostgreSQL, no infrastructure
- **Local persistence**: Data is stored on disk automatically
- **Metadata filtering**: Combine vector search with structured filters for precise results
- **Production-ready path**: Switch to client/server mode when you need to scale

## Ingesting and Vectorizing Documents

The first step is reading your documents, splitting them into manageable chunks, generating embeddings, and storing everything in ChromaDB.

```python
# ingest.py
import os
import chromadb
from google import genai

# Initialize Gemini client
client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

# ChromaDB with local persistence
chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection = chroma_client.get_or_create_collection(
    name="company_docs",
    metadata={"hnsw:space": "cosine"}  # Use cosine similarity
)


def chunk_text(text: str, chunk_size: int = 500, overlap: int = 100) -> list[str]:
    """Split text into overlapping chunks."""
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start += chunk_size - overlap
    return chunks


def embed_texts(texts: list[str]) -> list[list[float]]:
    """Generate embeddings using Gemini Embedding API."""
    result = client.models.embed_content(
        model="text-embedding-004",
        contents=texts,
        config={
            "task_type": "RETRIEVAL_DOCUMENT",
            "output_dimensionality": 768,
        },
    )
    return [e.values for e in result.embeddings]


def ingest_document(filepath: str, doc_name: str):
    """Read, chunk, embed, and store a document."""
    with open(filepath, "r", encoding="utf-8") as f:
        text = f.read()

    chunks = chunk_text(text)
    embeddings = embed_texts(chunks)

    ids = [f"{doc_name}_chunk_{i}" for i in range(len(chunks))]
    metadatas = [
        {"source": doc_name, "chunk_index": i}
        for i in range(len(chunks))
    ]

    collection.add(
        ids=ids,
        embeddings=embeddings,
        documents=chunks,
        metadatas=metadatas,
    )
    print(f"✅ {doc_name}: stored {len(chunks)} chunks")


# Ingest all .txt files in the documents directory
doc_dir = "./documents"
for filename in os.listdir(doc_dir):
    if filename.endswith(".txt"):
        filepath = os.path.join(doc_dir, filename)
        ingest_document(filepath, filename.replace(".txt", ""))

print(f"📊 Total chunks in database: {collection.count()}")

# Expected output:
# ✅ onboarding: stored 8 chunks
# ✅ leave-policy: stored 5 chunks
# ✅ security-guide: stored 12 chunks
# 📊 Total chunks in database: 25
```

Three key design decisions to note here. First, chunks overlap by 100 characters to prevent losing context at boundaries. Second, the `RETRIEVAL_DOCUMENT` task type tells the model these embeddings are for indexing, which optimizes vector quality. Third, metadata like source filename and chunk position makes it easy to trace results back to their origin.

## Building the Search Function

With documents ingested, you can now build the search interface. The key difference from ingestion is using the `RETRIEVAL_QUERY` task type for the search query.

```python
# search.py
import os
import chromadb
from google import genai

client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection = chroma_client.get_collection("company_docs")


def search(query: str, n_results: int = 5, source_filter: str = None) -> dict:
    """Run a semantic search query."""
    # Embed the query with RETRIEVAL_QUERY task type
    result = client.models.embed_content(
        model="text-embedding-004",
        contents=[query],
        config={
            "task_type": "RETRIEVAL_QUERY",
            "output_dimensionality": 768,
        },
    )
    query_embedding = result.embeddings[0].values

    # Query ChromaDB
    where_filter = {"source": source_filter} if source_filter else None
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=n_results,
        where=where_filter,
        include=["documents", "metadatas", "distances"],
    )

    return results


def display_results(results: dict):
    """Print search results in a readable format."""
    for i, (doc, meta, dist) in enumerate(zip(
        results["documents"][0],
        results["metadatas"][0],
        results["distances"][0],
    )):
        similarity = 1 - dist  # Convert cosine distance to similarity
        print(f"\n--- Result {i + 1} (similarity: {similarity:.4f}) ---")
        print(f"📄 Source: {meta['source']}")
        print(f"📝 Content: {doc[:200]}...")


# Example usage
query = "How do I set up VPN for remote work?"
print(f"🔍 Query: {query}\n")
results = search(query)
display_results(results)

# Expected output:
# 🔍 Query: How do I set up VPN for remote work?
#
# --- Result 1 (similarity: 0.8742) ---
# 📄 Source: onboarding
# 📝 Content: VPN connection setup instructions. To access the internal network remotely...
#
# --- Result 2 (similarity: 0.8105) ---
# 📄 Source: security-guide
# 📝 Content: Remote access security policy. All employees working from home must...
```

The `RETRIEVAL_QUERY` and `RETRIEVAL_DOCUMENT` task types work as a pair — they're trained together so that queries and documents are positioned optimally in the same vector space. This asymmetric approach consistently outperforms using a single task type for both.

## Extending to a RAG Pipeline

Returning raw document chunks is useful, but combining search results with **Gemini's generative capabilities** creates a much better user experience. Here's how to build a RAG pipeline that answers questions using your documents as context:

```python
# rag_search.py
import os
import chromadb
from google import genai

client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection = chroma_client.get_collection("company_docs")


def rag_answer(query: str) -> str:
    """Generate an answer grounded in document search results."""
    # Step 1: Retrieve relevant chunks
    embed_result = client.models.embed_content(
        model="text-embedding-004",
        contents=[query],
        config={"task_type": "RETRIEVAL_QUERY", "output_dimensionality": 768},
    )
    query_embedding = embed_result.embeddings[0].values

    search_results = collection.query(
        query_embeddings=[query_embedding],
        n_results=3,
        include=["documents", "metadatas"],
    )

    # Step 2: Format context from search results
    context_parts = []
    for doc, meta in zip(
        search_results["documents"][0],
        search_results["metadatas"][0],
    ):
        context_parts.append(f"[{meta['source']}]\n{doc}")
    context = "\n\n".join(context_parts)

    # Step 3: Generate answer with Gemini
    prompt = f"""Answer the user's question based on the following internal documents.
If the documents don't contain relevant information, say "No matching information found in the documents."

## Reference Documents
{context}

## User Question
{query}

## Answer"""

    response = client.models.generate_content(
        model="gemini-2.5-flash",
        contents=prompt,
    )

    return response.text


# Example usage
question = "What's the process for requesting remote work?"
print(f"❓ Question: {question}\n")
answer = rag_answer(question)
print(f"💡 Answer:\n{answer}")

# Expected output:
# ❓ Question: What's the process for requesting remote work?
#
# 💡 Answer:
# Based on the company documents, here's the process for requesting remote work:
# 1. Submit a remote work request through the HR portal at least 3 days in advance
# 2. Your manager will review and approve the request within 24 hours...
```

The RAG pattern — retrieve, then generate — grounds Gemini's responses in your actual documents, significantly reducing hallucination while providing natural, conversational answers.

## Advanced Filtering with Metadata

ChromaDB's metadata filtering lets you narrow searches beyond pure semantic similarity:

```python
# Filter to a specific document source
results = collection.query(
    query_embeddings=[query_embedding],
    n_results=5,
    where={"source": "security-guide"},  # Only search security docs
)

# Combine multiple conditions with $and / $or
results = collection.query(
    query_embeddings=[query_embedding],
    n_results=5,
    where={
        "$or": [
            {"source": "onboarding"},
            {"source": "leave-policy"},
        ]
    },
)
```

This is particularly useful when users want to search within a specific category — for example, "find password requirements, but only in the security policy documents."

## Production Considerations

When moving from a local prototype to a team-facing system, keep these design points in mind.

**Incremental updates**: When documents change, use `collection.update()` with the chunk IDs to replace outdated embeddings without re-indexing everything. For deleted documents, call `collection.delete()` to remove their vectors.

**Batch processing**: The Gemini Embedding API accepts multiple texts per request, so batch your chunks to minimize API calls. Be mindful of rate limits (RPM/TPM) and implement retry logic with exponential backoff for production workloads.

**Scaling ChromaDB**: For small teams, local file persistence works well. When you outgrow it, switch to ChromaDB's client/server mode (`chromadb.HttpClient`) to run it as a dedicated service. For very large datasets, consider alternatives like Pinecone or Weaviate.

**Security**: When handling internal documents, secure your API keys using environment variables or a secrets manager. Implement access controls so users can only search documents they're authorized to view.

## Frequently Asked Questions

### Q1: Should I use ChromaDB or pgvector?

For prototyping and small-to-medium projects, ChromaDB is the faster path — it requires zero infrastructure and runs entirely in Python. If you're already running PostgreSQL and need to handle millions of vectors at scale, pgvector is the better fit. For a detailed guide on the pgvector approach, check out our [Gemini API × pgvector Semantic Search Engine guide](/articles/gemini-api/gemini-api-pgvector-semantic-search-engine).

### Q2: What's the difference between text-embedding-004 and gemini-embedding-exp-03-07?

`text-embedding-004` is a text-only embedding model optimized for stability and cost efficiency. `gemini-embedding-exp-03-07` is an experimental multimodal model that can embed both text and images. For pure text search, stick with `text-embedding-004`. For a deeper look at embedding fundamentals, see our [Gemini Embedding API introduction](/articles/gemini-api/embeddings-api-guide).

### Q3: What's the optimal chunk size for documents?

A range of 300–800 characters generally works well. Chunks that are too small lose context; chunks that are too large dilute the signal. The best approach is to experiment with your actual data. Adding overlap between chunks (typically 50–150 characters) helps capture information that spans chunk boundaries.

### Q4: How can I improve search accuracy?

Several techniques help: tuning chunk size and overlap, leveraging metadata filters, query rewriting (using Gemini to rephrase the user's question before searching), and re-ranking (using Gemini to score and reorder search results by relevance).
