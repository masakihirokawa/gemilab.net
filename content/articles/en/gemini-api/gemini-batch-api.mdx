## Understanding Gemini Batch API

The Gemini Batch API enables you to process large volumes of text requests efficiently and cost-effectively when real-time responses aren't critical. By submitting requests in batches, you can reduce API costs by approximately 50% compared to standard pricing—making it ideal for enterprise-scale operations.

Consider scenarios like daily content generation pipelines, bulk data analysis, ML training workflows, or processing historical archives. The Batch API shines in these use cases where you can tolerate processing delays measured in hours rather than seconds. This guide walks you through everything from foundational concepts to production implementation.

## Core Concepts of Batch Processing

### Why Choose Batch Processing

- **Cost Efficiency**: 50% discount on standard API rates
- **Massive Scale**: Handle thousands to millions of requests in single batches
- **Flexible Scheduling**: Process on your timeline, not forced real-time constraints
- **Detailed Tracking**: Monitor success/failure for every individual request
- **Reliability**: Built-in retry logic and comprehensive error reporting

### How Batch API Differs from Real-time API

| Feature | Batch API | Real-time API |
|---------|-----------|---------------|
| Response Time | Minutes to hours | Seconds |
| Cost | 50% discount | Standard rate |
| Use Case | Volume processing | Interactive applications |
| Scalability | Extremely high | High |
| Best For | Background jobs | User-facing features |

## Implementation Guide

### Structuring Your Batch Requests

Batch API expects requests in JSONL format (JSON Lines)—one JSON object per line. Each request represents an individual query to the model.

```python
import json
from typing import List

def create_batch_requests(texts: List[str], instruction: str = "") -> str:
    """Convert a list of texts into Batch API request format"""
    requests = []
    for i, text in enumerate(texts):
        system_prompt = instruction or "Summarize the following text concisely:"

        request = {
            "custom_id": f"request_{i:05d}",
            "params": {
                "model": "gemini-2.0-flash",
                "system_instruction": system_prompt,
                "contents": [
                    {
                        "role": "user",
                        "parts": [
                            {
                                "text": text
                            }
                        ]
                    }
                ]
            }
        }
        requests.append(json.dumps(request))

    return "\n".join(requests)

# Example usage
articles = [
    "Artificial intelligence is transforming how we process information...",
    "Machine learning models require substantial training data...",
    "Cloud computing enables businesses to scale globally..."
]

batch_jsonl = create_batch_requests(articles, "Create a one-sentence summary:")
print("Generated batch requests:")
print(batch_jsonl.split('\n')[0])

# Expected output:
# {"custom_id": "request_00000", "params": {"model": "gemini-2.0-flash", ...}}
```

### Submitting Your Batch

Once your requests are formatted, submit them to the Batch API endpoint and receive a batch ID for tracking.

```python
import google.generativeai as genai

def submit_batch_job(jsonl_content: str) -> str:
    """Submit JSONL content to Batch API"""
    # Save to file (required by the API)
    with open("batch_input.jsonl", "w") as f:
        f.write(jsonl_content)

    # Upload and submit
    batch_response = genai.batch.create(
        model="gemini-2.0-flash",
        requests=jsonl_content
    )

    batch_id = batch_response.name
    return batch_id

# Submit the batch
batch_id = submit_batch_job(batch_jsonl)
print(f"Batch submitted successfully: {batch_id}")

# Expected output:
# Batch submitted successfully: projects/your-project/batches/batch_1234567890abcd
```

### Polling for Results

Monitor your batch's progress and retrieve results once processing completes.

```python
import time

def wait_for_batch_completion(batch_id: str, poll_interval: int = 30,
                               max_wait: int = 86400) -> dict:
    """Poll batch status until completion or timeout"""
    start_time = time.time()
    elapsed = 0

    while elapsed < max_wait:
        batch = genai.batch.get(batch_id)
        state = batch.state
        processed = getattr(batch, 'processed_count', 0)
        total = getattr(batch, 'request_count', 'unknown')

        print(f"[{elapsed}s] Status: {state} | Processed: {processed}/{total}")

        if state == "COMPLETED":
            print("✓ Batch processing complete")
            return {
                "status": "completed",
                "batch": batch,
                "processed": processed,
                "total": total
            }
        elif state == "FAILED":
            print("✗ Batch failed")
            return {
                "status": "failed",
                "error": batch.error
            }

        time.sleep(poll_interval)
        elapsed += poll_interval

    return {"status": "timeout"}

# Wait for completion
result = wait_for_batch_completion(batch_id)
if result["status"] == "completed":
    print(f"Completed: {result['processed']} requests processed")

# Expected output:
# [0s] Status: PROCESSING | Processed: 0/150
# [30s] Status: PROCESSING | Processed: 45/150
# [60s] Status: PROCESSING | Processed: 120/150
# ✓ Batch processing complete
# Completed: 150 requests processed
```

## Optimization Strategies

### Right-Sizing Your Batches

- **Minimum**: Start with at least 10 requests per batch
- **Sweet Spot**: 500-5,000 requests balances efficiency and manageability
- **Scale**: The API comfortably handles 100,000+ requests per batch
- **Monitoring**: Larger batches may take longer; split if you need faster feedback

### Robust Error Handling

```python
def extract_batch_results(batch_id: str) -> dict:
    """Extract results with comprehensive error handling"""
    batch = genai.batch.get(batch_id)

    results = {
        "successful": [],
        "failed": [],
        "summary": {
            "total": 0,
            "success_rate": 0.0
        }
    }

    response_count = 0
    for response in batch.responses:
        response_count += 1

        if hasattr(response, 'error') and response.error:
            results["failed"].append({
                "id": response.custom_id,
                "error_code": response.error.code,
                "error_message": response.error.message
            })
        else:
            try:
                output_text = response.result.candidates[0].content.parts[0].text
                results["successful"].append({
                    "id": response.custom_id,
                    "output": output_text
                })
            except (AttributeError, IndexError) as e:
                results["failed"].append({
                    "id": response.custom_id,
                    "error_message": f"Parsing error: {str(e)}"
                })

    results["summary"]["total"] = response_count
    results["summary"]["success_rate"] = (
        len(results["successful"]) / response_count * 100
        if response_count > 0 else 0
    )

    return results

# Process results
all_results = extract_batch_results(batch_id)
print(f"Success Rate: {all_results['summary']['success_rate']:.1f}%")
```

## Frequently Asked Questions

### Q1: When should I use Batch API versus real-time streaming?

**A:** Use Batch API for background jobs, data pipelines, and scenarios where processing delays are acceptable (minutes to hours). Choose real-time streaming when you need immediate responses for interactive features. The 50% cost savings make Batch API economical for volume processing.

### Q2: What's the typical processing time for a batch?

**A:** Most batches complete within 15 minutes to 2 hours, though large batches may take longer. The API guarantees completion within 24 hours. Processing time depends on batch size and current system load. For time-sensitive work, keep batches under 10,000 requests.

### Q3: How do I handle failed requests in a batch?

**A:** Check the error details returned for each failed request. Common causes include malformed input, rate limits, or model-specific constraints. You can resubmit failed requests in a new batch after fixing the underlying issue. The API provides specific error codes to help diagnose problems.
