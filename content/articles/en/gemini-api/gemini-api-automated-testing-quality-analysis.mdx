# Build an Automated Test Generation & Code Quality Analysis System with Gemini API

Writing tests and maintaining code quality are essential yet time-consuming parts of software development. By leveraging Gemini API's excellent code comprehension capabilities, you can dramatically streamline the entire quality assurance process — from unit test generation to code review and quality metrics analysis.

In this guide, we'll build a practical automated test generation and quality analysis system in Python, making full use of Gemini 2.5 Pro's code understanding and generation abilities. You'll get complete, working implementation code for every component.

## System Architecture Overview

The system we're building consists of three major components:

1. **Test Generation Engine** — Analyzes source code and automatically generates unit tests
2. **Code Review Analyzer** — Detects quality issues and suggests improvements
3. **Quality Metrics Dashboard** — Visualizes project-wide quality indicators

We'll implement these using a combination of Gemini API's Function Calling and Structured Output features.

## Building the Test Generation Engine

### Source Code Analysis Module

First, let's create a module that parses Python source code and extracts functions and classes for test generation.

```python
from dataclasses import dataclass, field

@dataclass
class FunctionInfo:
    """Information about an analyzed function"""
    name: str
    args: list[str]
    return_type: str | None
    docstring: str | None
    source: str
    complexity: int = 0

@dataclass
class AnalysisResult:
    """Source code analysis results"""
    functions: list[FunctionInfo] = field(default_factory=list)
    classes: list[str] = field(default_factory=list)
    imports: list[str] = field(default_factory=list)

def analyze_source_code(source_code: str) -> AnalysisResult:
    """Parse source code using AST and extract function/class information"""
    tree = ast.parse(source_code)
    result = AnalysisResult()

    for node in ast.walk(tree):
        if isinstance(node, ast.FunctionDef):
            args = [arg.arg for arg in node.args.args if arg.arg != "self"]
            return_type = ast.unparse(node.returns) if node.returns else None
            docstring = ast.get_docstring(node)
            source = ast.unparse(node)

            # Simple cyclomatic complexity calculation
            complexity = sum(
                1 for child in ast.walk(node)
                if isinstance(child, (ast.If, ast.For, ast.While, ast.ExceptHandler))
            ) + 1

            result.functions.append(FunctionInfo(
                name=node.name,
                args=args,
                return_type=return_type,
                docstring=docstring,
                source=source,
                complexity=complexity,
            ))

        elif isinstance(node, ast.ClassDef):
            result.classes.append(node.name)

        elif isinstance(node, (ast.Import, ast.ImportFrom)):
            result.imports.append(ast.unparse(node))

    return result
```

### Generating Tests with Gemini

Using the analysis results, we generate unit tests via the Gemini API. Structured Output ensures we receive test code in a reliably parseable format.

```python
def generate_tests_for_function(
    function_info: FunctionInfo,
    full_source: str,
) -> str:
    """Generate unit tests for a function using Gemini API

    Args:
        function_info: Analyzed function information
        full_source: Complete source code of the target file

    Returns:
        Generated test code as a string
    """
    prompt = f"""Generate comprehensive unit tests for the following Python function.

## Target Function
```python
{function_info.source}
```

## Full File Context
```python
{full_source}
```

## Requirements
- Use pytest
- Include happy path, error cases, and boundary value tests
- Add a clear docstring to each test
- Prefix test function names with test_
- Include all necessary imports
- Use unittest.mock where appropriate

Output only the pytest test code. No explanatory text.
"""

    # Receive test code via Structured Output
    response = client.models.generate_content(
        model=MODEL_ID,
        contents=prompt,
        config=types.GenerateContentConfig(
            temperature=0.2,  # Keep test generation deterministic
            response_mime_type="application/json",
            response_schema={
                "type": "object",
                "properties": {
                    "test_code": {
                        "type": "string",
                        "description": "Generated pytest test code",
                    },
                    "test_count": {
                        "type": "integer",
                        "description": "Number of generated test cases",
                    },
                    "coverage_notes": {
                        "type": "string",
                        "description": "Notes on test coverage",
                    },
                },
                "required": ["test_code", "test_count", "coverage_notes"],
            },
        ),
    )

    result = json.loads(response.text)
    print(f"  ✅ {function_info.name}: {result['test_count']} tests generated")
    print(f"     Coverage: {result['coverage_notes']}")
    return result["test_code"]
```

### The Test Generation Pipeline

Here's the pipeline that processes multiple files across an entire project.

```python
from pathlib import Path

def generate_tests_for_project(
    source_dir: str,
    output_dir: str,
) -> dict[str, str]:
    """Auto-generate tests for an entire project

    Args:
        source_dir: Path to source code directory
        output_dir: Path for test output

    Returns:
        Dictionary of {file_path: test_code}

    Example:
        >>> results = generate_tests_for_project("src/", "tests/")
        >>> print(f"Files generated: {len(results)}")
        Files generated: 5
    """
    source_path = Path(source_dir)
    output_path = Path(output_dir)
    output_path.mkdir(parents=True, exist_ok=True)
    results = {}

    for py_file in source_path.rglob("*.py"):
        if py_file.name.startswith("_"):
            continue

        print(f"\n📄 Analyzing: {py_file}")
        source_code = py_file.read_text(encoding="utf-8")
        analysis = analyze_source_code(source_code)

        if not analysis.functions:
            print("  ⏭️ No testable functions found — skipping")
            continue

        all_tests = []
        for func in analysis.functions:
            if func.name.startswith("_"):
                continue
            test_code = generate_tests_for_function(func, source_code)
            all_tests.append(test_code)

        # Write the test file
        test_filename = f"test_{py_file.stem}.py"
        test_file = output_path / test_filename
        combined = "\n\n".join(all_tests)
        test_file.write_text(combined, encoding="utf-8")
        results[str(test_file)] = combined
        print(f"  💾 Saved: {test_file}")

    return results
```

### The Review Execution Engine

```python
@dataclass
class ReviewIssue:
    """An issue detected during code review"""
    severity: str
    line_range: str
    category: str
    description: str
    suggestion: str

def review_code(source_code: str, filename: str) -> list[ReviewIssue]:
    """Run an AI-powered code review using Gemini API

    Args:
        source_code: Source code to review
        filename: Filename for context

    Returns:
        List of detected issues

    Expected output example:
        [
            ReviewIssue(
                severity='warning',
                line_range='12-15',
                category='error_handling',
                description='Exception catch clause is too broad',
                suggestion='except ValueError as e:'
            ),
        ]
    """
    # Generate line-numbered source
    numbered = "\n".join(
        f"{i+1:4d} | {line}"
        for i, line in enumerate(source_code.splitlines())
    )

    prompt = f"""You are a senior Python developer. Perform a thorough code review on the following code.

## File: {filename}
```python
{numbered}
```

## Review Criteria
1. Security vulnerabilities (SQL injection, insufficient input validation, etc.)
2. Performance issues (unnecessary loops, memory leaks, etc.)
3. Inadequate error handling
4. Type safety concerns
5. Naming conventions and code readability
6. Maintainability problems

Use the report_issue tool for problems found, and approve_section for well-implemented code."""

    issues: list[ReviewIssue] = []

    response = client.models.generate_content(
        model=MODEL_ID,
        contents=prompt,
        config=types.GenerateContentConfig(
            tools=review_tools,
            temperature=0.1,
        ),
    )

    # Process Function Call results
    for candidate in response.candidates:
        for part in candidate.content.parts:
            if part.function_call and part.function_call.name == "report_issue":
                args = dict(part.function_call.args)
                issues.append(ReviewIssue(**args))

    # Sort by severity (critical > warning > info)
    severity_order = {"critical": 0, "warning": 1, "info": 2}
    issues.sort(key=lambda x: severity_order.get(x.severity, 3))

    return issues
```

## Quality Metrics Analysis

Let's calculate project-wide quality scores and use Gemini API for a comprehensive analysis.

### Collecting Metrics

```python
@dataclass
class QualityMetrics:
    """Project quality metrics"""
    total_functions: int = 0
    avg_complexity: float = 0.0
    functions_with_docstring: int = 0
    functions_with_type_hints: int = 0
    total_issues: int = 0
    critical_issues: int = 0
    test_coverage_estimate: float = 0.0

def collect_metrics(
    source_dir: str,
    issues: list[ReviewIssue] | None = None,
) -> QualityMetrics:
    """Collect quality metrics for a project"""
    metrics = QualityMetrics()
    complexities = []

    for py_file in Path(source_dir).rglob("*.py"):
        if py_file.name.startswith("_"):
            continue

        source = py_file.read_text(encoding="utf-8")
        analysis = analyze_source_code(source)

        for func in analysis.functions:
            metrics.total_functions += 1
            complexities.append(func.complexity)
            if func.docstring:
                metrics.functions_with_docstring += 1
            if func.return_type:
                metrics.functions_with_type_hints += 1

    if complexities:
        metrics.avg_complexity = sum(complexities) / len(complexities)

    if issues:
        metrics.total_issues = len(issues)
        metrics.critical_issues = sum(
            1 for i in issues if i.severity == "critical"
        )

    return metrics
```

## Integrating with CI/CD Pipelines

Here's an example GitHub Actions workflow that runs the analysis automatically on every pull request.

# .github/workflows/ai-quality-check.yml
name: AI Quality Analysis

on:
  pull_request:
    paths: ["src/**/*.py"]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5

          python-version: "3.12"

        run: pip install google-genai pydantic pytest

      - name: Run AI Code Review
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
        run: python scripts/ai_review.py src/ --output review_report.md

      - name: Generate Tests
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
        run: python scripts/ai_test_gen.py src/ --output tests/generated/

      - name: Run Generated Tests
        run: pytest tests/generated/ -v --tb=short

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const report = fs.readFileSync('review_report.md', 'utf8');

              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: report

## Putting It All Together

Here's the main script that ties everything together into a complete pipeline.

```python
def main():
    """Main entry point for the AI quality analysis pipeline

    Example:
        $ python ai_quality_pipeline.py
        📊 AI Quality Analysis Pipeline Started
        ============================================================

        🔍 Phase 1: Code Review
        📄 Reviewing: src/utils.py
          ⚠️ [warning] L12-15 (error_handling): Exception catch too broad
          ℹ️ [info] L23-25 (naming): Variable name unclear
        ...

        🧪 Phase 2: Auto Test Generation
        📄 Analyzing: src/utils.py
          ✅ calculate_total: 8 tests generated
          ✅ validate_input: 5 tests generated
        ...

        📈 Phase 3: Quality Metrics Analysis
        Quality Score: 78/100
        ...
    """
    source_dir = "src"
    test_output = "tests/generated"

    print("📊 AI Quality Analysis Pipeline Started")
    print("=" * 60)

    # Phase 1: Code Review
    print("\n🔍 Phase 1: Code Review")

        if py_file.name.startswith("_"):

    # Phase 3: Metrics Analysis
    print(f"\n📈 Phase 3: Quality Metrics Analysis")
    metrics = collect_metrics(source_dir, all_issues)

    # Save report
    Path("quality_report.md").write_text(report, encoding="utf-8")

if __name__ == "__main__":

## Frequently Asked Questions (FAQ)

### Can I use the tests generated by Gemini API as-is?

About 80–90% of generated tests will run without modification, but there are a few things worth checking. First, verify that mocks are correctly configured for any external service interactions. Second, confirm that generated assertions match expected values accurately. Finally, make sure tests are independent — no shared state between test functions. A recommended workflow for CI/CD integration: run the generated tests once, fix any failures, then commit.

### Is there a quality difference between Gemini 2.5 Pro and 2.5 Flash for test generation?

Gemini 2.5 Pro has a stronger grasp of complex logic, producing better boundary value tests and edge case coverage. On the other hand, 2.5 Flash is roughly 3x faster and significantly cheaper, making it perfectly capable for straightforward utility functions. For large projects, a "hybrid strategy" works best: use Pro for high-complexity functions (cyclomatic complexity 5+) and Flash for everything else. This gives you the best balance of quality and cost.

### How can I reduce false positives in the code review?

Providing project-specific rules and coding conventions via System Instructions dramatically reduces false positives. For instance, you might specify: "This project uses Django ORM, and returning `objects.filter()` results directly is an accepted pattern." Also, setting the temperature to 0.1 or lower produces more conservative (and more precise) review results.
