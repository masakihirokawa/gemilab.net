## What Is Gemini 3.1 Pro Custom Tools?

If you've built AI agents before, you've likely run into a frustrating problem: you register multiple custom functions for the model to use, but it keeps falling back to running bash commands instead. Even when a dedicated function exists for a task, the model opts for the general-purpose bash tool.

Google released **`gemini-3.1-pro-preview-customtools`** in February 2026 to solve exactly this problem. It shares the same intelligence, context window (1 million tokens), and multimodal capabilities as standard Gemini 3.1 Pro, but with one critical difference: **it prioritizes your custom-registered tools over generic execution methods**.

In this guide, we'll walk through how the Custom Tools model works, how to implement it, and what makes it different from the standard model.

## How It Differs from Standard Gemini 3.1 Pro

With standard Gemini 3.1 Pro, when both bash tools and custom functions are registered, the model picks whichever it deems most efficient for the situation. While this works well for general tasks, it can be unpredictable for agent development.

The Custom Tools model improves on this in several important ways.

**Tool selection priority changes.** When a custom function is available for a task, the model suppresses bash fallback and actively chooses the registered function.

**Function calling reliability increases.** Argument formatting errors and missed function calls are significantly reduced.

**Agent workflows become more predictable.** You can more accurately predict which tool will be called, making debugging and testing much easier.

The best part? Pricing is identical to standard Gemini 3.1 Pro. You get more reliable agents at zero additional cost.

## Building an Agent with Custom Tools

Now for the core of this guide. Let's register custom functions and see how the Custom Tools model selects tools.

### Step 1: Define Your Tool Functions

In this example, we'll define two custom tools: one for fetching weather data and another for checking schedules.

```python
from google import genai
from google.genai import types

# Tool function definitions
def get_weather(city: str) -> dict:
    """Fetch weather information for a specified city"""
    # Mock data in place of actual API calls
    weather_data = {
        "Tokyo": {"temp": 18, "condition": "Sunny", "humidity": 45},
        "London": {"temp": 12, "condition": "Cloudy", "humidity": 70},
        "New York": {"temp": 15, "condition": "Rainy", "humidity": 65},
    }
    return weather_data.get(city, {"error": f"No data found for {city}"})

def get_schedule(date: str) -> dict:
    """Fetch schedule for a specified date"""
    schedules = {
        "2026-03-22": [
            {"time": "10:00", "title": "Team standup"},
            {"time": "14:00", "title": "Client review"},
        ],
    }
    return schedules.get(date, {"message": "No events scheduled"})
```

### Step 2: Pass Tool Definitions to the Model

```python
# Create tool definitions
weather_tool = types.FunctionDeclaration(
    name="get_weather",
    description="Fetch current weather information (temperature, conditions, humidity) for a given city",
    parameters=types.Schema(
        type="OBJECT",
        properties={
            "city": types.Schema(
                type="STRING",
                description="City name to fetch weather for (e.g., Tokyo, London)"
            ),
        },
        required=["city"],
    ),
)

schedule_tool = types.FunctionDeclaration(
    name="get_schedule",
    description="Fetch schedule and events for a specified date",
    parameters=types.Schema(
        type="OBJECT",
        properties={
            "date": types.Schema(
                type="STRING",
                description="Date to check schedule for (YYYY-MM-DD format)"
            ),
        },
        required=["date"],
    ),
)

tools = types.Tool(function_declarations=[weather_tool, schedule_tool])
```

### Step 3: Run the Agent

```python
client = genai.Client(api_key="YOUR_API_KEY")

response = client.models.generate_content(
    model="gemini-3.1-pro-preview-customtools",
    contents="What's the weather in Tokyo? Also check my schedule for today.",
    config=types.GenerateContentConfig(
        tools=[tools],
    ),
)

# Process tool calls from the response
for part in response.candidates[0].content.parts:
    if part.function_call:
        fn_name = part.function_call.name
        fn_args = dict(part.function_call.args)
        print(f"Tool called: {fn_name}")
        print(f"Arguments: {fn_args}")

# Expected output:
# Tool called: get_weather
# Arguments: {'city': 'Tokyo'}
# Tool called: get_schedule
# Arguments: {'date': '2026-03-22'}
```

The Custom Tools model correctly identifies both requests — weather and schedule — and calls the appropriate custom function for each. With the standard model, compound requests like this sometimes triggered bash fallback; the Custom Tools model keeps things on track.

## Multi-Turn Tool Result Handling

In real-world agents, the flow is: tool call → get result → send back to model → generate final response.

```python
import json

def run_agent(user_message: str):
    client = genai.Client(api_key="YOUR_API_KEY")

    # Map of available tool functions
    available_functions = {
        "get_weather": get_weather,
        "get_schedule": get_schedule,
    }

    messages = [types.Content(role="user", parts=[types.Part(text=user_message)])]

    # Step 1: Initial request
    response = client.models.generate_content(
        model="gemini-3.1-pro-preview-customtools",
        contents=messages,
        config=types.GenerateContentConfig(tools=[tools]),
    )

    # Step 2: Process tool calls
    tool_results = []
    for part in response.candidates[0].content.parts:
        if part.function_call:
            fn_name = part.function_call.name
            fn_args = dict(part.function_call.args)
            result = available_functions[fn_name](**fn_args)
            tool_results.append(
                types.Part(
                    function_response=types.FunctionResponse(
                        name=fn_name,
                        response={"result": result},
                    )
                )
            )

    if tool_results:
        # Step 3: Send tool results back to the model
        messages.append(response.candidates[0].content)
        messages.append(types.Content(role="user", parts=tool_results))

        final_response = client.models.generate_content(
            model="gemini-3.1-pro-preview-customtools",
            contents=messages,
            config=types.GenerateContentConfig(tools=[tools]),
        )
        return final_response.text
    else:
        return response.text

# Run the agent
answer = run_agent("What's the weather in Tokyo and my schedule for today?")
print(answer)
# Expected output:
# The weather in Tokyo is sunny with a temperature of 18°C and 45% humidity.
# Here's your schedule for today:
# - 10:00 Team standup
# - 14:00 Client review
```

## Mixing Bash and Custom Tools

The Custom Tools model doesn't completely disable bash or code execution. For tasks that aren't covered by your custom functions, bash remains available. The key difference is **priority**.

Here's how different requests get routed:

```python
# Tool definitions including code execution
bash_tool = types.Tool(
    code_execution=types.ToolCodeExecution()
)

custom_tools = types.Tool(
    function_declarations=[weather_tool, schedule_tool]
)

# "What's the weather in Tokyo?" → get_weather is called (custom tool priority)
# "Calculate the factorial of 100" → Code execution is used (no matching custom tool)
# "Get the weather and save it to CSV" → get_weather + code execution combined
```

This priority system ensures that your custom functions handle what they're designed for, while everything else falls through to bash or code execution gracefully.

## Practical Use Case: Customer Support Agent

Let's build something more practical — a customer support agent with multiple specialized tools.

```python
# Support tool definitions
support_tools = types.Tool(
    function_declarations=[
        types.FunctionDeclaration(
            name="search_faq",
            description="Search the FAQ database for answers matching a keyword query",
            parameters=types.Schema(
                type="OBJECT",
                properties={
                    "query": types.Schema(type="STRING", description="Search keyword"),
                },
                required=["query"],
            ),
        ),
        types.FunctionDeclaration(
            name="create_ticket",
            description="Create a new support ticket",
            parameters=types.Schema(
                type="OBJECT",
                properties={
                    "title": types.Schema(type="STRING", description="Ticket title"),
                    "priority": types.Schema(
                        type="STRING",
                        description="Priority level (low / medium / high)",
                        enum=["low", "medium", "high"],
                    ),
                    "description": types.Schema(type="STRING", description="Detailed description of the issue"),
                },
                required=["title", "priority", "description"],
            ),
        ),
        types.FunctionDeclaration(
            name="check_order_status",
            description="Check shipping status by order number",
            parameters=types.Schema(
                type="OBJECT",
                properties={
                    "order_id": types.Schema(type="STRING", description="Order number"),
                },
                required=["order_id"],
            ),
        ),
    ]
)
```

With the Custom Tools model, when a customer says "my order hasn't arrived," the agent reliably calls `check_order_status` first to check the delivery status, and if the issue persists, follows up with `create_ticket` to file a support ticket. This kind of multi-step workflow becomes far more dependable.

## Frequently Asked Questions (FAQ)

### Q1: Is the Custom Tools model more expensive than standard Gemini 3.1 Pro?

No, pricing is identical. It costs $1.25 per million input tokens and $10 per million output tokens — exactly the same as standard Gemini 3.1 Pro. You get improved function calling accuracy at zero additional cost.

### Q2: Can I still use Code Execution with the Custom Tools model?

Yes, absolutely. The Custom Tools model "prioritizes" custom functions but doesn't disable code execution. For tasks not covered by your custom tools — such as calculations or data processing — code execution is still available.

### Q3: When should I choose the Custom Tools model over the standard model?

The primary use case is agent development where you register multiple custom functions. It's especially valuable when bash tools and custom functions coexist and you need higher accuracy for custom function calls. For simple text generation or code-only tasks, standard Gemini 3.1 Pro works just fine.

### Q4: Is Provisioned Throughput available?

Currently, Provisioned Throughput is not supported for `gemini-3.1-pro-preview-customtools`. If you need to handle high request volumes, reach out to your Google account team for assistance.
